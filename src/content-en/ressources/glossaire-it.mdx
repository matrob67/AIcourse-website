## Why This Section?

This course is about AI, but AI lives within a technical ecosystem. When an engineer says "I pushed my commit to the main branch of the repo, the CI/CD will trigger the Docker build" -- it's gibberish if you're not a developer.

This section translates the everyday vocabulary of developers into plain language.

---

## Everyday Tools

### IDE (Integrated Development Environment)

The **IDE** is the developer's code editor -- the equivalent of Word for writing code. It's not a simple notepad: it colors code by type (keywords in blue, text in green, errors in red), detects errors in real time, offers autocomplete, and integrates dozens of tools.

The most popular ones:
- **VS Code** (Microsoft) -- free, the most popular, infinitely extensible
- **PyCharm** (JetBrains) -- specialized for Python, very powerful for ML
- **Cursor** -- VS Code with built-in AI (native copilot)

### Terminal / CLI (Command Line Interface)

The **terminal** is the command line interface -- the "black screen with text" where developers type commands. It's the opposite of a graphical interface (clicking buttons): here, you type text to give orders to the computer.

Why do devs use it instead of clicking? Because it's **faster, automatable, and reproducible**. You can write a script that executes 50 commands in a row -- impossible with clicks.

```bash
# Example commands
ls                    # list files in the folder
cd my-project         # enter a folder
python train.py       # run a Python script
pip install torch     # install a library
```

### Script vs Program

A **script** is a code file that runs directly (line by line). A **program** is more structured, often compiled, with multiple interconnected files. In ML, you mostly write Python scripts -- `.py` files that you run with `python my_script.py`.

## Git: Code Versioning

### The Problem Git Solves

Imagine a Word document shared among 10 people. Everyone makes changes. Very quickly, you end up with `report_v2_final_FINAL_v3_corrected.docx`. It's chaos.

**Git** solves this problem for code. It's a **version control system**: it records every change, who made it, when, and why. You can go back in time at any point, compare two versions, and merge the work of multiple people.

### Key Git Concepts

**Repository (repo)** -- the project folder, with its entire history of changes. It's like a shared folder, but with a perfect memory of every change ever made.

**Commit** -- a "snapshot" of the code at a given moment, with a message explaining what changed. It's like hitting `Ctrl+S` but smarter: you name each save. Example: "Fix the login bug" or "Add the homepage".

**Branch** -- a parallel copy of the code to work on a feature without touching the main code. Like making a copy of your document to test risky changes without damaging the original.

**Merge** -- reintegrating work from a branch into the main code. Git automatically merges changes and flags conflicts if two people modified the same line.

**Pull Request (PR)** -- a merge request. Before merging, you submit your changes so colleagues can review and approve them. This is the code review process.

**Main / Master** -- the main branch, the "official" production code. You never modify main directly -- you create a branch, work, make a PR, then merge.

<Callout type="tip" title={"Git in one sentence"}>
Git = the "Track Changes" feature in Word, but for code, 1,000x more powerful, and capable of managing the simultaneous work of hundreds of developers.
</Callout>

### GitHub / GitLab

**GitHub** and **GitLab** are web platforms that host Git repos. It's like Google Drive for code: the repo lives online, accessible to the entire team. GitHub is owned by Microsoft and hosts the majority of the world's open-source projects.

## API and SDK: Communicating with Services

### API (Application Programming Interface)

An **API** is an **entry point** for communicating with a service from code. When you use ChatGPT on the website, you're going through the graphical interface. When a developer wants to integrate GPT into their application, they go through OpenAI's **API**.

Concretely, an API is a web address (URL) to which you send structured requests, and which returns structured responses:

```
-> Request: "Translate 'hello' to French" (sent to api.openai.com)
<- Response: "bonjour"
```

An **endpoint** is a specific URL within the API. For example:
- `api.openai.com/v1/chat/completions` -> for chat
- `api.openai.com/v1/embeddings` -> for embeddings

### SDK (Software Development Kit)

An **SDK** is a **code library** provided by a service to make using its API easier. Without an SDK, you'd have to write raw HTTP requests -- complex and error-prone. The SDK does the heavy lifting for you.

```python
# Without SDK (raw HTTP request) — painful
import requests
response = requests.post(
    "https://api.openai.com/v1/chat/completions",
    headers={"Authorization": "Bearer sk-..."},
    json={"model": "gpt-4", "messages": [{"role": "user", "content": "Hello"}]}
)

# With SDK — simple
from openai import OpenAI
client = OpenAI()
response = client.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "Hello"}]
)
```

AI provider SDKs:
| Provider | Python SDK | Installation |
|---|---|---|
| OpenAI | `openai` | `pip install openai` |
| Anthropic | `anthropic` | `pip install anthropic` |
| Mistral | `mistralai` | `pip install mistralai` |
| Google | `google-genai` | `pip install google-genai` |

### REST / Webhook

**REST** is the most common architecture style for web APIs: you send HTTP requests (GET to read, POST to create) and receive JSON in return. Virtually all AI APIs are REST.

A **webhook** is the reverse of an API: instead of you going to fetch the information, the service **pushes** it to you automatically when an event occurs. Example: Stripe sends you a webhook when a payment is received.

## The Development Ecosystem

### Package Manager

A **package manager** installs and manages code libraries. In Python, it's **pip**:

```bash
pip install torch          # install PyTorch
pip install transformers   # install the Hugging Face library
pip install -r requirements.txt  # install all dependencies for a project
```

In JavaScript, it's **npm** or **yarn**. The real-life equivalent: the App Store, but for code libraries.

### Virtual Environment

A **virtual environment** isolates a project's dependencies. Why? Because two projects might need different versions of the same library. Project A needs PyTorch 2.0, Project B needs PyTorch 1.9 -- without isolation, there's a conflict.

```bash
python -m venv my_env         # create an environment
source my_env/bin/activate    # activate it
pip install torch             # install torch ONLY in this environment
```

**Conda** is a popular alternative in ML (also manages C/CUDA libraries, not just Python).

### Environment Variable / .env File

**Environment variables** store sensitive information (API keys, passwords) outside the code. You should **never** put an API key directly in the code (otherwise it ends up on GitHub, visible to everyone).

```bash
# .env file (NEVER committed to Git)
OPENAI_API_KEY=sk-abc123...
DATABASE_URL=postgres://user:pass@host/db
```

```python
# in the code, you read the variable
import os
api_key = os.getenv("OPENAI_API_KEY")
```

### JSON and YAML

**JSON** and **YAML** are formats for storing structured data -- the equivalent of an Excel spreadsheet, but in text:

```json
{
  "name": "GPT-4",
  "parameters": 1800000000000,
  "multimodal": true
}
```

```yaml
name: GPT-4
parameters: 1800000000000
multimodal: true
```

JSON is stricter (braces, quotes). YAML is more readable (indentation). Both are ubiquitous in project configuration.

## Infrastructure and Deployment

### Frontend / Backend

The **frontend** is what the user sees and interacts with (the web interface, buttons, forms). The **backend** is what runs on the server (business logic, database, calls to AI models). When you ask ChatGPT a question:
- The **frontend** displays the web page and captures your message
- The **backend** receives the message, calls the GPT model, and sends back the response

### Database (SQL, NoSQL)

A **database** stores data in a structured and queryable way. **SQL** (PostgreSQL, MySQL) organizes data in tables (like Excel). **NoSQL** (MongoDB, Redis) is more flexible, storing JSON documents or key-value pairs.

In AI, **vector databases** (Pinecone, Weaviate, Qdrant) store embeddings for semantic search (used in RAG -- Retrieval-Augmented Generation).

### Docker / Container

**Docker** packages an application with **all its dependencies** into a "container" -- an isolated box that runs identically everywhere. It's the solution to the classic "it works on my machine but not on the server" problem.

```bash
docker run -p 8080:80 my-application  # run the application in a container
```

Analogy: a Docker container is like a frozen meal -- everything is pre-packaged and ready to consume, regardless of which kitchen you reheat it in.

### CI/CD (Continuous Integration / Continuous Deployment)

**CI/CD** automates the process between "the developer pushes code" and "the code runs in production":

1. **CI** (Continuous Integration) -- with each commit, automated tests verify that the code works
2. **CD** (Continuous Deployment) -- if the tests pass, the code is automatically deployed to production

It's like an automated assembly line: the developer pushes their code, the robots verify and deploy.

### Staging vs Production

- **Production** (prod) -- the real environment, the one actual users use
- **Staging** -- a copy of production to test changes before putting them in prod
- **Local** (localhost) -- your computer, for developing and testing

A **port** is a number that identifies a service on a machine. `localhost:3000` = "my computer, service number 3000". It's like an apartment number in a building (the building = the machine, the apartment = the port).

### Cloud vs On-Premise

- **Cloud** -- your code runs on someone else's servers (AWS, Google Cloud, Azure). You pay per usage.
- **On-premise** (on-prem) -- your code runs on your own physical machines, in your own facilities.

For LLM training, cloud dominates (flexible access to GPUs). For production inference, it's a strategic choice: cloud (flexible, expensive long-term) vs on-prem (upfront investment, economical long-term).

## Debugging and Quality

### Logs

**Logs** are messages the program writes while it's running -- like a ship's logbook. When something crashes, logs are the first place you look:

```
[2025-01-15 14:32:01] INFO: Server started on port 8080
[2025-01-15 14:32:05] INFO: Request received from user #42
[2025-01-15 14:32:06] ERROR: Model not found: gpt-5-turbo
```

### Bug / Debug / Stack Trace

A **bug** is unintended program behavior. **Debugging** means finding and fixing the bug. A **stack trace** is the detailed error message the program displays when it crashes -- it shows exactly which line of code failed and why.

### Post-Mortem

A **post-mortem** is a document written after a production incident (outage, critical bug, data leak). It answers: **what happened, why, what was the impact, and how do we prevent it from happening again.**

It's the equivalent of a lessons-learned review -- but in tech culture, the post-mortem is formalized and often **public**. Google, Meta, Cloudflare regularly publish their post-mortems so the entire industry can learn from their mistakes.

Typical structure:
1. **Summary** -- "Service X was unavailable for 3 hours on January 15"
2. **Timeline** -- minute-by-minute chronology of the incident
3. **Root cause** -- the real underlying problem (not just the symptom)
4. **Impact** -- how many users affected, revenue lost
5. **Corrective actions** -- what's being put in place to prevent it from happening again

<Callout type="info" title={"Blameless culture"}>
A good post-mortem is **blameless** -- you look for systemic causes, not a scapegoat. "The deployment had no automated tests" rather than "John broke prod." This culture encourages transparency: people report problems instead of hiding them.
</Callout>

### Open Source / License

An **open source** project has its code visible to everyone, freely usable and modifiable. But "open source" doesn't mean "no rules": each project has a **license** that defines what you can do:

| License | You can... |
|---|---|
| MIT | Do anything, including commercial use |
| Apache 2.0 | Do anything, with patent protection |
| GPL | Use it, but your modifications must also be open source |
| Proprietary | Only what the contract allows |

Llama (Meta) and Mistral use permissive licenses (Apache 2.0 or similar). GPT and Claude are proprietary -- you access the service via API, but the model code is not available.

## Cross-Cutting Technical Concepts

### Latency / Throughput / Cache

- **Latency** -- the time between the request and the response. "The model takes 200ms to respond" = 200ms of latency.
- **Throughput** -- the number of requests processed per second. "The server handles 1,000 requests/second" = throughput of 1,000 req/s.
- **Cache** -- a fast temporary memory that stores frequent results to avoid recalculating them. When your browser loads a web page faster the second time, that's thanks to the cache.

### Sandbox

A **sandbox** is an isolated environment where you can experiment without risk. An AI agent that executes code often does so in a sandbox -- if the code is malicious, it can't affect the real system.

### Token (in Security)

Outside the NLP context (where token = piece of a word), a **token** in security is an authentication key -- a long string of characters that proves your identity. Your API key `sk-abc123...` is a token. "Revoking a token" means canceling the key.

<Callout type="tip" title={"Vocabulary evolves fast"}>
Tech constantly invents jargon. The important thing isn't to know everything, but to know **how to ask**. When an engineer uses an unfamiliar term, ask: "Can you explain what that actually means?" -- good engineers know how to translate.
</Callout>
