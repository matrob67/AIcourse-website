## The Problem: How Do You Test an Agent?

Evaluating a classic language model is relatively straightforward: you ask a question, it answers, it's right or wrong. You compare the answer to a reference and compute a score. But an AI agent? That's a whole different story.

An agent makes a **series of decisions in a dynamic environment**. It must explore, search for information, choose its tools, modify its approach when something doesn't work, and deal with the unexpected. Evaluating that is like evaluating a surgeon: not just "can you hold a scalpel?" but "can you handle an unexpected complication in the middle of an operation, adapt your plan, and achieve the right outcome despite the obstacles?"

<Callout type="info" title={"What a Good Agent Benchmark Measures"}>
Four dimensions are essential. **Success**: is the task completed correctly? **Efficiency**: how many steps and how many tokens were needed? **Robustness**: is the result stable across variations of the same task? **Safety**: did the agent take dangerous, destructive, or unauthorized actions along the way?
</Callout>

## SWE-bench: Testing Code Agents

### The Mission

SWE-bench is the reference benchmark for evaluating agents on their ability to **solve real bugs in real open-source projects**. Not fabricated programming exercises — real GitHub issues on Django, Flask, Scikit-learn, Matplotlib, and other major Python projects with thousands of users.

Each task consists of a GitHub issue along with the corresponding human fix commit. The agent doesn't know this fix — it must rediscover it on its own. This is testing at real-world difficulty, in complex codebases of hundreds of thousands of lines.

### How It Works

1. An issue is selected (for example: "The dependency graph breaks when...")
2. The agent receives the issue description and full access to the repository
3. It must navigate the code, understand the bug, and produce a corrective patch
4. The patch is automatically evaluated by running the project's test suite

It's objective, reproducible, and unforgiving — tests don't forgive partial fixes.

### Three Versions

- **Full** (2,294 tasks): including very difficult cases, sometimes barely solvable even for an experienced developer
- **Lite** (300 tasks): a selection of solvable tasks, faster to run
- **Verified**: a manually verified subset to guarantee test quality

### Current Results (2025-2026)

| System | Success Rate (%) |
|---|---|
| Simple RAG | ~2 |
| Claude 3.5 Sonnet (with agent, 2024) | ~49 |
| OpenAI o1 (2024) | ~69 |
| Claude Sonnet 4.6 (2026) | ~80.8 |
| Best system | ~81 |

(2026 update: Claude Sonnet 4.6 reaches 80.8% on SWE-bench.)

<Callout type="paper" title="A Spectacular Progression">
In 12 months, scores went from 3% to 75%. This progression comes from the combination of better foundation models and more sophisticated agent frameworks (better code search, better planning, better iterative testing). But the curve is flattening: the remaining tasks are the truly difficult ones — those requiring deep architectural understanding or coordinated changes across many files.
</Callout>

## WebArena: Web Agents

### The Mission

WebArena evaluates agents on **web navigation** tasks in realistic self-hosted environments. Not just "click the red button," but complete tasks that require navigating between pages, understanding interfaces, searching for information, and filling out forms — exactly what a human does daily on the web.

The environments include:
- A forum (Reddit-like)
- An e-commerce site (Amazon-like)
- A CMS (content management system)
- A GitLab server

### Example Tasks

- "Find the cheapest product in the electronics category and add it to the cart"
- "Create a new post in the forum with this title and content"
- "Solve issue #42 on GitLab by submitting a merge request"

Each task involves multiple navigation steps, contextual decisions, and sometimes handling unexpected elements (pop-ups, redirects, ambiguous forms).

### Metrics

- **Success rate**: is the task completed correctly end-to-end?
- **Step efficiency**: number of actions required (clicks, inputs, navigations)
- **Functional correctness**: is the final result truly functional?

### Results

The best agents achieve about 30-40% success — well below human performance. Autonomous web navigation remains a major challenge. The most frequent errors:

- Misinterpreting the interface (clicking the wrong button because the text is ambiguous)
- Losing spatial context (forgetting which page you're on in a multi-step workflow)
- Inability to recover from an error (a missed click that leads to a dead end)

## Other Important Benchmarks

### GAIA

GAIA evaluates AI assistants on general tasks requiring multiple combined skills:
- 466 questions of varying difficulty (levels 1 to 3)
- Requires web search, mathematical calculations, multi-step reasoning
- Humans succeed at 92%, the best agents at ~70%
- The human-machine gap remains significant on level 3 questions

### AgentBench

AgentBench tests agents across 8 distinct environments, covering a broad spectrum of skills:
- Bash (executing command-line instructions)
- SQL (querying databases)
- Knowledge graphs (reasoning over knowledge graphs)
- Text and strategy games (planning and reasoning)
- Web navigation

### ToolBench

ToolBench tests the ability to use **real-world APIs** — not toy APIs, but 16,000+ real APIs with their documentation and quirks. Tasks are practical ("book a hotel under $100") and measure the ability to chain multiple API calls coherently.

### OSWorld

OSWorld pushes evaluation into **complete desktop environments** (Ubuntu, Windows, macOS). Agents must edit documents, manage files, use applications like a human would. It's the benchmark closest to real usage, and also the one where scores are lowest.



## Designing Your Own Benchmark

### To Evaluate an Agent on Your Use Case

Public benchmarks will never cover your specific case. If you're deploying an agent in production, you must build your own evaluation:

1. **Define tasks**: 50-200 tasks representative of what the agent will actually do
2. **Success criteria**: automatic if possible (verification scripts, assertions, comparisons)
3. **Vary difficulty**: easy tasks (baseline), medium (common cases), and complex (edge cases)
4. **Include traps**: data errors, ambiguities, missing information
5. **Measure cost**: tokens consumed and API calls per task — an agent that succeeds but costs $10 per task isn't viable

### Metrics to Track

| Metric | Explanation | Importance |
|---|---|---|
| Success rate | Task completed correctly | Critical |
| Number of steps | How many actions to succeed | Cost/efficiency |
| Cost per task | Tokens x price per token | Budget |
| Recovery rate | Ability to recover after error | Robustness |
| Security violations | Unauthorized or destructive actions | Safety |
| Total time | End-to-end duration | User UX |

<Callout type="tip" title={"Evaluation Must Be Automated"}>
A benchmark you can't run automatically will never be executed regularly — and a benchmark you don't run regularly is useless. Invest in automation from the start: automatic result verification, reproducible scripts, CI/CD integration. The goal: `make evaluate` in one command, results in a few minutes.
</Callout>

## Limitations of Current Benchmarks

### Data Contamination

Public benchmarks risk being "contaminated": the latest models may have seen SWE-bench solutions in their training data. Artificially inflated scores from this contamination are hard to detect and skew comparisons.

### The Gap with Reality

Even the most sophisticated benchmarks remain simpler than real daily tasks. Benchmark tasks are well-defined, results objective, the environment controlled. Production involves noise, ambiguous specifications, systems that crash, and users who change their minds mid-task.

<Callout type="warning" title="Benchmark Doesn't Mean Production">
An excellent score on SWE-bench doesn't guarantee an agent will be reliable in your environment. Benchmarks test isolated capabilities under ideal conditions. Production is varied, unpredictable, and full of edge cases no benchmark covers. Continuous evaluation in production — monitoring results, user feedback, failure analysis — is indispensable and complementary to benchmarks.
</Callout>

## The Future of Evaluations

Agent evaluation is a rapidly evolving field. The most promising directions:

- **More realistic benchmarks**: open-ended, multi-session tasks, not just isolated one-shots
- **More diverse**: covering medicine, law, finance — not just code and web
- **Contamination-resistant**: dynamic benchmarks that renew themselves, or private evaluations
- **Continuous evaluation**: not a single snapshot, but continuous production monitoring

The combination of static benchmarks (SWE-bench to measure progress) and continuous evaluation (production monitoring to measure real reliability) remains the best available approach.
