## The Problem: We Trust Code We Don't Verify

In 2026, approximately 42% of production code is generated by AI. Yet 96% of developers say they **don't trust** the code produced by LLMs. And only 48% take the time to manually review it.

So we have a worrying paradox: we massively use AI to code, while knowing it can make mistakes, and without checking if it has. This is what researchers call the **verification debt**: a silent risk accumulating in codebases around the world, like a ticking time bomb.

<Callout type="warning" title={"The human bottleneck"}>
Reviewing AI code line by line is slow, expensive, and doesn't scale. The more models generate code, the more humans become the limiting factor. The solution isn't to review better, it's to automate the verification itself.
</Callout>

## Mistral's Idea: What if AI Proved Its Own Code Correct?

That's where **Leanstral** comes in, announced in March 2026. The concept is simple but radical: instead of generating code and hoping it works, you generate code **accompanied by a mathematical proof** of its correctness.

### The Analogy That Clarifies Everything

Imagine an architect handing you building plans. Today, LLMs are like a very fast but sometimes distracted architect: they draw quickly, but they might forget a load-bearing wall or miscalculate a load.

**Leanstral** is like that architect delivering their plans with an **independent inspection certificate** proving the structure holds up. Not an opinion, not a subjective review: a **formal proof** verified by a mathematical system (the Lean 4 language). If the certificate is valid, the building stands. Period.

### What is Lean 4?

Lean 4 is a programming language created by Microsoft Research, designed specifically to write **mathematical proofs** verifiable by computer. It's what's called a "proof assistant": you formulate a proposition ("this function always correctly sorts a list"), write a proof, and Lean automatically verifies whether that proof is valid.

The crucial point: if the proof compiles in Lean, it's **mathematically correct**. Not "probably correct," not "correct in the tested cases": correct for all possible cases, without exception.

<Callout type="info" title="Tests vs formal proofs: a fundamental difference">
Unit tests verify particular cases: "for the input [3, 1, 2], the function returns [1, 2, 3]." Formal verification proves the code is correct **for all possible inputs**. It's the difference between "I tried 1,000 combinations and it worked" and "I have the mathematical proof it will always work, no matter what."
</Callout>

## How Leanstral Works

Leanstral isn't just a model that generates text: it's an **agent** that interacts in a loop with the Lean 4 environment.

### The Work Cycle

1. **Leanstral receives a specification** -- for example: "prove that this sorting function always produces an ordered list"
2. **It generates a proof attempt** in Lean 4
3. **Lean verifies the proof** automatically -- either it compiles (correct), or Lean returns a precise error explaining why
4. **Leanstral reads the error** and corrects its proof accordingly
5. Back to step 3, until the proof compiles

It's this loop that makes Leanstral powerful. The LLM is the creative that proposes proof strategies. Lean is the implacable verifier that lets no error pass. One imagines, the other validates. And unlike a human reviewer, Lean never gets tired and never makes a verification error.

<Callout type="tip" title="Integration via MCP + Lean">
Leanstral uses the MCP protocol (Model Context Protocol) with the `lean-lsp-mcp` server to interact with Lean. This means it can also integrate into other tools and workflows via Mistral Vibe (Mistral's CLI). Formal verification isn't an isolated tool: it plugs into your existing workflow.
</Callout>


## The Architecture: An Ultra-Sparse MoE Built for Proofs

Under the hood, Leanstral uses the **Sparse Mixture-of-Experts** (MoE) architecture, Mistral's specialty, pushed to an extreme level of efficiency.

### Key Numbers

| Characteristic | Value |
|---|---|
| **Total parameters** | 120 billion |
| **Active parameters per token** | 6 billion (only 5%!) |
| **Number of experts** | 128 specialized modules |
| **Experts activated per token** | 4 out of 128 |
| **Efficiency ratio** | 18x -- the capacity of a 120B model at the cost of a 6.5B model |

To extend the MoE analogy: imagine a firm of 128 specialized consultants. For each question, you only consult the 4 most relevant ones instead of gathering all 128 in a meeting room. You benefit from the collective expertise, but only pay for 4 consultations. With 128 experts and top-4, Leanstral pushes the efficiency ratio well beyond what Mixtral (8 experts, top-2) achieved.

## The Results: Impressive Performance at Reduced Cost

On the **FLTEval** benchmark (formal proof completion on realistic Lean repositories), Leanstral delivers results that speak for themselves:

| Configuration | FLTEval Score | Cost |
|---|---|---|
| Leanstral pass@1 | 21.9 | **$18** |
| Leanstral pass@2 | 26.3 | **$36** |
| Leanstral pass@4 | 29.3 | **$72** |
| Leanstral pass@16 | 31.9 | **$290** |
| Claude Sonnet | 23.7 | **$549** |
| Claude Opus 4.6 | 33.5 | **$1,650** |

For a budget of 36 dollars, Leanstral **surpasses Claude Sonnet** (26.3 vs 23.7). To approach Claude Opus level, you need to go up to pass@16 for 290 dollars, still **5.7x cheaper**. Formal verification plays a key role here: you can launch multiple attempts and keep the one that compiles, with no risk of selecting a false positive.

<Callout type="info" title={"What is pass@k?"}>
pass@k means: "we run the model k times, and keep the best result." Normally, selecting the "best" result among k is a problem in itself (how do you know which one is good?). But with formal verification, it's free: Lean automatically verifies whether each proof is correct. You launch k attempts, keep the one that compiles. It's a fundamental advantage of the "generate + prove" paradigm.
</Callout>

Leanstral also beats much larger generalist open-source models like GLM5 (744B parameters), Kimi-K2.5 (1T parameters) and Qwen3.5 (397B), while being considerably smaller and cheaper.

## Access and Deployment

Leanstral is distributed under an **Apache 2.0 license** (fully open-source), with three access modes:

1. **Via Mistral Vibe** -- Mistral's CLI. Type `/leanstral` to start a proof agent in your Lean project
2. **Free API** -- a temporary endpoint (`labs-leanstral-2603`) for testing and giving feedback to Mistral
3. **Self-hosted** -- download the weights and deploy on your own infrastructure

<Callout type="tip" title="Try Leanstral">
To get started quickly: sign up on [console.mistral.ai](https://console.mistral.ai/codestral/cli), install Mistral Vibe, then run `/leanstral` in a project containing `.lean` files.
</Callout>

## Why This Is a Paradigm Shift

Leanstral doesn't just settle for being another good model. It changes the fundamental question of AI-assisted coding. Until now, the workflow was: *generate code, review it, hope it's correct*. With formal verification, it becomes: *generate code, prove it, have the mathematical guarantee it's correct*.

This is particularly critical in three domains:
- **Safety-critical code** (aerospace, medical, finance, nuclear) where a bug can cost lives or billions
- **Mathematics** -- Leanstral can help mathematicians formalize and verify complex proofs
- **Trust in AI** -- if generated code is proven correct, the question of AI reliability is resolved (at least for the formally verified portion)

Leanstral's announcement coincided with the release of Mistral Small 4 and Mistral's entry into the **Nvidia Nemotron Coalition** to co-develop open-source foundation models.
