## Pourquoi cette section ?

Ce cours parle d'IA, mais l'IA vit dans un écosystème technique. Quand un ingénieur dit "j'ai push mon commit sur la branch main du repo, le CI/CD va trigger le build Docker" — c'est du chinois si vous n'êtes pas développeur.

Cette section traduit le vocabulaire du quotidien des développeurs en langage humain.

---

## Les outils du quotidien

### IDE (Integrated Development Environment)

L'**IDE** est l'éditeur de code du développeur — l'équivalent de Word pour l'écriture de code. Ce n'est pas un simple bloc-notes : il colore le code par type (mots-clés en bleu, texte en vert, erreurs en rouge), détecte les erreurs en temps réel, propose de l'autocomplétion, et intègre des dizaines d'outils.

Les plus utilisés :
- **VS Code** (Microsoft) — gratuit, le plus populaire, extensible à l'infini
- **PyCharm** (JetBrains) — spécialisé Python, très puissant pour le ML
- **Cursor** — VS Code avec de l'IA intégrée (copilot natif)

### Terminal / CLI (Command Line Interface)

Le **terminal** est l'interface en ligne de commande — le "truc noir avec du texte" où les développeurs tapent des commandes. C'est l'opposé de l'interface graphique (cliquer sur des boutons) : ici, on tape du texte pour donner des ordres à l'ordinateur.

Pourquoi les devs l'utilisent au lieu de cliquer ? Parce que c'est **plus rapide, automatisable, et reproductible**. On peut écrire un script qui exécute 50 commandes d'affilée — impossible avec des clics.

```bash
# Exemples de commandes
ls                    # lister les fichiers du dossier
cd mon-projet         # entrer dans un dossier
python train.py       # lancer un script Python
pip install torch     # installer une bibliothèque
```

### Script vs Programme

Un **script** est un fichier de code qui s'exécute directement (ligne par ligne). Un **programme** est plus structuré, souvent compilé, avec plusieurs fichiers interconnectés. En ML, on écrit surtout des scripts Python — des fichiers `.py` qu'on lance avec `python mon_script.py`.

## Git : le versioning du code

### Le problème que Git résout

Imaginez un document Word partagé entre 10 personnes. Chacun fait des modifications. Très vite, vous avez `rapport_v2_final_FINAL_v3_corrigé.docx`. C'est le chaos.

**Git** résout ce problème pour le code. C'est un **système de contrôle de version** : il enregistre chaque modification, qui l'a faite, quand, et pourquoi. Vous pouvez revenir en arrière à tout moment, comparer deux versions, et fusionner le travail de plusieurs personnes.

### Les concepts clés de Git

**Repository (repo)** — le dossier du projet, avec tout son historique de modifications. C'est comme un dossier partagé, mais avec une mémoire parfaite de tous les changements jamais faits.

**Commit** — une "photo" du code à un instant donné, avec un message qui explique ce qui a changé. C'est comme faire `Ctrl+S` mais en plus intelligent : vous nommez chaque sauvegarde. Exemple : "Corrige le bug de connexion" ou "Ajoute la page d'accueil".

**Branch** (branche) — une copie parallèle du code pour travailler sur une fonctionnalité sans toucher au code principal. Comme faire une copie de votre document pour tester des modifications risquées, sans abîmer l'original.

**Merge** (fusion) — réintégrer le travail d'une branche dans le code principal. Git fusionne automatiquement les changements, et signale les conflits si deux personnes ont modifié la même ligne.

**Pull Request (PR)** — une demande de fusion. Avant de merger, on soumet ses modifications pour que les collègues relisent et valident. C'est le processus de revue de code.

**Main / Master** — la branche principale, le code "officiel" en production. On ne modifie jamais main directement — on crée une branche, on travaille, on fait une PR, on merge.

<Callout type="tip" title={"Git en une phrase"}>
Git = le "suivi des modifications" de Word, mais pour du code, en 1000× plus puissant, et capable de gérer le travail simultané de centaines de développeurs.
</Callout>

### GitHub / GitLab

**GitHub** et **GitLab** sont des plateformes web qui hébergent des repos Git. C'est comme Google Drive pour le code : le repo vit en ligne, accessible à toute l'équipe. GitHub appartient à Microsoft et héberge la majorité des projets open-source mondiaux.

## API et SDK : communiquer avec des services

### API (Application Programming Interface)

Une **API** est une **porte d'entrée** pour communiquer avec un service depuis du code. Quand vous utilisez ChatGPT sur le site web, vous passez par l'interface graphique. Quand un développeur veut intégrer GPT dans son application, il passe par l'**API** d'OpenAI.

Concrètement, une API c'est une adresse web (URL) à laquelle on envoie des requêtes structurées, et qui renvoie des réponses structurées :

```
→ Requête : "Traduis 'bonjour' en anglais" (envoyée à api.openai.com)
← Réponse : "hello"
```

Un **endpoint** est une URL spécifique de l'API. Par exemple :
- `api.openai.com/v1/chat/completions` → pour le chat
- `api.openai.com/v1/embeddings` → pour les embeddings

### SDK (Software Development Kit)

Un **SDK** est une **bibliothèque de code** fournie par un service pour faciliter l'utilisation de son API. Sans SDK, vous devriez écrire des requêtes HTTP brutes — complexe et error-prone. Le SDK fait le travail pour vous.

```python
# Sans SDK (requête HTTP brute) — pénible
import requests
response = requests.post(
    "https://api.openai.com/v1/chat/completions",
    headers={"Authorization": "Bearer sk-..."},
    json={"model": "gpt-4", "messages": [{"role": "user", "content": "Bonjour"}]}
)

# Avec SDK — simple
from openai import OpenAI
client = OpenAI()
response = client.chat.completions.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "Bonjour"}]
)
```

Les SDKs des providers d'IA :
| Provider | SDK Python | Installation |
|---|---|---|
| OpenAI | `openai` | `pip install openai` |
| Anthropic | `anthropic` | `pip install anthropic` |
| Mistral | `mistralai` | `pip install mistralai` |
| Google | `google-genai` | `pip install google-genai` |

### REST / webhook

**REST** est le style d'architecture le plus courant pour les APIs web : on envoie des requêtes HTTP (GET pour lire, POST pour créer) et on reçoit du JSON en retour. La quasi-totalité des APIs d'IA sont REST.

Un **webhook** est l'inverse d'une API : au lieu que vous alliez chercher l'information, le service vous la **pousse** automatiquement quand un événement se produit. Exemple : Stripe vous envoie un webhook quand un paiement est reçu.

## L'écosystème de développement

### Package manager (gestionnaire de paquets)

Un **package manager** installe et gère les bibliothèques de code. En Python, c'est **pip** :

```bash
pip install torch          # installe PyTorch
pip install transformers   # installe la bibliothèque Hugging Face
pip install -r requirements.txt  # installe toutes les dépendances d'un projet
```

En JavaScript, c'est **npm** ou **yarn**. L'équivalent dans la vie courante : l'App Store, mais pour des bibliothèques de code.

### Virtual environment (environnement virtuel)

Un **environnement virtuel** isole les dépendances d'un projet. Pourquoi ? Parce que deux projets peuvent avoir besoin de versions différentes de la même bibliothèque. Le projet A a besoin de PyTorch 2.0, le projet B de PyTorch 1.9 — sans isolation, c'est le conflit.

```bash
python -m venv mon_env    # créer un environnement
source mon_env/bin/activate  # l'activer
pip install torch         # installe torch DANS cet environnement uniquement
```

**Conda** est une alternative populaire en ML (gère aussi les bibliothèques C/CUDA, pas seulement Python).

### Variable d'environnement / fichier .env

Les **variables d'environnement** stockent les informations sensibles (clés API, mots de passe) en dehors du code. On ne met **jamais** une clé API directement dans le code (sinon elle se retrouve sur GitHub, visible par tous).

```bash
# fichier .env (JAMAIS commité sur Git)
OPENAI_API_KEY=sk-abc123...
DATABASE_URL=postgres://user:pass@host/db
```

```python
# dans le code, on lit la variable
import os
api_key = os.getenv("OPENAI_API_KEY")
```

### JSON et YAML

**JSON** et **YAML** sont des formats pour stocker des données structurées — l'équivalent d'un tableau Excel, mais en texte :

```json
{
  "nom": "GPT-4",
  "parametres": 1800000000000,
  "multimodal": true
}
```

```yaml
nom: GPT-4
parametres: 1800000000000
multimodal: true
```

JSON est plus strict (accolades, guillemets). YAML est plus lisible (indentation). Les deux sont omniprésents dans la config des projets.

## Infrastructure et déploiement

### Frontend / Backend

Le **frontend** est ce que l'utilisateur voit et touche (l'interface web, les boutons, les formulaires). Le **backend** est ce qui tourne sur le serveur (la logique métier, la base de données, les appels aux modèles d'IA). Quand vous posez une question à ChatGPT :
- Le **frontend** affiche la page web et capture votre message
- Le **backend** reçoit le message, appelle le modèle GPT, et renvoie la réponse

### Base de données (SQL, NoSQL)

Une **base de données** stocke les données de façon structurée et interrogeable. **SQL** (PostgreSQL, MySQL) organise les données en tables (comme Excel). **NoSQL** (MongoDB, Redis) est plus flexible, stocke des documents JSON ou des paires clé-valeur.

En IA, les **vector databases** (Pinecone, Weaviate, Qdrant) stockent des embeddings pour la recherche sémantique (utilisées dans le RAG -- Retrieval-Augmented Generation).

### Docker / Container

**Docker** emballe une application avec **toutes ses dépendances** dans un "container" — une boîte isolée qui fonctionne de façon identique partout. C'est la solution au classique "ça marche sur mon ordi mais pas sur le serveur".

```bash
docker run -p 8080:80 mon-application  # lance l'application dans un container
```

Analogie : un container Docker est comme un plat surgelé — tout est pré-emballé et prêt à consommer, peu importe la cuisine dans laquelle vous le réchauffez.

### CI/CD (Continuous Integration / Continuous Deployment)

Le **CI/CD** automatise le processus entre "le développeur pousse du code" et "le code tourne en production" :

1. **CI** (Continuous Integration) — à chaque commit, des tests automatiques vérifient que le code fonctionne
2. **CD** (Continuous Deployment) — si les tests passent, le code est automatiquement déployé en production

C'est comme une chaîne de montage automatisée : le développeur pousse son code, les robots vérifient et déploient.

### Staging vs Production

- **Production** (prod) — l'environnement réel, celui que les vrais utilisateurs utilisent
- **Staging** — une copie de la production pour tester les changements avant de les mettre en prod
- **Local** (localhost) — votre ordinateur, pour développer et tester

Un **port** est un numéro qui identifie un service sur une machine. `localhost:3000` = "mon ordinateur, service numéro 3000". C'est comme un numéro d'appartement dans un immeuble (l'immeuble = la machine, l'appartement = le port).

### Cloud vs On-premise

- **Cloud** — votre code tourne sur les serveurs de quelqu'un d'autre (AWS, Google Cloud, Azure). Vous payez à l'usage.
- **On-premise** (on-prem) — votre code tourne sur vos propres machines physiques, dans vos locaux.

Pour l'entraînement de LLM, le cloud domine (accès flexible à des GPU). Pour l'inférence en production, c'est un choix stratégique : cloud (flexible, coûteux à long terme) vs on-prem (investissement initial, économique à long terme).

## Debugging et qualité

### Logs

Les **logs** sont les messages que le programme écrit pendant qu'il tourne — comme un journal de bord. Quand quelque chose plante, les logs sont le premier endroit où on cherche :

```
[2025-01-15 14:32:01] INFO: Serveur démarré sur le port 8080
[2025-01-15 14:32:05] INFO: Requête reçue de l'utilisateur #42
[2025-01-15 14:32:06] ERROR: Modèle introuvable: gpt-5-turbo
```

### Bug / Debug / Stack trace

Un **bug** est un comportement non voulu du programme. **Debugger** (déboguer) c'est trouver et corriger le bug. Une **stack trace** est le message d'erreur détaillé que le programme affiche quand il plante — elle indique exactement quelle ligne de code a échoué et pourquoi.

### Post-mortem

Un **post-mortem** est un document rédigé après un incident en production (panne, bug critique, fuite de données). Il répond à : **que s'est-il passé, pourquoi, quel a été l'impact, et comment on empêche que ça se reproduise.**

C'est l'équivalent du retour d'expérience (RETEX) — mais dans la culture tech, le post-mortem est formalisé et souvent **public**. Google, Meta, Cloudflare publient régulièrement leurs post-mortems pour que toute l'industrie apprenne de leurs erreurs.

Structure typique :
1. **Résumé** — "Le service X a été indisponible pendant 3h le 15 janvier"
2. **Timeline** — chronologie minute par minute de l'incident
3. **Cause racine** — le vrai problème sous-jacent (pas juste le symptôme)
4. **Impact** — combien d'utilisateurs touchés, revenus perdus
5. **Actions correctives** — ce qu'on met en place pour que ça n'arrive plus

<Callout type="info" title={"Culture blameless"}>
Un bon post-mortem est **blameless** (sans accusation) — on cherche les causes systémiques, pas un coupable. "Le déploiement n'avait pas de test automatique" plutôt que "Jean a cassé la prod". Cette culture encourage la transparence : les gens signalent les problèmes au lieu de les cacher.
</Callout>

### Open source / Licence

Un projet **open source** a son code visible par tous, librement utilisable et modifiable. Mais "open source" ne signifie pas "sans règles" : chaque projet a une **licence** qui définit ce qu'on peut faire :

| Licence | Vous pouvez... |
|---|---|
| MIT | Tout faire, même usage commercial |
| Apache 2.0 | Tout faire, avec protection brevets |
| GPL | Utiliser, mais vos modifications doivent aussi être open source |
| Propriétaire | Seulement ce que le contrat autorise |

Llama (Meta) et Mistral utilisent des licences permissives (Apache 2.0 ou similaire). GPT et Claude sont propriétaires — vous accédez au service via API, mais le code du modèle n'est pas disponible.

## Concepts techniques transversaux

### Latence / Throughput / Cache

- **Latence** — le temps entre la requête et la réponse. "Le modèle met 200ms à répondre" = 200ms de latence.
- **Throughput** (débit) — le nombre de requêtes traitées par seconde. "Le serveur gère 1000 requêtes/seconde" = throughput de 1000 req/s.
- **Cache** — une mémoire temporaire rapide qui stocke les résultats fréquents pour éviter de les recalculer. Quand votre navigateur charge une page web plus vite la deuxième fois, c'est grâce au cache.

### Sandbox

Un **sandbox** (bac à sable) est un environnement isolé où on peut expérimenter sans risque. Un agent IA qui exécute du code le fait souvent dans un sandbox — si le code est malveillant, il ne peut pas affecter le système réel.

### Token (en sécurité)

En dehors du contexte NLP (où token = morceau de mot), un **token** en sécurité est une clé d'authentification — une longue chaîne de caractères qui prouve votre identité. Votre clé API `sk-abc123...` est un token. On dit "révoquer un token" = annuler la clé.

<Callout type="tip" title={"Le vocabulaire évolue vite"}>
La tech invente du jargon constamment. L'important n'est pas de tout connaître, mais de savoir **poser la question**. Quand un ingénieur utilise un terme inconnu, demandez : "Tu peux expliquer ce que ça veut dire concrètement ?" — les bons ingénieurs savent traduire.
</Callout>
