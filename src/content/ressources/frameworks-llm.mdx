## Plus qu'une IA qui parle

Un LLM tout seul, c'est un parleur sympa dans un salon vide. Il peut te raconter des trucs, mais il ne sait rien faire de concret. Ajoute une base de données ? Un outil de calcul ? Une source de vérité ? **Maintenant** c'est utile.

Le vrai pouvoir des LLMs, c'est quand ils sont connectés au monde réel : quand ils peuvent chercher dans tes documents, appeler des APIs, exécuter du code, et enchaîner des étapes de raisonnement. Et pour ça, il faut un **framework d'orchestration**.

LangChain règne sur le prototypage. LlamaIndex domine le RAG (Retrieval-Augmented Generation). Haystack brille en production. DSPy réinvente le prompt engineering. Chacun a sa force, et savoir lequel utiliser quand, c'est une compétence clé.

<Callout type="info" title={"Le besoin d'orchestration"}>
Une application LLM typique ne se limite pas à appeler une API. Elle doit : récupérer des documents pertinents (RAG), gérer des conversations multi-tours, appeler des outils externes, gérer les erreurs et les retries, structurer les sorties, et enchaîner plusieurs étapes de raisonnement. Sans framework, tu réinventes la roue à chaque projet.
</Callout>

## LangChain

### Présentation

[LangChain](https://www.langchain.com/) est le framework d'orchestration LLM le plus populaire, créé par Harrison Chase fin 2022. C'est le framework que tout le monde connaît et que beaucoup adorent (ou détestent -- les opinions sont fortes).

- **Abstraction des LLMs** : interface unifiée pour [OpenAI](https://platform.openai.com/docs), [Anthropic](https://docs.anthropic.com/), modèles locaux, etc.
- **Chains** : composition séquentielle d'opérations LLM
- **Agents** : LLMs qui **décident** quels outils utiliser et quand
- **Retrieval** : intégration avec des bases vectorielles pour le RAG
- **[LangGraph](https://langchain-ai.github.io/langgraph/)** : framework d'agents basé sur des graphes d'état (la v2 de LangChain en quelque sorte)

### Composants clés

- **LLM/Chat Models** : wrappers pour tous les fournisseurs de modèles
- **Prompts** : templates de prompts avec variables
- **Output Parsers** : extraction structurée des réponses (JSON, Pydantic)
- **Tools** : intégration d'outils (recherche web, calcul, APIs)
- **Vector Stores** : intégration avec [Pinecone](https://www.pinecone.io/), [Weaviate](https://weaviate.io/), [Chroma](https://www.trychroma.com/), [FAISS](https://github.com/facebookresearch/faiss)

Documentation : [docs.langchain.com](https://python.langchain.com/docs/get_started/introduction)
GitHub : [langchain-ai/langchain](https://github.com/langchain-ai/langchain) (90K+ stars)

### LangSmith et LangGraph

- **[LangSmith](https://smith.langchain.com/)** : plateforme de monitoring et debugging des applications LLM. Indispensable pour comprendre ce qui se passe dans tes chains en production.
- **[LangGraph](https://langchain-ai.github.io/langgraph/)** : framework d'agents stateful avec graphes de flux. Plus contrôlable que les agents LangChain classiques, c'est vers là que tout converge.

<Callout type="tip" title="LangChain vs LangGraph">
LangChain classique utilise des "chains" séquentielles et des "agents" autonomes. LangGraph offre plus de contrôle avec des **graphes d'état explicites**, permettant des workflows complexes avec boucles, branchements conditionnels et parallélisme. Pour les applications de production, **LangGraph est désormais recommandé** par l'équipe LangChain elle-même.
</Callout>

## LlamaIndex

### Présentation

[LlamaIndex](https://www.llamaindex.ai/) (anciennement GPT Index) se spécialise dans le **RAG** et la connexion des LLMs aux données. Si ton use case c'est "je veux que mon LLM réponde à partir de MES documents", c'est LlamaIndex qu'il te faut.

- **Data Connectors** : ingestion de données depuis [160+ sources](https://llamahub.ai/) (PDF, bases de données, APIs, Slack, Notion)
- **Indexing** : création d'index vectoriels et structures sur les données
- **Query Engine** : moteur de requete combinant recherche et génération
- **Agents** : agents de recherche d'information capables de naviguer dans les données

### Points forts

- **Spécialisation RAG** : le framework le plus complet pour le retrieval-augmented generation
- **Data agents** : agents spécialisés dans l'interrogation de données structurées et non-structurées
- **Multi-modal** : support des documents avec images
- **[LlamaParse](https://docs.llamaindex.ai/en/stable/llama_cloud/llama_parse/)** : parsing intelligent de documents complexes (tableaux, graphiques)

Documentation : [docs.llamaindex.ai](https://docs.llamaindex.ai/)
GitHub : [run-llama/llama_index](https://github.com/run-llama/llama_index) (35K+ stars)

## Haystack

### Présentation

[Haystack](https://haystack.deepset.ai/), développé par deepset (Allemagne), est un framework NLP orienté **pipelines** et **production** :

- **Pipelines** : composition modulaire de composants (retrievers, readers, generators)
- **Production-ready** : conçu dès le départ pour le déploiement en production, pas juste le prototypage
- **Flexible** : support de nombreux backends ([Elasticsearch](https://www.elastic.co/), FAISS, [Weaviate](https://weaviate.io/))
- **Évaluation** : outils intégrés d'évaluation des pipelines RAG

### Haystack 2.0

La [version 2.0](https://docs.haystack.deepset.ai/docs/intro) a été réécrite avec une architecture de pipelines beaucoup plus flexible :
- Composants connectables via input/output types
- Support natif des LLMs (OpenAI, Anthropic, local)
- Intégration avec les modèles [Hugging Face](https://huggingface.co/)
- Pipeline builder visuel

Documentation : [docs.haystack.deepset.ai](https://docs.haystack.deepset.ai/)
GitHub : [deepset-ai/haystack](https://github.com/deepset-ai/haystack) (17K+ stars)

<Callout type="paper" title="Haystack vs LangChain">
Haystack se distingue par son approche plus structurée et production-ready. Là où LangChain offre une grande flexibilité (parfois au prix de la complexité et du "spaghetti code"), Haystack impose une structure de pipeline claire qui facilite le déploiement et la maintenance. Si tu construis pour la prod, Haystack mérite un regard sérieux.
</Callout>

## DSPy

### Présentation

[DSPy](https://dspy-docs.vercel.app/) (Declarative Self-improving Language Programs), créé par Omar Khattab à Stanford, adopte une approche **radicalement différente** des autres frameworks. L'idée : et si on arrêtait d'écrire des prompts à la main ?

- **Programmation déclarative** : on déclare la structure du programme, pas les prompts
- **Optimisation automatique** : les prompts et les exemples few-shot sont optimisés automatiquement
- **Modules composables** : ChainOfThought, ReAct, RAG comme modules réutilisables
- **Compilateurs** : transforment un programme déclaratif en prompts optimisés

### Le paradigme DSPy

Au lieu d'écrire manuellement des prompts (et de passer des heures à les tweaker), DSPy :
1. Définit des **signatures** (entrées/sorties attendues)
2. Compose des **modules** (CoT, RAG, multi-hop reasoning)
3. **Compile** le programme avec un optimiseur qui trouve les meilleurs prompts et exemples
4. Évalue automatiquement sur un jeu de données de développement

C'est comme passer du SQL au machine learning : au lieu de spécifier **comment** faire, tu spécifies **quoi** faire, et le système optimise le comment.

### Avantages

- **Pas de prompt engineering manuel** : l'optimiseur trouve les meilleurs prompts. Fini les heures à tweaker des virgules.
- **Robustesse** : les programmes sont moins fragiles que les prompts manuels
- **Portabilité** : change de modèle (GPT-4 -> Claude -> Mistral) sans réécrire les prompts
- **Reproductibilité** : les programmes sont des objets Python vérifiables et testables

Documentation : [dspy-docs.vercel.app](https://dspy-docs.vercel.app/)
GitHub : [stanfordnlp/dspy](https://github.com/stanfordnlp/dspy) (17K+ stars)
Paper : [DSPy (Khattab et al., 2023)](https://arxiv.org/abs/2310.03714)

<Callout type="warning" title="Maturité des frameworks">
L'écosystème des frameworks LLM évolue **très** rapidement. Les APIs changent fréquemment, de nouveaux frameworks apparaissent, et les meilleures pratiques se redéfinissent constamment. Ne t'attache pas trop rigidement à un seul framework. Comprends les **concepts** (RAG, agents, orchestration) et le framework spécifique devient interchangeable.
</Callout>

## Choisir le bon framework

| Besoin | Framework recommandé |
|--------|---------------------|
| RAG et recherche dans des docs | [LlamaIndex](https://www.llamaindex.ai/) |
| Agents complexes avec logique conditionnelle | [LangGraph](https://langchain-ai.github.io/langgraph/) |
| Pipeline de production robuste | [Haystack](https://haystack.deepset.ai/) |
| Optimisation automatique de prompts | [DSPy](https://dspy-docs.vercel.app/) |
| Prototypage rapide et exploration | [LangChain](https://www.langchain.com/) |
| Application simple (chatbot basique) | Appel direct à l'API ([OpenAI](https://platform.openai.com/docs), [Anthropic](https://docs.anthropic.com/)) |

## Autres outils notables

L'écosystème ne s'arrête pas à ces quatre frameworks. Voici d'autres outils qui méritent ton attention :

- **[Semantic Kernel](https://github.com/microsoft/semantic-kernel)** (Microsoft) : framework d'orchestration pour .NET et Python. Le choix si tu es dans l'écosystème Microsoft.
- **[CrewAI](https://github.com/crewAIInc/crewAI)** : framework d'agents multi-rôles collaboratifs. Chaque agent a un "rôle" et ils travaillent ensemble.
- **[AutoGen](https://github.com/microsoft/autogen)** (Microsoft) : framework d'agents conversationnels multi-agents
- **[Instructor](https://github.com/jxnl/instructor)** : extraction structurée de données depuis les LLMs. Simple, élégant, efficace.
- **[Outlines](https://github.com/outlines-dev/outlines)** : génération structurée (JSON, regex) avec des LLMs. Garantit que la sortie respecte un schéma.
