## Le problème fondamental de l'évaluation

On a besoin d'évaluer les modèles. Donc on crée des tests standardisés. Le problème ? Ces tests peuvent **mentir**. Pas intentionnellement, mais de nombreuses manières subtiles : contamination des données d'entraînement, optimisation ciblée des scores, saturation rapide, et surtout, le fossé énorme entre un score de benchmark et une performance en conditions réelles.

C'est un problème qui touche toute l'industrie IA et qui rend la comparaison entre modèles beaucoup plus difficile qu'il n'y paraît.

<Callout type="info" title="La loi de Goodhart en action">
"Quand une mesure devient un objectif, elle cesse d'être une bonne mesure." (Loi de Goodhart). Les entreprises optimisent pour les benchmarks. Les modèles obtiennent d'excellents scores. Mais est-ce que les modèles s'améliorent **vraiment** en compétences utiles ? Ou est-ce qu'ils deviennent juste meilleurs à passer des tests ?
</Callout>

## La contamination : quand le modèle a déjà vu l'examen

**La contamination** (data leakage) est le problème le plus insidieux. Le modèle a vu les questions du benchmark dans ses données d'entraînement. Il ne les *résout* pas. Il s'en *souvient*.

Comment ça arrive en pratique ?

**Web scraping massif.** Les benchmarks sont publiés publiquement (c'est nécessaire pour la reproductibilité). Lors du pré-entraînement, le modèle absorbe une portion massive d'internet : pages web, forums, blogs, GitHub. Les questions de benchmark ? Probablement incluses quelque part dans cette masse de données.

**Données synthétiques.** D'autres LLMs génèrent des données d'entraînement. Ils peuvent produire des paraphrases ou des variantes proches des questions de benchmark, sans que personne ne le réalise.

**Discussions en ligne.** Les forums (Reddit, Stack Overflow), les blogs de data science, les tutoriels discutent des benchmarks, reproduisent les questions et les réponses. Ces discussions sont dans les données d'entraînement.

Résultat concret : des études ont montré que les modèles réussissent **10 à 20 points de pourcentage mieux** sur les questions dont on peut confirmer la contamination que sur des questions équivalentes mais non contaminées. C'est une différence massive.

<Callout type="tip" title="Trois méthodes pour détecter la contamination">
**(1) Perplexité anormale** : si un modèle voit une question pour la première fois, il devrait être "surpris" (perplexité élevée). Une perplexité anormalement basse suggère la mémorisation.

**(2) Test par paraphrase** : posez la même question reformulée. Un modèle contaminé échoue souvent sur la paraphrase (il a mémorisé la formulation exacte, pas le concept).

**(3) Recherche de n-grams** : cherchez les séquences exactes du benchmark dans les données d'entraînement. Si des correspondances exactes existent, la contamination est confirmée.
</Callout>

## Le gaming : optimiser les scores sans améliorer le modèle

Au-delà de la contamination (souvent involontaire), il y a l'optimisation délibérée des scores. Pas de la triche au sens strict, mais des pratiques qui gonflent les chiffres.

**Sélection du meilleur checkpoint.** Tu entraînes pendant 10 jours. Tu sauvegardes le modèle chaque jour. Lequel publies-tu ? Celui qui maximise les scores de benchmark. Pas nécessairement le meilleur modèle en général.

**Prompt engineering ciblé.** Tu testes 100 formulations de prompt différentes. Tu trouves celle qui donne le meilleur score sur MMLU. Tu publies avec ce prompt spécifique. Une autre équipe, avec un autre prompt, obtient un score différent. Le "score MMLU" n'est pas une valeur absolue, c'est une valeur **conditionnelle au prompt**.

**Données synthétiques ciblées.** Tu génères des données d'entraînement proches du format des benchmarks (structure QCM, vocabulaire similaire, domaines alignés) et tu les inclus dans le fine-tuning. Le modèle apprend à exceller sur le format du test, pas sur la compétence sous-jacente.

**Optimisation des hyperparamètres.** Tu ajustes température, top-p, nombre de few-shot examples jusqu'à obtenir le meilleur résultat sur le leaderboard.

<Callout type="warning" title="Les scores publiés sont rarement vérifiés de manière indépendante">
Quand une entreprise annonce "Notre modèle atteint 92.5% sur MMLU", c'est généralement dans les conditions les plus favorables qu'elle a trouvées. Une équipe indépendante qui évalue le même modèle obtient souvent **88-90%**. La différence vient des conditions : prompts, température, few-shot setup, version de l'évaluation. Ces détails sont rarement publiés de manière transparente.
</Callout>

## La saturation : quand le test perd son pouvoir

Un benchmark saturé, c'est quand tous les modèles récents obtiennent 90%+ et deviennent indistinguables.

L'état actuel :
- **GSM8K** (maths élémentaires) : 96%+ --- **saturé**
- **ARC-Challenge** (sciences collège) : 96%+ --- **saturé**
- **HellaSwag** (bon sens) : 95%+ --- **saturé**
- **MMLU** (culture générale) : 90%+ --- en voie de saturation

Un benchmark saturé ne dit plus rien. Le modèle A est-il meilleur que le modèle B ? Les deux obtiennent 96%, donc impossible de les distinguer.

Le cycle des benchmarks est devenu prévisible :

1. **Création** : "C'est vraiment difficile !"
2. **Adoption** : tout le monde l'utilise comme référence
3. **Progrès rapide** : les modèles s'améliorent en quelques mois
4. **Saturation** : 12-24 mois plus tard, tous les modèles récents obtiennent 90%+
5. **Obsolescence** : le benchmark ne sert plus qu'à cocher une case
6. **Remplacement** : on crée un nouveau benchmark "plus difficile"
7. Retour à l'étape 1

## Le fossé benchmark-réalité : le vrai problème

Le problème le plus profond n'est ni la contamination ni la saturation. C'est que les **benchmarks ne testent pas ce qui compte en pratique**.

**Interaction prolongée.** Les benchmarks testent : une question, une réponse. La réalité : des conversations de 20 messages où le contexte s'accumule, les instructions évoluent, et le modèle doit s'adapter aux clarifications et aux changements de direction.

**Fiabilité.** Un modèle peut avoir 90% sur un benchmark et être **peu fiable**. Les 10% où il échoue sont imprévisibles. En production, un modèle qui se trompe de manière imprévisible est plus dangereux qu'un modèle légèrement moins bon mais dont les échecs sont prévisibles.

**Suivi d'instructions complexes.** Un QCM c'est simple : quatre options, choisis-en une. Une vraie instruction professionnelle peut avoir 5 conditions, des cas limites, des exceptions, des priorités contradictoires. Les benchmarks ne capturent pas cette complexité.

**Jugement et nuance.** Les situations réelles sont ambiguës. "Est-ce que je devrais ?", "Est-ce éthique ?", "Quelle est la meilleure approche ?" Les benchmarks testent le noir et blanc. La réalité est en nuances de gris.

**Hallucinations en production.** Un modèle peut avoir d'excellents scores et halluciner régulièrement en usage réel. Les benchmarks ne capturent pas la fréquence et la gravité des hallucinations dans un contexte de conversation libre.

**Latence et coût.** Les benchmarks ignorent des facteurs critiques en production : combien de temps pour une réponse ? Combien ça coûte par requête ? Un modèle 3% meilleur qui coûte 10x plus cher n'est pas un meilleur choix.

Exemple concret : un modèle excelle sur MMLU (connaissances factuelles) mais hallucine en continu dans les conversations libres. Un autre a un score MMLU moyen mais est beaucoup plus utile en pratique parce qu'il comprend mieux le contexte, admet son incertitude, et produit des réponses actionnables.

<Callout type="paper" title="Vers une évaluation plus réaliste">
Des benchmarks comme **LiveBench** (qui se renouvelle mensuellement), **SWE-bench** (résoudre de vrais bugs dans de vrais repos GitHub), et **LMArena** (votes humains directs) représentent la direction. L'évaluation **composite** -- benchmarks automatiques + évaluation humaine + cas d'usage réels -- est l'avenir. Aucun chiffre unique ne capture la qualité d'un modèle.
</Callout>

## Les solutions émergentes

**LiveBench** : benchmark mensuel qui se renouvelle. Les questions changent chaque mois, éliminant le risque de contamination à long terme. Le prix : il faut recréer les questions régulièrement.

**LiveCodeBench** : problèmes de code récents issus de compétitions (Codeforces, LeetCode), postérieurs aux données d'entraînement. Pas de contamination possible.

**SWE-bench** : résoudre de **vrais bugs** dans de **vrais repos** GitHub (Django, Flask, scikit-learn). Pas une question abstraite, une tâche réelle : lire le code existant, localiser le problème, écrire un patch qui passe les tests. ~50% pour les meilleurs agents.

**GAIA** : questions nécessitant des outils, du raisonnement multi-étapes, et de la recherche d'information réelle. Pas juste "que sais-tu ?" mais "peux-tu trouver et raisonner ?"

**LMArena** : votes humains directs. Quelle réponse est meilleure ? C'est subjectif, mais c'est le plus proche de la satisfaction utilisateur réelle.

**Red teaming systématique** : combien de façons peux-tu faire échouer le modèle ? Trouver les failles est aussi informatif que mesurer les succès.

<Callout type="warning" ’>
La solution n'est pas de trouver un super-benchmark unique qui mesure tout. C'est d'accepter qu'il faut **de multiples évaluations complémentaires** : benchmarks automatiques (rapidité, reproductibilité), évaluation humaine (qualité perçue, utilité), cas d'usage réels (fiabilité en production), et audits indépendants (vérification des claims). Un modèle qui brille sur **tous** ces fronts est probablement bon. Un modèle qui brille sur un benchmark mais échoue ailleurs ? Les benchmarks mentent.
</Callout>
