## Le problème fondamental

On a besoin d'évaluer les modèles. Donc on crée des tests. Le problème ? Les tests peuvent mentir. Pas intentionnellement, mais de nombreuses façons subtiles — contamination des données, optimisation ciblée, saturation, et finalement, le fossé énorme entre un score de test et une performance réelle.

<Callout type="info" title="La loi de Goodhart en action">
"Quand une mesure devient un objectif, elle cesse d'être une bonne mesure." Les entreprises optimisent pour les benchmarks. Les modèles obtiennent d'excellents scores. Mais est-ce qu'ils s'améliorent vraiment en compétences ?
</Callout>

## La contamination : quand le modèle a déjà vu la réponse

**Contamination** (data leakage). Le modèle a vu les questions du benchmark pendant l'entraînement. Il ne les résout pas. Il se souvient.

Où arrive ça ?

**Web scraping**. Les benchmarks sont publiés publiquement. Lors du pré-entraînement, le modèle absorbe une énorme quantité d'internet. Les questions du benchmark ? Probablement incluses.

**Données synthétiques**. D'autres LLMs génèrent des données d'entraînement. Ils peuvent générer des paraphrases ou des variations des benchmarks.

**Discussions**. Les forums, les blogs discutent des benchmarks. Les questions et réponses sont là.

Résultat ? Les modèles réussissent mieux sur les questions contaminées que non-contaminées. Parfois un écart de 10-20 points.

<Callout type="tip" title="Détecter la contamination : trois méthodes">
(1) Perplexité anormale : si un modèle voit une question pour la première fois, il devrait être surpris. Une perplexité basse = mémorisation.
(2) Paraphrases : posez la même question en d'autres mots. Un modèle contaminé échoue souvent sur la paraphrase.
(3) N-grams : cherchez les sequences exactes du benchmark dans les données d'entraînement. Trouvez des correspondances ? Contamination détectée.
</Callout>

## Le gaming : optimiser pour les scores, pas pour la vérité

Les développeurs de modèles peuvent tricher. Pas avec les données, mais avec l'optimisation.

**Sélection de checkpoints**. Vous entraînez pendant 10 jours. Vous sauvegardez le checkpoint tous les jours. Lequel publiez-vous ? Celui qui maximise les scores de benchmark.

**Prompt engineering ciblé**. Vous essayez 100 prompts différents. Vous trouvez celui qui donne le meilleur score. Vous publiez avec ce prompt. Autre équipe, autre prompt, autre score.

**Données synthétiques ciblées**. Vous générez des données proches du format des benchmarks et les incluez dans le fine-tuning.

**Hyperparamètres**. Vous réglablez jusqu'à obtenir le meilleur résultat sur le leaderboard.

Résultat ? Les scores rapportés ne sont pas complètement faux, mais ils sont optimisés jusqu'à un point où un laboratoire indépendant ne les reproduit pas.

<Callout type="warning" title="Les scores reportés sont rarement vérifiés">
Quand une entreprise publie : "Notre modèle atteint 92.5% sur MMLU", c'est souvent dans les conditions idéales qu'ils ont trouvées. Une autre équipe qui évalue le même modèle de manière indépendante obtient parfois 88-90%. Les conditions (prompts, température, few-shot setup) changent tout.
</Callout>

## La saturation : quand le test perd son pouvoir discriminant

Un benchmark saturé, c'est quand les modèles obtiennent tous 90%+.

- **GSM8K** (math élémentaire) : 96%+ saturé.
- **ARC-Challenge** (sciences collège) : 96%+ saturé.
- **HellaSwag** (bon sens) : 95%+ saturé.
- **MMLU** (culture générale) : 90%+ en voie de saturation.

Un benchmark saturé ne vous dit plus rien. Est-ce que le nouveau modèle A est meilleur que le modèle B ? Les deux obtiennent 96%, donc aucune idée.

Le cycle des benchmarks :

1. Création → "Ça c'est difficile !"
2. Adoption → Tout le monde y court après.
3. Progrès rapide → Les modèles s'améliorent en mois.
4. Saturation → 12-24 mois plus tard, c'est banal.
5. Obsolescence → On crée un nouveau benchmark.
6. Retour à 1.

## Le fossé benchmark-réalité : les tests ne testent pas ce qui compte

Les benchmarks testent des choses isolées, statiques, fermées. La réalité est fluide, dynamique, ouverte.

**Interaction prolongée**. Les benchmarks : une question, une réponse. La réalité : des conversations de 20 messages où le contexte s'accumule et le modèle doit s'adapter.

**Fiabilité**. Un modèle peut avoir 90% sur un benchmark et rester peu fiable. Les 10% où il échoue peuvent être imprévisibles et dangereux.

**Suivi d'instructions**. Un QCM c'est simple : quatre options, choisir une. Une vrai instruction peut avoir 5 conditions différentes, des cas limites, des exceptions.

**Jugement et nuance**. Les situations réelles sont ambiguës. "Devrais-je ?", "Est-ce éthique ?" Les benchmarks testent le noir et blanc.

**Hallucinations en production**. Un modèle peut avoir un très bon score et halluciner régulièrement en usage réel. Les benchmarks ne capturent pas ce problème.

**Latence et coût**. Les performances brutes ignorent : combien de temps ça prend ? Combien ça coûte ? En production, ce compte beaucoup.

Exemples réels : Un modèle excelle sur MMLU (connaissances factuelles) mais hallucine en continu. Un autre a un score moyen mais est beaucoup plus utile car il comprend mieux le contexte et les nuances.

<Callout type="paper" title="Vers une évaluation réelle">
Les benchmarks comme **LiveBench** (qui se renouvelle mensuellement), **SWE-bench** (résoudre de vrais bugs), et **Chatbot Arena** (votes humains directs) sont la direction. L'évaluation composite — benchmarks automatiques + humains + cas réels — est l'avenir.
</Callout>

## Les solutions émergentes

**LiveBench** : benchmark mensuel qui change. Pas de risque de contamination long-terme.

**LiveCodeBench** : problèmes de code récents, vrais contextes.

**SWE-bench** : résoudre des vrais bugs dans des vrais repos GitHub. Pas une question abstraite, une tâche réelle.

**GAIA** : questions nécessitant des outils, du raisonnement multi-étapes, une vraie recherche.

**LMArena** : votes humains directs. Quel modèle a donné la meilleure réponse ?

**Red teaming systématique** : combien de façons pouvez-vous faire échouer le modèle ?

<Callout type="warning" title="Aucun benchmark n'est parfait">
La solution n'est pas un super-benchmark unique. C'est l'acceptation qu'il faut multiple évaluations : benchmarks, humains, cas réels, audits indépendants. Si un modèle brille sur tous ces fronts, il est probablement bon. S'il brille sur un benchmark et échoue ailleurs ? Les benchmarks mentent.
</Callout>
