## Introduction : L'évaluation honnête

L'objectif du machine learning est la **généralisation** : ton modèle doit bien fonctionner sur des données **qu'il n'a jamais vues**. Jamais.

Si tu évalues sur les données qu'il a apprises, il va paraître parfait. Mais en production, il échoue. C'est du triche.

Solution : **réserve des données pour l'évaluation honnête**.



<Callout type="warning" title="C'est la fondation">
Un split rigoureux sans data leakage c'est ce qui sépare un modèle de recherche d'un modèle deployable. Aucune métrique n'est fiable sans ça.
</Callout>

## Les trois ensembles sacrés

### Training Set (70-80%)

Les données sur lesquelles le modèle **apprend vraiment**. Les poids sont ajustés pour minimiser l'erreur sur ces données.

### Validation Set (10-15%)

Les données que tu **regardes pendant le développement**. Tu les utilises pour :
- Choisir les hyperparamètres
- Sélectionner la meilleure architecture
- Décider quand arrêter l'entraînement (early stopping)

Le modèle ne s'entraîne pas directement dessus, tu les **observes** juste.

### Test Set (10-15%)

Les données que tu **ne touches qu'une seule fois, à la fin**. C'est ton estimé **honnête** de la vraie performance. Sacré. Jamais utilisé pour choisir des hyperparamètres ou des architectures.

## Pourquoi trois et pas deux ?

Si tu utilises le même ensemble pour **choisir les hyperparamètres ET estimer la performance finale**, tu introduis un biais optimiste.

Tu finiras par choisir le modèle qui "gagne" sur cet ensemble, potentiellement par chance. Le test set, intact, donne une estimation non biaisée.

C'est invisible mais dévastateur : tu penses avoir un modèle excellent mais il échoue en production.



## Ratios typiques

### Gros datasets (> 100k exemples)

90/5/5 ou 98/1/1. Même 1%, c'est des milliers d'exemples. Suffisant pour l'évaluation.

### Datasets moyens (1k-100k)

70/15/15 ou 80/10/10. Le classique équilibré.

### Petits datasets (< 1k)

Oublie les ratios fixes. Utilise la **cross-validation** (voir ci-dessous) pour utiliser toutes tes données.

<Callout type="tip" title="Toujours une graine aléatoire fixe">
Utilise `random_state=42` ou similaire. Sans ça, chaque exécution crée un split différent et tu ne peux pas reproduire tes résultats. Reproducibilité = critère de base.
</Callout>

## Cross-validation : utiliser 100% des données

Quand tu as peu de données, les splits fixes gaspillent. La **k-fold cross-validation** utilise toutes tes données.

**Algorithme** :

1. Divise les données en k parts égales
2. Pour chaque part i : entraîne le modèle sur les k-1 autres, évalue sur i
3. Performance finale = moyenne (et écart-type) des k résultats

**Valeurs courantes** : k=5 ou k=10. k=5 est un bon compromis.

**Avantage** : chaque donnée est utilisée pour apprendre (k-1 fois) et pour évaluation (1 fois).

**Inconvénient** : k fois plus lent qu'un simple split.

Exemple avec k=5 et 1000 données :

```
Fold 1 : entraîne sur 800, évalue sur 200
Fold 2 : entraîne sur 800 (autres), évalue sur 200
...
Performance = moyenne des 5 résultats
```

## Splits stratifiés : respecter les proportions

En classification avec classes déséquilibrées, un split aléatoire peut créer des folds où une classe rare disparaît ou est surreprésente.

Le **split stratifié** garantit que chaque fold a les mêmes proportions que les données globales.

Exemple : si 20% de tes données sont la classe rare, chaque fold aura aussi 20%.

<Callout type="info" title="Toujours stratifier en classification">
Utilise `StratifiedKFold` ou `stratified_train_test_split`. C'est zéro surcoût et rend les évaluations beaucoup plus stables.
</Callout>

## Time Series Splits : respecter l'ordre temporel

Pour les données **temporelles** (stocks, météo, logs), un split aléatoire c'est un **bug majeur**.

Ça crée du **data leakage temporel** : le modèle s'entraîne sur le futur pour prédire le passé. Ses performances explosent en dev mais s'effondrent en production.

**Time Series Split** : entraîne **toujours** sur le passé, valide sur le futur.

```
Fold 1 : entraîne sur mois 1-3, valide sur mois 4
Fold 2 : entraîne sur mois 1-4, valide sur mois 5
Fold 3 : entraîne sur mois 1-5, valide sur mois 6
...
```

C'est la seule façon juste.

## Data Leakage : le bug silencieux

Le **data leakage** c'est quand l'information du test/validation "fuit" dans l'entraînement. Le modèle a un avantage injuste.

Tes performances paraissent excellentes en dev mais s'effondrent en production. Pire : c'est **invisible**. Le leakage peut survivre des mois avant d'être détecté.

<Callout type="warning" title="Le problème est insidieux">
Tes métriques de test sont excellentes. Tout marche parfaitement... jusqu'à ce que tu déploies. Là, effondrement. Le leakage a sabotée tout ton travail.
</Callout>

## Types de leakage courants

### Leakage direct

Une feature **contient directement l'information cible**. Exemple : prédire si quelqu'un sera hospitalisé en utilisant une feature "numéro de chambre d'hôpital" (qui n'existe que pour les hospitalisés).

### Leakage temporel

Utiliser le futur pour prédire le passé. Splits aléatoires sur données temporelles. Features basées sur la "date de l'événement" qu'on essaie de prédire.

### Leakage de prétraitement

Calculer la normalisation (moyenne/variance) sur **train + test**, puis normaliser. Le test set influence indirectement l'entraînement via les statistiques.

Correct : calculer les stats sur train SEULEMENT, puis les utiliser pour normaliser train ET test.

### Leakage par duplication

Les mêmes données (ou quasi-identiques) dans train et test. Courant en imagerie médicale (plusieurs coupes du même patient) ou NLP (paraphrases du même texte).

### Leakage par groupes

Des données liées (du même patient, du même utilisateur) réparties entre train et test. Le modèle reconnaît le groupe au lieu du pattern qu'on cherche.

Solution : utilise `GroupKFold` pour garder les groupes ensemble.

## Prévention : la checklist

**1. Divise d'abord, puis prétraite**

La séparation doit être la **toute première étape**. Normalisation, imputation, feature engineering : tout sur train set SEULEMENT, puis applique aux autres.

**2. Utilise des pipelines**

scikit-learn Pipelines garantissent que les transformations sont appliquées correctement et ne fuitent pas.

**3. Cherche les features suspectes**

Une corrélation quasi parfaite avec la cible ? Leakage probable. Examine la provenance.

**4. Respecte les groupes**

Si des exemples sont liés (du même client, du même image médicale), garde-les ensemble. `GroupKFold`.

**5. Valide en conditions réelles**

Simule la production : le modèle n'a accès qu'aux données disponibles au moment de la prédiction.

## Checklist anti-leakage

Avant de crier victoire :

1. Split avant tout prétraitement
2. Aucune feature n'utilise des données du futur ou du test
3. Exemples liés sont dans le même split
4. Statistiques de normalisation viennent UNIQUEMENT du train
5. Performance cohérente entre validation et test (écart suspect = alerte rouge)
6. Les features importantes sont plausibles en production
7. Performance en production correspond aux métriques de test

## Pourquoi c'est important

Un split rigoureux, c'est l'évaluation honnête de ton modèle. Aucune metrique n'est fiable sans ça. Et le data leakage t'es invisible. Tu construis en aveugle si tu ne fais pas attention. C'est fondamental.
