## L'idée élégante: tout est text-to-text

Imagine un seul modèle qui résout:
- **Sentiment**: lire "Excellent film!" → générer "positif"
- **Traduction**: lire "Hello" → générer "Bonjour"
- **Résumé**: lire [article de 1000 mots] → générer [3 lignes]
- **Similarité de phrases**: lire "Phrase A" et "Phrase B" → générer "0.95" (score numérique sous forme texte)
- **Extraction**: lire "Pierre habite à Paris" → générer "Pierre | Paris"

Même architecture. Même processus d'entraînement. Zéro architecture spécialisée.

**T5** (Text-to-Text Transfer Transformer, Google 2020) pose cette idée simple et élégante: **toutes les tâches NLP sont juste des transformations texte → texte**. Rien d'autre.

### Comment ça marche en pratique

Tu donnes un **préfixe** qui indique la tâche, puis le texte:

```
Input:  "sentiment: Excellent film!"
Output: "positif"

Input:  "translate English to French: Hello"
Output: "Bonjour"

Input:  "summarize: [article de 1000 mots]"
Output: "[résumé en 3 lignes]"

Input:  "stsb: Phrase A ||| Phrase B"
Output: "3.8"                          (similarité sur échelle 0-5)
```

La tâche est encodée **dans le texte**, pas dans l'architecture.

<Callout type="tip" title="Avantages de l'unification">
- Un seul modèle à entraîner (pas d'architectures spécialisées)
- Multi-task learning naturel (mélange les tâches pendant l'entraînement)
- Flexible: ajoute une nouvelle tâche? Ajoute un nouveau préfixe. Pas de recablâge.
- Transférable: un modèle entraîné sur sentiment + traduction + résumé gère aussi des variantes
</Callout>

## Architecture: l'encoder-decoder hybride

T5 utilise une architecture complète **encoder-decoder**:

- **Encoder**: regarde le texte d'entrée entièrement (attention bidirectionnelle). "Comprend" le contexte.
- **Decoder**: génère le texte de sortie token par token (attention causale), avec une "cross-attention" vers l'encoder. "Parle" en regardant l'encoder.

C'est différent de:
- **BERT** (encoder-only): juste compréhension, pas génération
- **GPT** (decoder-only): juste génération, pas compréhension d'une entrée structurée

T5 allie les deux: compression via l'encoder + génération via le decoder.

### Les variantes (et pourquoi les tailles importent)

```
T5-Small:  60M params    → rapide, laptop, peu de données
T5-Base:   220M params   → bon équilibre, GPU grand public
T5-Large:  770M params   → state-of-the-art sur benchmarks
T5-11B:    11 milliards  → énorme, nécessite multi-GPU
```

### Positional embeddings relatifs: le trick technique

BERT/GPT utilisent des embeddings de position **absolus**: position 0, 1, 2, etc.

T5 utilise des **embeddings de position relatifs**: chaque tête d'attention apprend comment traiter "à distance 1" vs "à distance 5". La distance relative, pas la position absolue.

Avantage: peut généraliser à des séquences plus longues que le training. BERT entraîné sur 512 tokens? Peine si tu lui donnes 1000. T5 généralise mieux.

<Callout type="info" title="Architecturales: encoder-decoder vs decoder-only">
Pour des tâches de transformation (traduction, résumé), encoder-decoder est **plus efficient**. L'encoder "compresse" l'entrée, le decoder "génère" la sortie. Comme un compilateur: parsing (encode) + generation (decode).

Pour la flexibilité brute (tout-en-un), decoder-only gagne. D'où GPT dominer, et T5 rester spécialisé.
</Callout>

## Coin technique: entraînement par "span corruption"

BERT masque des **tokens individuels**. T5 masque des **spans** (séquences contiguës).

```
Original:        "Le chat noir dort sur le canapé"
Spans masqués:   "Le <extra_id_0> dort sur <extra_id_1>"
Cible (ce que le modèle doit générer):
"<extra_id_0> chat noir <extra_id_1> le canapé"
```

Chaque `<extra_id_0>`, `<extra_id_1>`, etc. est un token spécial unique qui marque "ici se passe quelque chose".

### Pourquoi les spans au lieu de tokens?

1. **Cible plus courte**: prédire "chat noir" (2 tokens) est plus rapide que prédire token-par-token après masquage. Entraînement plus efficace.

2. **Cohérence**: masquer "chat noir" (un concept) force le modèle à **raisonner sur des chunks**, pas des mots isolés. C'est plus pertinent pour des tâches comme le résumé ("Erase whole clauses, reconstruct meaning") vs juste "guess the masked word".

3. **Naturel pour la génération**: T5 est un encoder-decoder pour la génération. Masquer spans = plus proche de "générer du texte" que "prédire un mot".

### Corpus C4: la référence silencieuse

T5 s'entraîne sur **C4** (Colossal Clean Crawled Corpus): 750 Go de texte anglais tirés de Common Crawl, soigneusement filtré:
- Supprime les doublons (deduplicate)
- Enlève le contenu offensif (heuristiques basiques)
- Enlève le HTML brut et les artefacts

C4 est devenu **l'étalon-or** pour pré-entraîner des LLM. Presque tous (GPT-3, BLOOM, LLaMA, Llama2) utilisent C4 ou des dérivés.

<Callout type="info" title="Qualité vs taille">
Plutôt que 10 TB de web brut (beaucoup de bruit), T5 choisit 750 GB de web filtré. Moins mais mieux. Résultat: plus efficace qu'un corpus 10x plus gros mais non filtré.
</Callout>

## La vraie contribution: les ablations systématiques

Le papier T5 est intéressant, mais pas pour le modèle T5 lui-même. C'est pour la **recherche systématique** :

Les auteurs testent **des dizaines de décisions d'entraînement** séparément:

```
Quelle architecture gagne?
 - Encoder-decoder (T5)
 - Decoder-only (GPT-style)
 - Prefix LM (input visible au début, puis causality)

Quel objectif de pré-entraînement?
 - MLM (BERT-style: tokens individuels)
 - Span corruption (ici, contiguës)
 - Denoising (autre reconstruction)

Taille des spans?
 - 3 mots en moyenne
 - 8 mots
 - 15 mots

Combien de données?
 - 10 GB
 - 100 GB
 - 750 GB
...
```

Pour chaque variable, ils mesure l'impact. C'est pas "on innove!", c'est "on comprend **quels sont les leviers réels**".

### La leçon: l'ablation scientifique

Cette approche méthodique ("ablation study") est devenue le **standard** en recherche NLP sérieuse. Plutôt que présenter "Voici notre idée cool!", c'est "Voici l'impact mesurable de chaque choix."

Ça a changé comment les papiers NLP se font. Beaucoup d'innovation = juste "meilleures hyper-parameters" ou "meilleures données", pas d'architecture réelle.

## FLAN-T5: le super-modèle (instruction tuning)

**FLAN-T5** (Google 2022): prends T5, puis fine-tune-le sur **centaines de tâches NLP différentes**, mais formulées en **instructions naturelles**:

```
Au lieu de juste "sentiment: Excellent!",
FLAN reçoit:
"Détermine le sentiment du texte suivant. Réponse: positive ou negative.
Texte: Excellent restaurant!"

Sortie: "positive"
```

FLAN a vu traduction ("Traduis du français..."), summarization ("Résume le texte..."), Q&A ("Réponds à la question..."), etc. Des centaines d'instructions différentes.

**Résultat**: FLAN-T5 généralise fantastiquement en **zero-shot**. Donne une tâche complètement nouvelle (pas vue pendant l'entraînement) avec une instruction naturelle? FLAN-T5 la comprend et répond correctement souvent.

C'est comme si T5 avait appris **le pattern général** de comment interpréter une instruction.

<Callout type="tip" title="Instruction tuning vs in-context learning">
Deux approches différentes d'adaptation:

**Instruction Tuning** (FLAN-T5):
- Fine-tune le modèle sur de nombreuses instructions
- Coûteux (entraînement), mais très robuste
- Généralise bien au zero-shot

**In-Context Learning** (GPT-3/4):
- Pas de fine-tuning, juste des exemples dans le prompt
- Coûteux en tokens de prompt, moins robuste
- Adapte en runtime (instantané)

Complémentaires. FLAN-T5 pour fiabilité, GPT-4 pour flexibilité.
</Callout>

## Encoder-decoder vs decoder-only: le choix stratégique

Deux philosophies pour les LLM:

### T5 (encoder-decoder)
**Forces**:
- Excellent pour **transformer** du texte (traduction, résumé)
- L'encoder comprend l'entrée complète avant génération
- Génération plus "ciblée" (pas distractions)

**Faiblesses**:
- Deux modules = plus complexe à entraîner à énorme échelle
- Moins flexible (encode-decode est structuré)
- Moins naturel pour le dialogue libre ou la génération ouverte

### GPT (decoder-only)
**Forces**:
- Simple: une pile de couches, flux direct
- Scalable à 175B+ paramètres facilement
- Flexible: fait tout (classification, génération, résumé, dialogue) uniformément

**Faiblesses**:
- Moins efficace pour les transformations structurées (traduction)
- L'absence d'encodage séparé = moins de compréhension préalable

### Le choix de l'industrie

2018-2022: encoder-decoder et encoder-only compétaient.
2022+: decoder-only a écrasé. Pourquoi?

Parce que **la flexibilité universelle battu l'efficacité ciblée**. Un LLM qui fait tout est plus utile qu'un expert en traduction. Et scaling décoder-only s'est avéré plus simple qu'encoder-decoder.

Mais encoder-decoder a toujours sa place: systèmes NLP spécialisés (traduction automatique, résumé), où la structure aide.

## Pourquoi c'est important

T5 démontre que **l'unification conceptuelle** (tout est text-to-text) apporte clarté et généralité. Le span corruption (vs token MLM) montre que les détails d'entraînement importent. L'étude systématique du papier T5 a établi le **template pour la bonne science NLP**: ablations, mesures contrôlées, reproduction.

FLAN-T5 ajoute: instruction tuning = accès à zéro-shot puissant sans énorme prompt.

T5 reste un excellent choix pour les systèmes de production spécialisés (traduction, résumé, extraction) aujourd'hui encore. Mais pour "un seul LLM pour tout", decoder-only a gagné.
