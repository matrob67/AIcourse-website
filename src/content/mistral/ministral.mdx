## Ministral : l'IA qui tient dans votre poche

Tout le monde parle de modèles de plus en plus gros. Ministral prend le chemin inverse : des modèles suffisamment petits pour tourner **en local**, sur du matériel léger, sans connexion internet.

La famille Ministral (3B, 8B, 14B paramètres) est spécifiquement optimisée pour le **edge computing** (calcul embarqué en périphérie de réseau) : NVIDIA Jetson, GPU RTX grand public, et même des CPU puissants. C'est de l'IA embarquée, pas de l'IA cloud.

### Pourquoi l'edge computing ?

Le cloud n'est pas toujours une option. Quatre situations rendent le déploiement local indispensable :

**Latence critique** : un véhicule autonome qui attend 500ms de réponse du cloud, c'est un accident en puissance. Un robot industriel qui dépend d'internet pour prendre une décision, c'est une ligne de production arrêtée. L'IA doit être en local, au plus près de l'action.

**Confidentialité absolue** : certaines données ne doivent jamais quitter le réseau local. Données médicales, secrets industriels, informations classifiées : avec un modèle embarqué, rien ne transite par internet. La confidentialité est garantie par construction.

**Absence de connectivité** : l'usine en zone rurale n'a pas de fibre. Le bateau en mer n'a pas de 5G. Le site minier en zone isolée n'a pas de Wi-Fi. L'IA embarquée fonctionne hors ligne, point.

**Économie à grande échelle** : déployer de l'IA via le cloud sur 10 000 appareils coûte des millions par an en abonnements API. En embarqué, vous achetez le hardware une fois, et les coûts opérationnels sont quasi nuls après amortissement.

<Callout type="info" title={"L'explosion du edge computing"}>
L'IA embarquée est un marché en pleine explosion : robots industriels, véhicules autonomes, usines connectées, appareils IoT (Internet of Things), smartphones. Ministral est positionné pour répondre à cette demande croissante.
</Callout>

### Ministral 3B : le minimaliste

**3 milliards de paramètres**, c'est la version ultra-légère.

En quantification INT4 (compression mathématique des poids), le modèle ne pèse que 1,5 à 2 Go de RAM. Il fonctionne sur :
- Un NVIDIA Jetson Nano (appareil edge à quelques centaines d'euros)
- Un téléphone haut de gamme
- Un ordinateur portable même ancien

**Cas d'usage** : classification de texte, extraction d'entités simples, génération de réponses courtes, assistants vocaux embarqués.

**Limitations** : pas de raisonnement complexe, pas de génération de texte long, capacités multilingues limitées. C'est un modèle de tâche simple, pas un modèle générique.

### Ministral 8B : le sweet spot

**8 milliards de paramètres**, c'est l'équilibre parfait entre capacité et légèreté.

Il tourne confortablement sur un RTX 4070 (carte graphique grand public). Ses performances sont comparables aux petits modèles cloud, mais il fonctionne entièrement en local.

**Cas d'usage** : chatbot embarqué, analyse de documents locaux, résumé, traduction, aide à la rédaction.

**Le favori des développeurs** : pouvoir prototyper et tester un modèle sur sa propre machine, sans dépendance au cloud, sans clé API, sans latence réseau, c'est un cycle de développement radicalement plus rapide.

<Callout type="tip" title="Le preferé des devs">
Pouvoir itérer localement, sans connexion internet, sans abonnement cloud, avec des réponses en millisecondes : c'est pour ça que Ministral 8B est devenu le modèle de choix pour le développement et le prototypage rapide dans la communauté IA.
</Callout>

### Ministral 14B : la limite haute du edge

**14 milliards de paramètres** représentent la limite haute de ce qu'on peut raisonnablement déployer sur du matériel edge haut de gamme.

Les capacités approchent celles de Mistral Small en mode cloud. Un RTX 4090 ou un bon serveur avec un GPU performant suffisent.

**Cas d'usage** : déploiements on-premise pour les entreprises, serveurs d'inférence légers, stations de travail IA pour les équipes techniques.

### Architecture dense : un choix délibéré

Contrairement aux grands modèles Mistral (qui utilisent l'architecture MoE pour maximiser l'efficacité), les Ministral sont **denses** : tous les paramètres participent à chaque token généré.

Pourquoi ce choix ?

**Simplicité d'inférence** : pas de routage d'experts, pas de logique conditionnelle complexe. Le matériel edge n'est pas conçu pour gérer les subtilités du MoE. La simplicité garantit la fiabilité.

**Prévisibilité** : la consommation de RAM et de CPU est constante et prévisible. Pas de variations selon la requête, pas de pics inattendus. C'est crucial pour les systèmes embarqués où les ressources sont fixes.

**Efficacité à petite échelle** : en dessous de 15 milliards de paramètres, les modèles denses sont souvent plus rapides que les MoE équivalents, car l'overhead du routage d'experts n'est pas compensé par les gains.

### Optimisés pour le matériel NVIDIA

Mistral travaille en étroite collaboration avec NVIDIA pour optimiser les Ministral :

**NVIDIA Jetson** : des profils d'optimisation TensorRT sont disponibles prêts à l'emploi pour les appareils edge Jetson.

**RTX** : les modèles exploitent pleinement les Tensor Cores des cartes RTX pour des gains de performance massifs par rapport à une inférence PyTorch standard.

**TensorRT** : compilez vos modèles pour atteindre la performance maximale sur votre hardware spécifique.

### Formats de distribution

| Format | Usage | Avantage |
|--------|-------|----------|
| Hugging Face (safetensors) | Dev + fine-tuning | Écosystème HF complet |
| GGUF | CPU/GPU via llama.cpp | Quantification intégrée, portable |
| TensorRT | GPU NVIDIA | Performance maximale |
| ONNX | Multi-plateforme | Portabilité vers différents runtimes |

Choisissez le format qui correspond à votre contrainte : portabilité (ONNX), performance maximale (TensorRT), ou simplicité (GGUF).

<Callout type="warning" title="Attentes réalistes">
Un modèle de 3B ou 8B paramètres n'est pas GPT-4. Ne vous attendez pas aux mêmes capacités de raisonnement ou de génération longue. Mais pour des tâches edge spécifiques et bien définies, les performances sont excellentes. La clé : définissez précisément votre tâche et testez si la qualité est suffisante.
</Callout>

### Le calcul économique : edge vs cloud

Une fois le hardware acheté, le coût par token en local approche zéro. Pas d'abonnement mensuel, pas de facture à l'usage.

Pour les déploiements massifs (10 000 robots dans une usine, une flotte de véhicules, une chaîne de production entière), l'économie par rapport au cloud est **colossale**. L'investissement hardware initial est amorti en quelques mois, après quoi chaque inférence est essentiellement gratuite.
