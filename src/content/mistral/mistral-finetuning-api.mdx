## Fine-tuning : adapter Mistral a VOTRE domaine

Un modele generaliste parle un peu de tout, mais n'excelle sur rien de specifique. Pour votre domaine particulier, vous voulez qu'il **parle votre langage**, suive vos conventions, et comprenne les subtilites de votre metier.

Le **fine-tuning** consiste a entrainer un modele Mistral sur **vos propres donnees**. Le resultat : un modele qui comprend votre jargon, suit votre style de reponse, et performe nettement mieux sur vos taches specifiques.

L'analogie : un diplome universitaire donne une culture generale solide. Le fine-tuning, c'est la specialisation professionnelle qui suit : on prend quelqu'un de competent et on le forme aux particularites d'un metier precis.

<Callout type="info" title="Fine-tuning vs Prompt Engineering vs RAG : trois niveaux de personnalisation">
Trois approches complementaires pour adapter un modele a vos besoins :
1. **Prompt Engineering** : ajuster les instructions et le format du prompt. Gratuit, rapide, mais limite.
2. **RAG** : fournir du contexte documentaire au modele a chaque requete. Puissant pour l'information factuelle, mais pas pour le style ou le comportement.
3. **Fine-tuning** : modifier les poids du modele avec vos donnees. Le plus puissant pour le style, le format, et les comportements recurrents.

Le fine-tuning est la bonne approche quand le prompt engineering et le RAG ne suffisent pas : quand vous voulez changer le style de reponse, le format de sortie, ou inculquer des comportements specifiques au modele.
</Callout>

### Modeles supportes pour le fine-tuning

| Modele | Fine-tuning | Cas d'usage typique |
|--------|------------|---------------------|
| Mistral Small | Oui | Classification, applications a haut volume |
| Mistral Medium | Oui | Equilibre qualite/cout pour l'enterprise |
| Mistral NeMo | Oui | Applications multilingues, contexte long |
| Codestral | Oui | Code specialise pour un langage ou framework |

Les modeles fine-tunes sont accessibles via l'API standard avec un identifiant custom. Pas de changement de code pour passer du modele de base a votre modele personnalise.

### Format des donnees : JSONL conversationnel

Chaque ligne du fichier JSONL est un exemple de conversation :

```json
{"messages": [{"role": "system", "content": "Tu es un assistant juridique spécialisé en droit français."}, {"role": "user", "content": "Qu'est-ce que la force majeure ?"}, {"role": "assistant", "content": "La force majeure est un événement imprévisible, irrésistible et extérieur..."}]}
```

**Structure requise** :
- Chaque exemple est une conversation (liste de messages)
- Roles disponibles : `system`, `user`, `assistant`
- Minimum requis : au moins un message `user` et un message `assistant`
- Le message `system` est optionnel mais fortement recommande pour definir le comportement attendu

Chaque exemple doit representer le **comportement ideal** que vous attendez du modele. Si le modele doit repondre dans un format specifique, tous vos exemples doivent utiliser ce format.

### Preparation des donnees : 90% du succes

La qualite de vos donnees d'entrainement determine 90% du succes de votre fine-tuning. C'est la etape la plus importante, et souvent la plus sous-estimee.

**Volume recommande** :
- **Minimum** : 50 a 100 exemples pour observer un premier effet. Suffisant pour ajuster le style et le format.
- **Optimal** : 500 a 5 000 exemples. C'est le sweet spot pour la plupart des cas d'usage en entreprise.
- **Avance** : 10 000+ exemples pour des domaines tres specifiques ou une precision maximale est necessaire.

<Callout type="tip" title="Qualite > Quantite, toujours">
100 exemples excellents produiront un meilleur modele que 10 000 exemples mediocres. Investissez dans la qualite de chaque exemple. Chaque ligne de votre fichier JSONL doit etre un exemple parfait du comportement que vous voulez voir chez votre modele.
</Callout>

**Bonnes pratiques pour la preparation** :
- **Diversite** : couvrez la variete des situations reelles que le modele rencontrera. Si 80% de vos exemples sont des cas simples, le modele ne saura pas gerer les cas complexes.
- **Coherence** : maintenez le meme style et le meme format dans tous vos exemples. Les incoherences confondent le modele.
- **Realisme** : vos exemples doivent ressembler aux vraies requetes de vos utilisateurs. Des exemples artificiels produisent un modele qui fonctionne bien en test mais pas en production.
- **Equilibre des classes** : si vous faites de la classification, assurez-vous que chaque classe est representee proportionnellement. Un desequilibre massif biaisera le modele.

### Hyperparametres : les boutons a regler

**Learning rate** : la vitesse a laquelle le modele apprend. C'est le parametre le plus critique.
- Trop eleve → le modele apprend trop vite et "oublie" ses connaissances de base (catastrophic forgetting)
- Trop bas → le modele n'apprend quasiment rien de vos donnees
- En general, les valeurs par defaut de Mistral fonctionnent bien pour commencer

**Epochs** : le nombre de passes completes sur vos donnees. Typiquement 1 a 3 epochs suffisent. Au-dela, le risque de surapprentissage augmente rapidement.

**Batch size** : le nombre d'exemples traites simultanement. Influence la stabilite de l'entrainement (plus gros = plus stable) et la vitesse (plus gros = plus rapide).

**Warmup ratio** : la fraction de l'entrainement pendant laquelle le learning rate augmente progressivement depuis zero. Ca stabilise le debut de l'entrainement.

### Le processus de bout en bout

1. **Upload des donnees** : envoyez votre fichier JSONL via l'API ou la console
2. **Validation** : Mistral verifie automatiquement le format, la compatibilite et la qualite des donnees
3. **Lancement** : demarrez le job d'entrainement avec vos hyperparametres
4. **Entrainement** : le modele s'entraine sur l'infrastructure Mistral (duree proportionnelle au volume de donnees)
5. **Monitoring en temps reel** : suivez la training loss et la validation loss pendant l'entrainement
6. **Deploiement automatique** : une fois l'entrainement termine, le modele est automatiquement accessible via l'API avec son identifiant custom

### Monitoring pendant l'entrainement

Deux metriques a surveiller comme un faucon :

**Training loss** : mesure l'erreur du modele sur les donnees d'entrainement. Elle doit diminuer regulierement. Si elle stagne, il y a probablement un probleme avec les donnees ou le learning rate.

**Validation loss** : mesure l'erreur sur des donnees que le modele n'a jamais vues. Si elle augmente alors que la training loss continue de baisser, c'est le signe classique du **surapprentissage** : le modele memorise les exemples au lieu d'apprendre les patterns generalistes.

### Evaluation apres l'entrainement

**Evaluation qualitative** : testez votre modele fine-tune sur des cas representatifs que vous connaissez bien. Les reponses sont-elles meilleures que le modele de base ? Le style est-il correct ? Le format est-il respecte ?

**Evaluation quantitative** : si vous disposez de metriques objectives (precision, recall, F1 pour la classification), evaluez sur un test set que le modele n'a jamais vu.

**A/B testing** : comparez votre modele fine-tune avec le modele de base sur de vraies requetes utilisateur. C'est le test le plus fiable car il mesure l'impact reel en conditions de production.

<Callout type="warning" title="Le piege du surapprentissage">
Le risque principal du fine-tuning est le surapprentissage : le modele memorise vos exemples au lieu d'en extraire les patterns generalistes. Symptomes : excellentes performances sur les exemples deja vus, performances mediocres sur des cas nouveaux. Solutions : augmentez le volume de donnees, reduisez le nombre d'epochs, ou ajoutez de la regularisation.
</Callout>

### Cas d'usage reussis en production

**Classification de tickets support** : fine-tuning de Small sur 2 000 tickets categorises → 95% de precision, contre 80% sans fine-tuning. Le modele a appris les categories specifiques de l'entreprise.

**Extraction de donnees de factures** : fine-tuning de Small sur 500 exemples de factures annotees → extraction structuree fiable sans prompts complexes. Le modele sait ou chercher chaque champ.

**Assistant juridique** : fine-tuning de Medium sur 3 000 paires question-reponse juridiques → reponses dans la terminologie et le style juridiques appropriate, avec les references aux articles de loi pertinents.

### Couts du fine-tuning

Le fine-tuning est facture en fonction des tokens traites pendant l'entrainement. Le cout depend du modele choisi, du volume de donnees, et du nombre d'epochs.

Le modele fine-tune resultant est facture legerement plus cher a l'inference que le modele de base. C'est le prix de la personnalisation, mais le gain en qualite justifie generalement cette difference.
