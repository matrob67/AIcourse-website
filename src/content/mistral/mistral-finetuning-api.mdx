## Fine-tuning : adapter Mistral à votre domaine

Un modèle généraliste parle un peu de tout, mais n'excelle sur rien de spécifique. Pour votre domaine particulier, vous voulez qu'il **parle votre langage**, suive vos conventions, et comprenne les subtilités de votre métier.

Le **fine-tuning** consiste à entraîner un modèle Mistral sur **vos propres données**. Le résultat : un modèle qui comprend votre jargon, suit votre style de réponse, et performe nettement mieux sur vos tâches spécifiques.

L'analogie : un diplôme universitaire donne une culture générale solide. Le fine-tuning, c'est la spécialisation professionnelle qui suit : on prend quelqu'un de compétent et on le forme aux particularités d'un métier précis.

<Callout type="info" title="Fine-tuning vs Prompt Engineering vs RAG : trois niveaux de personnalisation">
Trois approches complémentaires pour adapter un modèle à vos besoins :
1. **Prompt Engineering** : ajuster les instructions et le format du prompt. Gratuit, rapide, mais limité.
2. **RAG** : fournir du contexte documentaire au modèle à chaque requête. Puissant pour l'information factuelle, mais pas pour le style ou le comportement.
3. **Fine-tuning** : modifier les poids du modèle avec vos données. Le plus puissant pour le style, le format, et les comportements récurrents.

Le fine-tuning est la bonne approche quand le prompt engineering et le RAG ne suffisent pas : quand vous voulez changer le style de réponse, le format de sortie, ou inculquer des comportements spécifiques au modèle.
</Callout>

### Modèles supportés pour le fine-tuning

| Modèle | Fine-tuning | Cas d'usage typique |
|--------|------------|---------------------|
| Mistral Small | Oui | Classification, applications à haut volume |
| Mistral Medium | Oui | Équilibre qualité/coût pour l'enterprise |
| Mistral NeMo | Oui | Applications multilingues, contexte long |
| Codestral | Oui | Code spécialisé pour un langage ou framework |

Les modèles fine-tunés sont accessibles via l'API standard avec un identifiant custom. Pas de changement de code pour passer du modèle de base à votre modèle personnalisé.

### Format des données : JSONL conversationnel

Chaque ligne du fichier JSONL est un exemple de conversation :

```json
{"messages": [{"role": "system", "content": "Tu es un assistant juridique spécialisé en droit français."}, {"role": "user", "content": "Qu'est-ce que la force majeure ?"}, {"role": "assistant", "content": "La force majeure est un événement imprévisible, irrésistible et extérieur..."}]}
```

**Structure requise** :
- Chaque exemple est une conversation (liste de messages)
- Rôles disponibles : `system`, `user`, `assistant`
- Minimum requis : au moins un message `user` et un message `assistant`
- Le message `system` est optionnel mais fortement recommandé pour définir le comportement attendu

Chaque exemple doit représenter le **comportement idéal** que vous attendez du modèle. Si le modèle doit répondre dans un format spécifique, tous vos exemples doivent utiliser ce format.

### Préparation des données : 90% du succès

La qualité de vos données d'entraînement détermine 90% du succès de votre fine-tuning. C'est l'étape la plus importante, et souvent la plus sous-estimée.

**Volume recommandé** :
- **Minimum** : 50 à 100 exemples pour observer un premier effet. Suffisant pour ajuster le style et le format.
- **Optimal** : 500 à 5 000 exemples. C'est le sweet spot pour la plupart des cas d'usage en entreprise.
- **Avancé** : 10 000+ exemples pour des domaines très spécifiques où une précision maximale est nécessaire.

<Callout type="tip" title="Qualité > Quantité, toujours">
100 exemples excellents produiront un meilleur modèle que 10 000 exemples médiocres. Investissez dans la qualité de chaque exemple. Chaque ligne de votre fichier JSONL doit être un exemple parfait du comportement que vous voulez voir chez votre modèle.
</Callout>

**Bonnes pratiques pour la préparation** :
- **Diversité** : couvrez la variété des situations réelles que le modèle rencontrera. Si 80% de vos exemples sont des cas simples, le modèle ne saura pas gérer les cas complexes.
- **Cohérence** : maintenez le même style et le même format dans tous vos exemples. Les incohérences confondent le modèle.
- **Réalisme** : vos exemples doivent ressembler aux vraies requêtes de vos utilisateurs. Des exemples artificiels produisent un modèle qui fonctionne bien en test mais pas en production.
- **Équilibre des classes** : si vous faites de la classification, assurez-vous que chaque classe est représentée proportionnellement. Un déséquilibre massif biaisera le modèle.

### Hyperparamètres : les boutons à régler

**Learning rate** : la vitesse à laquelle le modèle apprend. C'est le paramètre le plus critique.
- Trop élevé : le modèle apprend trop vite et "oublie" ses connaissances de base (catastrophic forgetting, ou oubli catastrophique)
- Trop bas : le modèle n'apprend quasiment rien de vos données
- En général, les valeurs par défaut de Mistral fonctionnent bien pour commencer

**Epochs** : le nombre de passes complètes sur vos données. Typiquement 1 à 3 epochs suffisent. Au-delà, le risque de surapprentissage augmente rapidement.

**Batch size** : le nombre d'exemples traités simultanément. Influence la stabilité de l'entraînement (plus gros = plus stable) et la vitesse (plus gros = plus rapide).

**Warmup ratio** : la fraction de l'entraînement pendant laquelle le learning rate augmente progressivement depuis zéro. Ça stabilise le début de l'entraînement.

### Le processus de bout en bout

1. **Upload des données** : envoyez votre fichier JSONL via l'API ou la console
2. **Validation** : Mistral vérifie automatiquement le format, la compatibilité et la qualité des données
3. **Lancement** : démarrez le job d'entraînement avec vos hyperparamètres
4. **Entraînement** : le modèle s'entraîne sur l'infrastructure Mistral (durée proportionnelle au volume de données)
5. **Monitoring en temps réel** : suivez la training loss et la validation loss pendant l'entraînement
6. **Déploiement automatique** : une fois l'entraînement terminé, le modèle est automatiquement accessible via l'API avec son identifiant custom

### Monitoring pendant l'entraînement

Deux métriques à surveiller comme un faucon :

**Training loss** : mesure l'erreur du modèle sur les données d'entraînement. Elle doit diminuer régulièrement. Si elle stagne, il y a probablement un problème avec les données ou le learning rate.

**Validation loss** : mesure l'erreur sur des données que le modèle n'a jamais vues. Si elle augmente alors que la training loss continue de baisser, c'est le signe classique du **surapprentissage** (overfitting) : le modèle mémorise les exemples au lieu d'apprendre les patterns généralistes.

### Évaluation après l'entraînement

**Évaluation qualitative** : testez votre modèle fine-tuné sur des cas représentatifs que vous connaissez bien. Les réponses sont-elles meilleures que le modèle de base ? Le style est-il correct ? Le format est-il respecté ?

**Évaluation quantitative** : si vous disposez de métriques objectives (précision, recall, F1 pour la classification), évaluez sur un test set que le modèle n'a jamais vu.

**A/B testing** : comparez votre modèle fine-tuné avec le modèle de base sur de vraies requêtes utilisateur. C'est le test le plus fiable car il mesure l'impact réel en conditions de production.

<Callout type="warning" title="Le piège du surapprentissage">
Le risque principal du fine-tuning est le surapprentissage : le modèle mémorise vos exemples au lieu d'en extraire les patterns généralistes. Symptômes : excellentes performances sur les exemples déjà vus, performances médiocres sur des cas nouveaux. Solutions : augmentez le volume de données, réduisez le nombre d'epochs, ou ajoutez de la régularisation.
</Callout>

### Cas d'usage réussis en production

**Classification de tickets support** : fine-tuning de Small sur 2 000 tickets catégorisés, résultat : 95% de précision, contre 80% sans fine-tuning. Le modèle a appris les catégories spécifiques de l'entreprise.

**Extraction de données de factures** : fine-tuning de Small sur 500 exemples de factures annotées, résultat : extraction structurée fiable sans prompts complexes. Le modèle sait où chercher chaque champ.

**Assistant juridique** : fine-tuning de Medium sur 3 000 paires question-réponse juridiques, résultat : réponses dans la terminologie et le style juridiques appropriés, avec les références aux articles de loi pertinents.

### Coûts du fine-tuning

Le fine-tuning est facturé en fonction des tokens traités pendant l'entraînement. Le coût dépend du modèle choisi, du volume de données, et du nombre d'epochs.

Le modèle fine-tuné résultant est facturé légèrement plus cher à l'inférence que le modèle de base. C'est le prix de la personnalisation, mais le gain en qualité justifie généralement cette différence.
