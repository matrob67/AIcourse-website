## Fine-tuning : adapter Mistral à VOTRE domaine

Un modèle généraliste parle un peu de tout. Mais pour votre domaine spécifique, vous voulez qu'il **parle votre langage** et suive vos conventions.

Le **fine-tuning** = entraîner un modèle Mistral sur **vos données propres**. Résultat : il comprend votre jargon, suit votre style, excelle sur votre tâche.

Analogie : la formation généraliste à l'université, le fine-tuning c'est la spécialisation professionnelle.

<Callout type="info" title="Fine-tuning vs Prompt Engineering vs RAG">
Trois niveaux :
1. **Prompt Engineering** : tweaker les instructions
2. **RAG** : donner du contexte au modèle
3. **Fine-tuning** : modifier les poids du modèle

Fine-tuning gagne quand les deux premiers ne suffisent pas : style, format, comportements récurrents.
</Callout>

### Modèles supportés

| Modèle | Fine-tuning | Cas d'usage typique |
|--------|------------|---------------------|
| Mistral Small | Oui | Classification, apps haut volume |
| Mistral Medium | Oui | Enterprise équilibré |
| Mistral NeMo | Oui | Multilingue, long contexte |
| Codestral | Oui | Code spécialisé |

Les modèles fine-tunés sont accessibles via l'API standard avec un ID custom.

### Format des données

JSONL : chaque ligne = un exemple conversationnel.

```json
{"messages": [{"role": "system", "content": "Tu es un assistant juridique spécialisé en droit français."}, {"role": "user", "content": "Qu'est-ce que la force majeure ?"}, {"role": "assistant", "content": "La force majeure est un événement imprévisible, irrésistible et extérieur..."}]}
```

**Structure requise** :
- Chaque exemple = une conversation (liste de messages)
- Rôles : `system`, `user`, `assistant`
- Min : un `user` + un `assistant`
- `system` : optionnel mais recommandé

### Préparation des données

La qualité des données = 90% du succès du fine-tuning.

**Volume recommandé** :
- **Min** : 50-100 exemples (voir un effet initial)
- **Optimal** : 500-5000 (plupart des cas)
- **Avancé** : 10k+ (domaines très spécifiques)

<Callout type="tip" title="Qualité > Quantité">
100 exemples excellents > 10 000 mauvais. Investissez dans la **qualité**. Chaque exemple = exemple du comportement désiré.
</Callout>

**Bonnes pratiques** :
- **Diversité** : couvrez la variété des situations réelles
- **Cohérence** : même style/format dans tous les exemples
- **Réalisme** : ressemblent aux vraies requêtes utilisateurs
- **Équilibre** : si c'est de la classification, équilibrez les classes

### Hyperparamètres

**Learning rate** : la vitesse d'apprentissage. Trop haut = instabilité/oubli. Trop bas = peu d'apprentissage.

**Epochs** : nombre de passes sur les données. 1-3 = typique. Au-delà = risque de surapprentissage.

**Batch size** : combien d'exemples traités simultanément. Influence la stabilité et la vitesse.

**Warmup ratio** : fraction de l'entraînement où le learning rate augmente progressivement.

### Le processus

1. **Upload des données** : fichier JSONL via API ou console
2. **Validation** : Mistral vérifie le format, la compatibilité
3. **Lancement** : job entraînement démarre avec vos hyperparams
4. **Entraînement** : sur l'infrastructure Mistral (durée = dépend du volume)
5. **Monitoring** : voyez la loss en temps réel
6. **Déploiement** : modèle automatiquement accessible via API une fois terminé

### Monitoring pendant l'entraînement

**Training loss** : doit diminuer régulièrement. Stagnation = problème.

**Validation loss** : si elle augmente alors que training loss baisse = **surapprentissage**. Le modèle mémorise vs. apprendre les patterns.

### Évaluation après l'entraînement

**Évaluation qualitative** : testez sur des cas représentatifs. Est-ce que les réponses sont meilleures que le modèle de base ?

**Évaluation quantitative** : si vous avez des métriques (précision, F1), évaluez le modèle sur un test set.

**A/B testing** : comparez fine-tuned vs. base model sur requêtes réelles.

<Callout type="warning" title="Surapprentissage">
Le risque principal : le modèle **mémorise** les exemples au lieu d'apprendre les patterns. Signes : excellentes perf sur les exemples vus, pourries sur du nouveau. Solutions : plus de data, moins d'epochs, ou regula.
</Callout>

### Cas d'usage réussis

**Classification de tickets support** : fine-tuned Small sur 2000 tickets catégorisés → 95% de précision vs. 80% sans fine-tuning.

**Extraction de données factures** : fine-tuned Small sur 500 exemples → extraction structurée fiable sans prompts complexes.

**Assistant juridique** : fine-tuned Medium sur 3000 QA juridiques → réponses dans le style/terminologie juridique appropriés.

### Coûts

Fine-tuning facturé sur tokens traités. Dépend du modèle, du volume, du nombre d'epochs. Le modèle fine-tuné résultant facturable légèrement plus cher à l'inférence.
