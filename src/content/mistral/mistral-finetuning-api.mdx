## Fine-tuning : adapter Mistral à votre domaine

Un modèle generaliste parle un peu de tout, mais n'excelle sur rien de spécifique. Pour votre domaine particulier, vous voulez qu'il **parle votre langage**, suive vos conventions, et comprenne les subtilites de votre metier.

Le **fine-tuning** consiste à entraîner un modèle Mistral sur **vos propres données**. Le résultat : un modèle qui comprend votre jargon, suit votre style de réponse, et performe nettement mieux sur vos tâches spécifiques.

L'analogie : un diplome universitaire donne une culture générale solide. Le fine-tuning, c'est la spécialisation professionnelle qui suit : on prend quelqu'un de competent et on le forme aux particularites d'un metier précis.

<Callout type="info" title="Fine-tuning vs Prompt Engineering vs RAG : trois niveaux de personnalisation">
Trois approches complementaires pour adapter un modèle à vos besoins :
1. **Prompt Engineering** : ajuster les instructions et le format du prompt. Gratuit, rapide, mais limite.
2. **RAG** : fournir du contexte documentaire au modèle à chaque requete. Puissant pour l'information factuelle, mais pas pour le style ou le comportement.
3. **Fine-tuning** : modifier les poids du modèle avec vos données. Le plus puissant pour le style, le format, et les comportements recurrents.

Le fine-tuning est la bonne approche quand le prompt engineering et le RAG ne suffisent pas : quand vous voulez changer le style de réponse, le format de sortie, ou inculquer des comportements spécifiques au modèle.
</Callout>

### Modèles supportes pour le fine-tuning

| Modèle | Fine-tuning | Cas d'usage typique |
|--------|------------|---------------------|
| Mistral Small | Oui | Classification, applications a haut volume |
| Mistral Medium | Oui | Équilibre qualité/cout pour l'enterprise |
| Mistral NeMo | Oui | Applications multilingues, contexte long |
| Codestral | Oui | Code spécialisé pour un langage ou framework |

Les modèles fine-tunes sont accessibles via l'API standard avec un identifiant custom. Pas de changement de code pour passer du modèle de base à votre modèle personnalise.

### Format des données : JSONL conversationnel

Chaque ligne du fichier JSONL est un exemple de conversation :

```json
{"messages": [{"role": "system", "content": "Tu es un assistant juridique spécialisé en droit français."}, {"role": "user", "content": "Qu'est-ce que la force majeure ?"}, {"role": "assistant", "content": "La force majeure est un événement imprévisible, irrésistible et extérieur..."}]}
```

**Structure requise** :
- Chaque exemple est une conversation (liste de messages)
- Rôles disponibles : `system`, `user`, `assistant`
- Minimum requis : au moins un message `user` et un message `assistant`
- Le message `system` est optionnel mais fortement recommande pour définir le comportement attendu

Chaque exemple doit représenter le **comportement idéal** que vous attendez du modèle. Si le modèle doit répondre dans un format spécifique, tous vos exemples doivent utiliser ce format.

### Préparation des données : 90% du succes

La qualité de vos données d'entraînement déterminé 90% du succes de votre fine-tuning. C'est la étape la plus importante, et souvent la plus sous-estimee.

**Volume recommande** :
- **Minimum** : 50 à 100 exemples pour observer un premier effet. Suffisant pour ajuster le style et le format.
- **Optimal** : 500 à 5 000 exemples. C'est le sweet spot pour la plupart des cas d'usage en entreprise.
- **Avance** : 10 000+ exemples pour des domaines très spécifiques ou une précision maximale est nécessaire.

<Callout type="tip" title="Qualité > Quantite, toujours">
100 exemples excellents produiront un meilleur modèle que 10 000 exemples médiocres. Investissez dans la qualité de chaque exemple. Chaque ligne de votre fichier JSONL doit être un exemple parfait du comportement que vous voulez voir chez votre modèle.
</Callout>

**Bonnes pratiques pour la préparation** :
- **Diversite** : couvrez la variété des situations reelles que le modèle rencontrera. Si 80% de vos exemples sont des cas simples, le modèle ne saura pas gerer les cas complexes.
- **Coherence** : maintenez le même style et le même format dans tous vos exemples. Les incoherences confondent le modèle.
- **Realisme** : vos exemples doivent ressembler aux vraies requetes de vos utilisateurs. Des exemples artificiels produisent un modèle qui fonctionne bien en test mais pas en production.
- **Équilibre des classes** : si vous faites de la classification, assurez-vous que chaque classe est représentée proportionnellement. Un déséquilibre massif biaisera le modèle.

### Hyperparamètres : les boutons a régler

**Learning rate** : la vitesse a laquelle le modèle apprend. C'est le paramètre le plus critique.
- Trop élevé → le modèle apprend trop vite et "oublié" ses connaissances de base (catastrophic forgetting)
- Trop bas → le modèle n'apprend quasiment rien de vos données
- En général, les valeurs par defaut de Mistral fonctionnent bien pour commencer

**Epochs** : le nombre de passes completes sur vos données. Typiquement 1 à 3 epochs suffisent. Au-dela, le risque de surapprentissage augmente rapidement.

**Batch size** : le nombre d'exemples traites simultanément. Influence la stabilité de l'entraînement (plus gros = plus stable) et la vitesse (plus gros = plus rapide).

**Warmup ratio** : la fraction de l'entraînement pendant laquelle le learning rate augmente progressivement depuis zéro. Ça stabilise le début de l'entraînement.

### Le processus de bout en bout

1. **Upload des données** : envoyez votre fichier JSONL via l'API ou la console
2. **Validation** : Mistral vérifié automatiquement le format, la compatibilite et la qualité des données
3. **Lancement** : demarrez le job d'entraînement avec vos hyperparamètres
4. **Entraînement** : le modèle s'entraîne sur l'infrastructure Mistral (duree proportionnelle au volume de données)
5. **Monitoring en temps reel** : suivez la training loss et la validation loss pendant l'entraînement
6. **Déploiement automatique** : une fois l'entraînement termine, le modèle est automatiquement accessible via l'API avec son identifiant custom

### Monitoring pendant l'entraînement

Deux metriques a surveiller comme un faucon :

**Training loss** : mesure l'erreur du modèle sur les données d'entraînement. Elle doit diminuer régulièrement. Si elle stagne, il y a probablement un problème avec les données ou le learning rate.

**Validation loss** : mesure l'erreur sur des données que le modèle n'a jamais vues. Si elle augmente alors que la training loss continue de baisser, c'est le signe classique du **surapprentissage** : le modèle memorise les exemples au lieu d'apprendre les patterns generalistes.

### Évaluation après l'entraînement

**Évaluation qualitative** : testez votre modèle fine-tune sur des cas representatifs que vous connaissez bien. Les réponses sont-elles meilleures que le modèle de base ? Le style est-il correct ? Le format est-il respecte ?

**Évaluation quantitative** : si vous disposez de metriques objectives (précision, recall, F1 pour la classification), evaluez sur un test set que le modèle n'a jamais vu.

**A/B testing** : comparez votre modèle fine-tune avec le modèle de base sur de vraies requetes utilisateur. C'est le test le plus fiable car il mesure l'impact reel en conditions de production.

<Callout type="warning" title="Le piege du surapprentissage">
Le risque principal du fine-tuning est le surapprentissage : le modèle memorise vos exemples au lieu d'en extraire les patterns generalistes. Symptomes : excellentes performances sur les exemples déjà vus, performances médiocres sur des cas nouveaux. Solutions : augmentez le volume de données, reduisez le nombre d'epochs, ou ajoutez de la régularisation.
</Callout>

### Cas d'usage reussis en production

**Classification de tickets support** : fine-tuning de Small sur 2 000 tickets categorises → 95% de précision, contre 80% sans fine-tuning. Le modèle a appris les catégories spécifiques de l'entreprise.

**Extraction de données de factures** : fine-tuning de Small sur 500 exemples de factures annotees → extraction structuree fiable sans prompts complexes. Le modèle sait ou chercher chaque champ.

**Assistant juridique** : fine-tuning de Medium sur 3 000 paires question-réponse juridiques → réponses dans la terminologie et le style juridiques appropriate, avec les références aux articles de loi pertinents.

### Couts du fine-tuning

Le fine-tuning est facture en fonction des tokens traites pendant l'entraînement. Le cout dépend du modèle choisi, du volume de données, et du nombre d'epochs.

Le modèle fine-tune resultant est facture légèrement plus cher à l'inference que le modèle de base. C'est le prix de la personnalisation, mais le gain en qualité justifie généralement cette différence.
