## Function Calling : donner des mains a votre IA

Un LLM seul ne peut que generer du texte. Il ne connait pas la meteo actuelle. Il ne peut pas interroger votre base de donnees. Il ne peut pas envoyer un email ni creer un ticket Jira.

Le **Function Calling** change ca : vous definissez des outils (fonctions), et le modele peut les **appeler** quand c'est pertinent. Soudainement, l'IA a des mains. Elle peut interagir avec le monde reel, recuperer des informations fraaches, et declencher des actions concretes.

### Le flux complet, etape par etape

Comprendre le flux est essentiel, car il y a une subtilite importante :

1. **Vous definissez des outils** : vous decrivez des fonctions avec leur nom, description et parametres (en JSON Schema)
2. **L'utilisateur pose une question** qui necessite un outil ("Quel temps fait-il a Paris ?")
3. **Le modele identifie le bon outil** et genere les arguments sous forme de JSON
4. **Votre application execute la fonction** (appel a une API meteo, requete SQL, etc.)
5. **Vous renvoyez le resultat** au modele
6. **Le modele formule la reponse finale** en integrant le resultat de la fonction

<Callout type="info" title="Point crucial : qui execute quoi ?">
Le modele ne "execute" jamais la fonction lui-meme. Il genere du JSON descriptif : "appelle la fonction X avec les arguments Y". C'est **votre application** qui execute. Le modele est le cerveau qui decide quoi faire. Votre application est les mains qui agissent. Cette separation est fondamentale pour la securite.
</Callout>

### Definition des outils en JSON Schema

Les outils sont definis dans la requete API, au format JSON Schema :

```json
{
  "type": "function",
  "function": {
    "name": "get_weather",
    "description": "Obtient la météo actuelle pour une ville donnée",
    "parameters": {
      "type": "object",
      "properties": {
        "city": {
          "type": "string",
          "description": "Le nom de la ville"
        },
        "unit": {
          "type": "string",
          "enum": ["celsius", "fahrenheit"],
          "description": "Unité de température"
        }
      },
      "required": ["city"]
    }
  }
}
```

**Bonnes pratiques pour la definition d'outils** :
- **Descriptions claires et detaillees** : c'est le principal levier de qualite. Le modele choisit l'outil en se basant sur la description. Une description vague = un mauvais routage.
- **Parametres bien documentes** : decrivez le format attendu, les contraintes, les valeurs par defaut
- **Enum pour les valeurs fixes** : utilisez `enum` quand les valeurs possibles sont connues a l'avance
- **Limitez le nombre d'outils** : au-dela de 20 outils, le modele commence a se tromper plus souvent dans son choix. Si vous en avez plus, regroupez-les logiquement.

### Appels paralleles : plusieurs outils en un seul tour

Mistral supporte l'appel de **plusieurs outils simultanement** dans une seule reponse.

Exemple concret : l'utilisateur demande "Compare la meteo de Paris et Londres".
1. Le modele genere **deux** appels `get_weather` en meme temps (un pour Paris, un pour Londres)
2. Votre application execute les deux en parallele
3. Vous renvoyez les deux resultats
4. Le modele compare et formule sa reponse

**L'avantage** : au lieu de deux allers-retours sequentiels (question → outil 1 → reponse → outil 2 → reponse finale), tout se fait en un seul aller-retour. La latence est divisee par deux ou plus.

<Callout type="tip" title="Parallelisation = gain de latence massif">
Sans appels paralleles, chaque outil necessite un aller-retour complet avec le modele. Avec les appels paralleles, plusieurs outils sont invoques en un seul echange. Pour les cas ou le modele a besoin de 3 ou 4 outils simultanement, le gain de latence est considerable.
</Callout>

### Les quatre patterns d'integration

**Outil unique** : un seul outil defini, le modele l'utilise quand c'est pertinent.
Exemple : un chatbot de support client qui consulte une base de connaissances. Simple, efficace, facile a debugger.

**Multi-outils** : plusieurs outils disponibles, le modele choisit le bon selon la question.
Exemple : un assistant qui gere l'agenda (consulter, creer, modifier), les emails (lire, envoyer), les taches (creer, completer), et la recherche documentaire.

**Outils chaines** : le resultat d'un outil sert d'entree pour un autre.
Exemple : chercher un client par nom → recuperer ses commandes → generer un rapport de synthese. Le modele orchestre la sequence.

**Agentique** : le modele planifie une strategie, execute une sequence d'outils, et adapte son plan en fonction des resultats intermediaires. C'est le niveau le plus avance, ou le modele a une veritable autonomie de decision (voir API Agents).

### Gestion des erreurs : les trois cas problematiques

**Mauvais outil choisi** : le modele selectionne un outil inapproprie pour la question.
Solution : ameliorez les descriptions des outils, ajoutez des exemples dans le system prompt, et assurez-vous que les outils ont des noms et descriptions clairement distincts.

**Parametres invalides** : le modele genere des arguments incorrects ou malformes.
Solution : validez systematiquement les parametres cote serveur avant d'executer. Si les arguments sont invalides, renvoyez un message d'erreur explicatif au modele pour qu'il corrige.

**Echec de la fonction externe** : l'API que vous appelez est indisponible, la donnee n'existe pas, le service est en maintenance.
Solution : renvoyez l'erreur formatee au modele. Il peut alors informer l'utilisateur du probleme ou proposer une approche alternative.

<Callout type="warning" title="Securite : validez TOUJOURS les arguments">
Un utilisateur malveillant peut tenter de manipuler le modele via du prompt injection pour appeler une fonction non autorisee ou avec des arguments dangereux. Regles de base : validez les arguments cote serveur, implementez des controles d'autorisation, et ne permettez jamais d'actions destructives (suppression, envoi) sans confirmation humaine explicite.
</Callout>

### Cas d'usage reels

**E-commerce** : l'utilisateur demande une recommandation → le modele cherche dans le catalogue → verifie le stock → calcule les frais de livraison → propose la commande.

**RH** : un employe demande ses jours de conges restants → le modele interroge le SIRH → verifie les politiques de l'entreprise → propose les dates possibles → cree la demande.

**Data analytics** : un manager pose une question business → le modele interroge la base de donnees → effectue les calculs → genere un graphique → exporte un rapport PDF.

**Finance** : un analyste veut une synthese de marche → le modele recupere les cours de bourse → calcule les indicateurs techniques → genere l'analyse avec les tendances.

**Le Function Calling** transforme le modele d'un generateur de texte passif en un agent capable d'interagir avec le monde reel. C'est la brique fondamentale de toute application IA utile en entreprise.
