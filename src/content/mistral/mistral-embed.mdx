## Mistral Embed : convertir du texte en maths

**Embedding** = convertir du texte en un vecteur (liste de nombres) qui représente le **sens** du texte.

Mistral Embed transforme « Le chat dort sur le canapé » en un vecteur de **1024 nombres**. Deux textes avec du sens proche auront des vecteurs proches, même s'ils utilisent des mots différents.

Pourquoi c'est utile ? Parce que les maths sur des vecteurs = rapide. La recherche sémantique, la clustering, la détection de doublons : tout utilise des embeddings.

<Callout type="info" title="Similarité cosinus">
La mesure standard pour comparer des embeddings = cosinus de l'angle entre les deux vecteurs.
- 1 = identique sémantiquement
- 0 = sans rapport
- -1 = opposé

En pratique, typiquement 0,2 à 0,95.
</Callout>

### Spécifications

| Paramètre | Valeur |
|-----------|--------|
| Dimensions | 1024 |
| Contexte max | 8192 tokens |
| Normalisation | Vecteurs L2-normalisés |
| API uniquement | Oui |
| Batch max | 512 textes/requête |



### Cas d'usage

**Recherche sémantique** : vs. la recherche by keywords, trouvez des documents similaires **en sens**.

Pipeline :
1. Indexez tous vos docs : chacun → embedding → stockez dans une DB vectorielle
2. Requête utilisateur → embedding
3. Cherchez les docs les plus proches (similarité cosinus)
4. Fournissez ces docs comme contexte à un LLM

**RAG** (Retrieval-Augmented Generation) : le pipeline précédent, mais ajoutez un LLM Mistral en bout pour générer la réponse finale.

<Callout type="tip" title="Embeddings + LLM Mistral">
Utilisez Mistral Embed + Mistral Large, c'est cohérent. Les représentations sémantiques et les capacités du LLM sont alignées.
</Callout>

**Classification zero-shot** : comparez l'embedding de votre texte avec les embeddings de textes représentatifs de chaque catégorie. La classe la plus proche = votre classe.

**Clustering** : regroupez automatiquement des textes similaires sans labels. Les modèles de clustering (K-means) travaillent sur les embeddings.

**Détection de doublons** : identifiez des textes similaires ou dupliqués dans une base, même s'ils sont formulés différemment.

### Utilisation de l'API

Envoyez une liste de textes, recevez une liste de vecteurs 1024-dim.

Les textes peuvent être courts (phrases) ou longs (jusqu'à 8K tokens). Recommandation : découper les long docs en **chunks** avant embedding.

### Bonnes pratiques

**Chunking** : découper les longs docs est critique.
- Taille idéale : 200-500 tokens
- Trop court : perd le contexte
- Trop long : dilue l'info spécifique
- Découper aux frontières naturelles (paragraphes, sections)
- Overlap de 10-20% entre chunks

**Préfixes d'instruction** : pour certains cas, préfixer le texte oriente l'embedding.
Exemple : « passage: » pour les docs, « query: » pour les questions. Utile pour le RAG.

<Callout type="warning" title="Limites des embeddings">
Ils capturent le sens global, mais peuvent miser les nuances subtiles, les négations, les logiques complexes. Pour du vrai cross-matching (« est-ce que ce contrat et cette loi sont en contradiction ? »), un LLM cross-encoder donne de meilleurs résultats, mais coûte en latence.
</Callout>

### Intégrations

**Pinecone, Weaviate, Qdrant** : supportent les 1024-dim nativement.

**PostgreSQL + pgvector** : gardez tout en SQL, embeddings inclus.

**ChromaDB** : base vectorielle lightweight, parfait pour le prototypage.
