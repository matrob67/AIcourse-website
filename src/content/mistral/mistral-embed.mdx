## Mistral Embed : convertir du texte en maths

Un **embedding**, c'est la conversion d'un texte en un vecteur (une liste de nombres) qui capture le **sens** de ce texte. Mistral Embed transforme la phrase "Le chat dort sur le canape" en un vecteur de **1024 nombres**. La magie : deux textes qui ont un sens proche auront des vecteurs proches dans l'espace mathematique, meme s'ils utilisent des mots completement differents.

Pourquoi c'est utile ? Parce que comparer des vecteurs, c'est rapide et facile pour un ordinateur. La recherche semantique, le clustering, la detection de doublons, la classification : tous ces cas d'usage reposent sur les embeddings.

Pour prendre une analogie : imaginez que chaque texte est un point sur une carte. Les textes qui parlent du meme sujet se retrouvent dans le meme quartier. Mistral Embed cree cette carte.

<Callout type="info" title="Similarite cosinus : la mesure de reference">
Pour comparer deux embeddings, on utilise la similarite cosinus (le cosinus de l'angle entre les deux vecteurs).
- 1 = les textes sont semantiquement identiques
- 0 = les textes n'ont aucun rapport
- -1 = les textes sont semantiquement opposes

En pratique, les valeurs se situent typiquement entre 0,2 et 0,95.
</Callout>

### Specifications

| Parametre | Valeur |
|-----------|--------|
| Dimensions | 1024 |
| Contexte max | 8192 tokens |
| Normalisation | Vecteurs L2-normalises |
| API uniquement | Oui |
| Batch max | 512 textes/requete |



### Cas d'usage concrets

**Recherche semantique** : contrairement a la recherche par mots-cles (qui cherche des correspondances exactes), la recherche semantique trouve des documents similaires **en sens**. "Comment rescilier mon contrat" trouvera aussi les documents qui parlent de "procedure d'annulation" ou "fin de l'engagement".

Le pipeline classique :
1. **Indexation** : chaque document de votre base → embedding → stockage dans une base vectorielle
2. **Requete** : la question de l'utilisateur → embedding
3. **Recherche** : trouver les documents dont les vecteurs sont les plus proches (similarite cosinus)
4. **Enrichissement** : fournir ces documents comme contexte a un LLM pour generer la reponse

**RAG (Retrieval-Augmented Generation)** : c'est le pipeline precedent, complete par un LLM Mistral en bout de chaine pour generer la reponse finale. Le RAG combine la precision de la recherche vectorielle avec la capacite de generation du LLM.

<Callout type="tip" title="Coherence Mistral Embed + LLM Mistral">
Utilisez Mistral Embed pour la vectorisation et Mistral Large pour la generation. Les representations semantiques et les capacites du LLM sont concues pour fonctionner ensemble. C'est plus coherent que de melanger des fournisseurs differents.
</Callout>

**Classification zero-shot** : vous n'avez pas de donnees d'entrainement pour un classifieur ? Pas de probleme. Comparez l'embedding de votre texte avec les embeddings de descriptions de chaque categorie. La categorie la plus proche = votre classification. Par exemple, embeddings de "plainte client", "question technique", "demande de devis", puis comparez avec chaque email entrant.

**Clustering** : regroupez automatiquement des textes similaires sans avoir besoin de labels prealables. Un algorithme de clustering comme K-means travaille directement sur les embeddings. Utile pour decouvrir des themes dans un corpus de milliers de documents.

**Detection de doublons** : identifiez des textes similaires ou dupliques dans votre base, meme s'ils sont formules differemment. Deux descriptions de produit qui disent la meme chose avec des mots differents ? Leurs embeddings seront tres proches.

### Utilisation de l'API

L'utilisation est simple : envoyez une liste de textes, recevez une liste de vecteurs de 1024 dimensions.

Les textes peuvent etre courts (une phrase) ou longs (jusqu'a 8K tokens). Pour les documents longs, la recommandation est de les decouper en **chunks** avant de generer les embeddings.

### Bonnes pratiques

**Chunking (decoupage)** : c'est l'etape la plus critique pour la qualite de votre systeme de recherche.
- **Taille ideale** : 200 a 500 tokens par chunk
- **Trop court** : le chunk perd son contexte et l'embedding devient peu informatif
- **Trop long** : l'information specifique est diluee dans un embedding trop general
- **Frontières naturelles** : decoupez aux frontieres naturelles du texte (paragraphes, sections, titres)
- **Overlap** : un recouvrement de 10 a 20% entre les chunks evite de couper une idee en deux

**Prefixes d'instruction** : pour certains cas d'usage, prefixer le texte avec un indicateur oriente l'embedding.
Par exemple : "passage: " devant les documents indexes, et "query: " devant les questions des utilisateurs. Ca aide le modele a produire des embeddings optimises pour le matching question-reponse.

<Callout type="warning" title="Limites des embeddings">
Les embeddings capturent le sens global d'un texte, mais peuvent manquer les nuances subtiles : negations ("ce produit n'est PAS bon" vs "ce produit est bon"), logiques complexes, ou implications indirectes. Pour des comparaisons fines ("est-ce que ce contrat et cette loi sont en contradiction ?"), un LLM en cross-encoder donne de meilleurs resultats, au prix d'une latence plus elevee.
</Callout>

### Comparaison avec les alternatives

| Modele | Dimensions | Contexte | Cout | Open-source |

| Mistral Embed | 1024 | 8K | Moyen | Non (API) |
| OpenAI text-embedding-3-large | 3072 | 8K | Moyen | Non (API) |
| BGE-M3 (open) | 1024 | 8K | Gratuit | **Oui** |
| Cohere Embed v3 | 1024 | 512 | Moyen | Non (API) |

Le choix depend de vos contraintes : si la souverainete europeenne compte, Mistral Embed est le choix naturel parmi les solutions API. Si vous voulez du self-hosted gratuit, BGE-M3 est une excellente alternative open-source.

### Integrations avec les bases vectorielles

**Pinecone, Weaviate, Qdrant** : ces bases vectorielles managees supportent nativement les vecteurs 1024 dimensions de Mistral Embed.

**PostgreSQL + pgvector** : si vous voulez tout garder dans votre base SQL existante, pgvector ajoute le support vectoriel a PostgreSQL. Embeddings et donnees relationnelles au meme endroit.

**ChromaDB** : base vectorielle legere et simple, parfaite pour le prototypage et les projets de taille moderee.
