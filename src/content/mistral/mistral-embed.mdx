## Mistral Embed : convertir du texte en maths

Un **embedding**, c'est la conversion d'un texte en un vecteur (une liste de nombres) qui capture le **sens** de ce texte. Mistral Embed transforme la phrase "Le chat dort sur le canapé" en un vecteur de **1024 nombres**. La magie : deux textes qui ont un sens proche auront des vecteurs proches dans l'espace mathématique, même s'ils utilisent des mots complètement différents.

Pourquoi c'est utile ? Parce que comparer des vecteurs, c'est rapide et facile pour un ordinateur. La recherche sémantique, le clustering, la détection de doublons, la classification : tous ces cas d'usage reposent sur les embeddings.

Pour prendre une analogie : imaginez que chaque texte est un point sur une carte. Les textes qui parlent du même sujet se retrouvent dans le même quartier. Mistral Embed crée cette carte.

<Callout type="info" title="Similarité cosinus : la mesure de référence">
Pour comparer deux embeddings, on utilise la similarité cosinus (le cosinus de l'angle entre les deux vecteurs).
- 1 = les textes sont sémantiquement identiques
- 0 = les textes n'ont aucun rapport
- -1 = les textes sont sémantiquement opposés

En pratique, les valeurs se situent typiquement entre 0,2 et 0,95.
</Callout>

### Spécifications

| Paramètre | Valeur |
|-----------|--------|
| Dimensions | 1024 |
| Contexte max | 8192 tokens |
| Normalisation | Vecteurs L2-normalisés |
| API uniquement | Oui |
| Batch max | 512 textes/requête |



### Cas d'usage concrets

**Recherche sémantique** : contrairement à la recherche par mots-clés (qui cherche des correspondances exactes), la recherche sémantique trouve des documents similaires **en sens**. "Comment résilier mon contrat" trouvera aussi les documents qui parlent de "procédure d'annulation" ou "fin de l'engagement".

Le pipeline classique :
1. **Indexation** : chaque document de votre base → embedding → stockage dans une base vectorielle
2. **Requête** : la question de l'utilisateur → embedding
3. **Recherche** : trouver les documents dont les vecteurs sont les plus proches (similarité cosinus)
4. **Enrichissement** : fournir ces documents comme contexte à un LLM pour générer la réponse

**RAG (Retrieval-Augmented Generation, ou génération augmentée par la recherche)** : c'est le pipeline précédent, complété par un LLM Mistral en bout de chaîne pour générer la réponse finale. Le RAG combine la précision de la recherche vectorielle avec la capacité de génération du LLM.

<Callout type="tip" title="Cohérence Mistral Embed + LLM Mistral">
Utilisez Mistral Embed pour la vectorisation et Mistral Large pour la génération. Les représentations sémantiques et les capacités du LLM sont conçues pour fonctionner ensemble. C'est plus cohérent que de mélanger des fournisseurs différents.
</Callout>

**Classification zéro-shot** : vous n'avez pas de données d'entraînement pour un classifieur ? Pas de problème. Comparez l'embedding de votre texte avec les embeddings de descriptions de chaque catégorie. La catégorie la plus proche = votre classification. Par exemple, embeddings de "plainte client", "question technique", "demande de devis", puis comparez avec chaque email entrant.

**Clustering** : regroupez automatiquement des textes similaires sans avoir besoin de labels préalables. Un algorithme de clustering comme K-means travaille directement sur les embeddings. Utile pour découvrir des thèmes dans un corpus de milliers de documents.

**Détection de doublons** : identifiez des textes similaires ou dupliqués dans votre base, même s'ils sont formulés différemment. Deux descriptions de produit qui disent la même chose avec des mots différents ? Leurs embeddings seront très proches.

### Utilisation de l'API

L'utilisation est simple : envoyez une liste de textes, recevez une liste de vecteurs de 1024 dimensions.

Les textes peuvent être courts (une phrase) ou longs (jusqu'à 8K tokens). Pour les documents longs, la recommandation est de les découper en **chunks** (segments de texte) avant de générer les embeddings.

### Bonnes pratiques

**Chunking (découpage)** : c'est l'étape la plus critique pour la qualité de votre système de recherche.
- **Taille idéale** : 200 à 500 tokens par chunk
- **Trop court** : le chunk perd son contexte et l'embedding devient peu informatif
- **Trop long** : l'information spécifique est diluée dans un embedding trop général
- **Frontières naturelles** : découpez aux frontières naturelles du texte (paragraphes, sections, titres)
- **Overlap** : un recouvrement de 10 à 20% entre les chunks évite de couper une idée en deux

**Préfixes d'instruction** : pour certains cas d'usage, préfixer le texte avec un indicateur oriente l'embedding.
Par exemple : "passage: " devant les documents indexés, et "query: " devant les questions des utilisateurs. Ça aide le modèle à produire des embeddings optimisés pour le matching question-réponse.

<Callout type="warning" title="Limites des embeddings">
Les embeddings capturent le sens global d'un texte, mais peuvent manquer les nuances subtiles : négations ("ce produit n'est PAS bon" vs "ce produit est bon"), logiques complexes, ou implications indirectes. Pour des comparaisons fines ("est-ce que ce contrat et cette loi sont en contradiction ?"), un LLM en cross-encoder donne de meilleurs résultats, au prix d'une latence plus élevée.
</Callout>

### Comparaison avec les alternatives

| Modèle | Dimensions | Contexte | Coût | Open-source |

| Mistral Embed | 1024 | 8K | Moyen | Non (API) |
| OpenAI text-embedding-3-large | 3072 | 8K | Moyen | Non (API) |
| BGE-M3 (open) | 1024 | 8K | Gratuit | **Oui** |
| Cohere Embed v3 | 1024 | 512 | Moyen | Non (API) |

Le choix dépend de vos contraintes : si la souveraineté européenne compte, Mistral Embed est le choix naturel parmi les solutions API. Si vous voulez du self-hosted gratuit, BGE-M3 est une excellente alternative open-source.

### Intégrations avec les bases vectorielles

**Pinecone, Weaviate, Qdrant** : ces bases vectorielles managées supportent nativement les vecteurs 1024 dimensions de Mistral Embed.

**PostgreSQL + pgvector** : si vous voulez tout garder dans votre base SQL existante, pgvector ajoute le support vectoriel à PostgreSQL. Embeddings et données relationnelles au même endroit.

**ChromaDB** : base vectorielle légère et simple, parfaite pour le prototypage et les projets de taille modérée.
