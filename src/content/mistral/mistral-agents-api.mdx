## API Agents : les LLM qui agissent

Un LLM classique reçoit une question et renvoie une réponse. C'est un one-shot : une entrée, une sortie. Un **agent IA**, c'est fondamentalement différent : il **observe** une situation, **réfléchit** à la meilleure stratégie, **agit** en utilisant des outils, puis **évalue** le résultat pour décider de la suite.

L'API Agents de Mistral transforme les modèles en agents autonomes capables de résoudre des problèmes complexes en plusieurs étapes, en interagissant avec le monde réel via des outils que vous définissez.

### Du LLM à l'agent : trois niveaux d'autonomie

Pour bien comprendre où se situent les agents, voici les trois niveaux :

**Niveau 1 : LLM classique** — question → réponse. Le modèle génère du texte, point final. Il ne peut ni chercher sur internet, ni consulter une base de données, ni envoyer un email.

**Niveau 2 : LLM + Function Calling** — le modèle peut appeler des outils (fonctions), mais c'est votre application qui contrôle le flux. Le modèle dit "j'ai besoin d'appeler telle fonction", votre code l'exécute, et vous renvoyez le résultat au modèle.

**Niveau 3 : Agent IA** — le modèle **planifie** une stratégie complete, **choisit** quels outils utiliser et dans quel ordre, **execute** les actions, et **adapte** son plan en fonction des résultats intermédiaires. C'est une boucle autonome.

<Callout type="info" title="La boucle agentique">
Observer → Réfléchir → Agir → Évaluer → recommencer. Un agent n'est pas une chaîne linéaire mais une boucle : il revient sur ses pas, corrige ses erreurs, et affine sa stratégie au fur et à mesure.
</Callout>

### Anatomie d'un agent Mistral

Un agent construit avec l'API Mistral se compose de plusieurs éléments :

- **Modèle de base** : le "cerveau" de l'agent (Mistral Large, Magistral, ou même Medium selon la complexité)
- **System prompt** : les instructions qui définissent le rôle de l'agent, ses contraintes et son style
- **Outils** : l'ensemble des fonctions que l'agent peut appeler (API externes, bases de données, calculs...)
- **Mémoire** : l'historique des actions effectuées et des résultats obtenus, qui sert de contexte pour les décisions suivantes
- **Configuration** : limites de sécurité (nombre max d'iterations, politique d'escalade, timeout)

### Les patterns d'architecture agentique

**ReAct (Reasoning + Acting)** : c'est le pattern le plus populaire. Le modèle écrit explicitement son raisonnement avant chaque action, ce qui rend ses décisions transparentes et traçables.

Exemple concret :
1. **Thought** : "L'utilisateur veut le chiffre d'affaires du Q3. Je dois d'abord interroger la base de données financière."
2. **Action** : `query_database(table="revenues", quarter="Q3")`
3. **Observation** : le résultat de la requête revient avec les chiffres
4. **Thought** : "J'ai les données brutes. Je dois maintenant les formater en tableau pour l'utilisateur."
5. **Action** : génère la réponse finale

L'avantage majeur de ReAct : la **traçabilité complète**. En entreprise, pouvoir montrer exactement pourquoi l'agent a pris chaque décision est crucial pour l'audit et la compliance.

**Plan and Execute** : l'agent commence par élaborer un plan complet, puis exécute chaque étape séquentiellement. Plus structuré que ReAct, idéal pour les tâches prévisibles.

1. Analyser le problème et identifier les sous-tâches
2. Produire un plan détaillé (liste ordonnée d'étapes)
3. Exécuter chaque étape du plan
4. Adapter le plan si un résultat intermédiaire le nécessite

**Multi-agent** : plusieurs agents spécialisés collaborent sur un problème, chacun avec son expertise. Un agent coordinateur orchestre le tout, un agent recherche trouve l'information, un agent analyse la traite, un agent rédaction produit le résultat final. C'est comme une équipe où chaque membre a un rôle précis.

<Callout type="tip" title={"ReAct en entreprise : l'audit trail"}>
En contexte enterprise, le pattern ReAct est particulièrement précieux car il génère un audit trail complet du raisonnement de l'agent. Pour les secteurs réglementés (finance, santé, administration), cette traçabilité n'est pas un bonus, c'est une obligation.
</Callout>

### Cas d'usage concrets

**Assistant de recherche** : l'agent reçoit une question complexe, cherche dans plusieurs sources, vérifie la cohérence des informations, synthétise les résultats, et produit un rapport structuré avec sources citées.

**Automatisation métier de bout en bout** : un agent qui traite des demandes clients du début à la fin. Il analyse la demande, interroge les systèmes internes (CRM, ERP), prend une décision selon les règles métier, exécute l'action, et confirme au client.

**Analyse de données** : l'agent reçoit une question business ("Quel produit a la meilleure marge ce trimestre ?"), interroge les bases de données, effectue les calculs, génère des graphiques, et produit un rapport synthétique.

**DevOps intelligent** : l'agent surveille les métriques d'un système, détecte les anomalies, diagnostique la cause probable, et propose ou exécute des actions correctives (scaling, redémarrage de services, rollback).

### Gestion des erreurs et garde-fous

Les agents peuvent dérailler. C'est inévitable avec des systèmes autonomes. D'où l'importance des garde-fous :

**Limite d'itérations** : définissez un maximum (par exemple 20 itérations) pour empêcher les boucles infinies. Si l'agent n'a pas résolu le problème après N étapes, il s'arrête et escalade.

**Gestion des échecs d'outils** : si une fonction externe échoue (API indisponible, données manquantes), l'agent doit savoir quoi faire : tenter une approche alternative, signaler le problème, ou escalader vers un humain.

**Escalade humaine** : pour les décisions critiques ou irréversibles, configurez l'agent pour qu'il demande confirmation plutôt que d'agir seul. Mieux vaut un agent prudent qu'un agent destructeur.

<Callout type="warning" title="Autonomie vs Contrôle : le curseur à ajuster">
Plus un agent est autonome, plus il peut accomplir de choses, mais aussi plus il peut causer de dégâts. La bonne pratique : commencez avec des agents "supervisés" qui demandent confirmation pour les actions risquées (suppression, envoi d'email, modification de données). Augmentez l'autonomie progressivement, une fois que la confiance est établie.
</Callout>

### Sécurité : les principes fondamentaux

**Moindre privilège** : ne donnez à l'agent accès qu'aux outils strictement nécessaires à sa mission. Un agent de support client n'a pas besoin d'accéder à la base de données financière.

**Validation des arguments** : vérifiez systématiquement les paramètres générés par l'agent avant d'exécuter une fonction. Un modèle peut halluciner des arguments invalides ou dangereux.

**Monitoring en temps réel** : surveillez le comportement des agents en production. Détectez les anomalies : trop d'itérations, appels inhabituels, coûts qui explosent.

**Sandboxing** : isolez les actions potentiellement dangereuses dans un environnement contrôlé. L'agent peut proposer une action destructive, mais l'exécution passe par une couche de validation.

### Métriques de monitoring

En production, suivez ces indicateurs pour évaluer la santé de vos agents :

- **Taux de succès** : pourcentage de tâches résolues avec succès
- **Itérations moyennes** : nombre d'étapes par tâche (un indicateur d'efficacité)
- **Coût par tâche** : nombre de tokens consommés (ça chiffre vite)
- **Taux d'escalade** : combien de fois l'agent a demandé l'intervention humaine
- **Latence** : temps moyen de résolution d'une tâche
