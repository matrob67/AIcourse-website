## API Agents : les LLM qui agissent

Un LLM normal reçoit une question et balance une réponse. Un **agent IA**, c'est différent : il **pense**, **planifie**, **agit**, puis **adapte** en fonction des résultats.

L'API Agents de Mistral transforme vos modèles en agents autonomes capables de résoudre des problèmes multi-étapes.

### LLM vs Agent vs Autonome

**LLM classique** : question → réponse. One-shot.

**LLM + Function Calling** : le modèle peut appeler des outils, mais l'app reste en contrôle du flux.

**Agent IA** : le modèle **planifie** la stratégie, **exécute** les actions, et adapte en fonction des résultats intermédiaires.

<Callout type="info" title="La boucle agentique">
Observer → Penser → Agir → Évaluer → répéter.
C'est une boucle, pas une chaîne linéaire.
</Callout>

### Architecture d'un agent

Un agent Mistral = plusieurs pièces :

- **Modèle de base** : le cerveau (Mistral Medium, Large, Magistral)
- **System prompt** : le rôle et les contraintes
- **Outils** : l'ensemble des fonctions disponibles
- **Mémoire** : l'historique des actions et résultats
- **Configuration** : max itérations, politiques de sécurité

### Les patterns

**ReAct (Reasoning + Acting)** : le modèle écrit son raisonnement avant chaque action.

```
Thought: "L'utilisateur veut le CA du Q3. Je dois interroger la DB financière."
Action: query_database(...)
Observation: [résultat]
Thought: "J'ai les données. Je dois les formater."
```

Avantage : **traçabilité complète**. Vous voyez exactement pourquoi l'agent a pris chaque décision. Super en audit/compliance.

**Plan and Execute** : d'abord planifier, puis exécuter.

1. Analyser le problème
2. Produire un plan (liste d'étapes)
3. Exécuter chaque étape
4. Adapter si nécessaire

**Multi-agent** : plusieurs agents spécialisés collaborent.
- Agent coordinateur orchestre
- Agent recherche = trouve l'info
- Agent analyse = traite l'info
- Agent rédaction = produit l'output

<Callout type="tip" title="ReAct pour l'audit">
En entreprise, ReAct c'est or : vous avez un audit trail complet du raisonnement de l'agent. Pour la compliance, c'est crucial.
</Callout>

### Cas d'usage

**Assistant de recherche** : trouve des infos, vérifie les sources, synthétise, produit un rapport.

**Automatisation métier** : traite des demandes clients de bout en bout (analyser → interroger les systèmes → décider → exécuter → confirmer).

**Analyse de données** : reçoit une question métier, interroge les DBs, calcule, génère des graphiques, produit un rapport.

**DevOps intelligent** : surveille, détecte les anomalies, diagnostique, propose/exécute des corrections.

### Gestion des erreurs

Les agents peuvent dérailler. D'où :

**Limite d'itérations** : un max (ex: 20) pour éviter les boucles infinies.

**Gestion des échecs d'outils** : si une fonction échoue, l'agent tente une alternative ou escalade vers un humain.

**Escalade** : pour les décisions critiques, l'agent demande confirmation plutôt que de décider seul.

<Callout type="warning" title="Autonomie vs Contrôle">
Plus un agent est autonome, plus il peut faire de dégâts. Commencez avec des agents « supervisés » (qui demandent confirmation pour les actions risquées). Augmentez l'autonomie progressivement.
</Callout>

### Sécurité

**Moindre privilège** : ne donnez accès qu'aux outils nécessaires.

**Validation des actions** : vérifiez les arguments avant d'exécuter.

**Monitoring** : détectez les comportements bizarres.

**Sandboxing** : isolez les actions dangereuses.

### Monitoring réel

- **Taux de succès** : combien de tâches résolues ?
- **Itérations moyennes** : combien de steps par tâche ?
- **Coût** : combien de tokens consommés ?
- **Escalades** : combien de fois l'agent a demandé help ?
- **Temps** : latence moyenne ?
