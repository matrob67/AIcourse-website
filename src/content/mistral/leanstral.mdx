## Le problème : on fait confiance à du code qu'on ne vérifie pas

En 2026, environ 42% du code en production est généré par l'IA. Pourtant, 96% des développeurs disent ne **pas faire confiance** au code produit par les LLMs. Et seuls 48% prennent le temps de le relire manuellement.

On a donc un paradoxe inquiétant : on utilise massivement l'IA pour coder, tout en sachant qu'elle peut se tromper, et sans vérifier si elle s'est trompée. C'est ce que les chercheurs appellent la **dette de vérification** : un risque silencieux qui s'accumule dans les bases de code du monde entier, comme une bombe à retardement.

<Callout type="warning" title="Le goulot d{"'"}étranglement humain">
Relire du code IA ligne par ligne est lent, coûteux, et ne passe pas à l'échelle. Plus les modèles génèrent de code, plus l'humain devient le facteur limitant. La solution n'est pas de mieux relire, c'est d'automatiser la vérification elle-même.
</Callout>

## L'idée de Mistral : et si l'IA prouvait elle-même que son code est correct?

C'est là qu'intervient **Leanstral**, annoncé en mars 2026. Le concept est simple mais radical : au lieu de générer du code et d'espérer qu'il fonctionne, on génère du code **accompagné d'une preuve mathématique** de sa correction.

### L'analogie qui clarifie tout

Imaginez un architecte qui vous remet les plans d'un immeuble. Aujourd'hui, les LLMs sont comme un architecte très rapide mais parfois distrait : il dessine vite, mais il peut oublier un mur porteur ou mal calculer une charge.

**Leanstral**, c'est comme si cet architecte livrait ses plans avec un **certificat d'un bureau de contrôle indépendant** prouvant que la structure tient debout. Pas une opinion, pas une relecture subjective : une **preuve formelle** vérifiée par un système mathématique (le langage Lean 4). Si le certificat est valide, l'immeuble tient. Point.

### Qu'est-ce que Lean 4?

Lean 4 est un langage de programmation créé par Microsoft Research, conçu spécifiquement pour écrire des **preuves mathématiques** vérifiables par ordinateur. C'est ce qu'on appelle un "assistant de preuve" : vous formulez une proposition ("cette fonction trie toujours correctement une liste"), vous écrivez une preuve, et Lean vérifie automatiquement si cette preuve est valide.

Le point crucial : si la preuve compile dans Lean, elle est **mathématiquement correcte**. Pas "probablement correcte", pas "correcte dans les cas testés" : correcte pour tous les cas possibles, sans exception.

<Callout type="info" title="Tests vs preuves formelles : une différence fondamentale">
Les tests unitaires vérifient des cas particuliers : "pour l'entrée [3, 1, 2], la fonction retourne [1, 2, 3]". La vérification formelle prouve que le code est correct **pour toutes les entrées possibles**. C'est la différence entre "j'ai essayé 1000 combinaisons et ça marchait" et "j'ai la preuve mathématique que ça marchera toujours, quoi qu'il arrive".
</Callout>

## Comment Leanstral fonctionne

Leanstral n'est pas juste un modèle qui génère du texte : c'est un **agent** qui interagit en boucle avec l'environnement Lean 4.

### Le cycle de travail

1. **Leanstral reçoit une spécification** -- par exemple : "prouve que cette fonction de tri produit toujours une liste ordonnée"
2. **Il génère une tentative de preuve** en Lean 4
3. **Lean vérifie la preuve** automatiquement -- soit elle compile (correct), soit Lean renvoie une erreur précise expliquant pourquoi
4. **Leanstral lit l'erreur** et corrige sa preuve en conséquence
5. Retour à l'étape 3, jusqu'à ce que la preuve compile

C'est cette boucle qui rend Leanstral puissant. Le LLM est le créatif qui propose des stratégies de preuve. Lean est le vérificateur implacable qui ne laisse passer aucune erreur. L'un imagine, l'autre valide. Et contrairement à un relecteur humain, Lean ne fatigue jamais et ne fait jamais d'erreur de vérification.

<Callout type="tip" title="Intégration via MCP + Lean">
Leanstral utilise le protocole MCP (Model Context Protocol) avec le serveur `lean-lsp-mcp` pour interagir avec Lean. Cela signifie qu'il peut aussi s'intégrer dans d'autres outils et workflows via Mistral Vibe (la CLI de Mistral). La vérification formelle n'est pas un outil isolé : elle s'insère dans votre workflow existant.
</Callout>


## L'architecture : un MoE ultra-sparse taillé pour la preuve

Sous le capot, Leanstral utilise l'architecture **Sparse Mixture-of-Experts** (MoE), la spécialité de Mistral, poussée à un niveau d'efficacité extrême.

### Les chiffres clés

| Caractéristique | Valeur |
|---|---|
| **Paramètres totaux** | 120 milliards |
| **Paramètres actifs par token** | 6 milliards (seulement 5% !) |
| **Nombre d'experts** | 128 modules spécialisés |
| **Experts activés par token** | 4 sur 128 |
| **Ratio d'efficacité** | 18x -- la capacité d'un modèle 120B au coût d'un modèle 6.5B |

Pour reprendre l'analogie MoE : imaginez un cabinet de 128 consultants spécialisés. Pour chaque question, vous ne consultez que les 4 plus pertinents au lieu de réunir les 128 en salle de réunion. Vous bénéficiez de l'expertise collective, mais ne payez que 4 consultations. Avec 128 experts et top-4, Leanstral pousse le ratio d'efficacité bien au-delà de ce que Mixtral (8 experts, top-2) atteignait.

## Les résultats : des performances impressionnantes à coût réduit

Sur le benchmark **FLTEval** (complétion de preuves formelles sur des dépôts Lean réalistes), Leanstral affiche des résultats qui parlent d'eux-mêmes :

| Configuration | Score FLTEval | Coût |
|---|---|---|
| Leanstral pass@1 | 21.9 | **18 $** |
| Leanstral pass@2 | 26.3 | **36 $** |
| Leanstral pass@4 | 29.3 | **72 $** |
| Leanstral pass@16 | 31.9 | **290 $** |
| Claude Sonnet | 23.7 | **549 $** |
| Claude Opus 4.6 | 33.5 | **1 650 $** |

Pour un budget de 36 dollars, Leanstral **dépasse Claude Sonnet** (26.3 vs 23.7). Pour approcher le niveau de Claude Opus, il faut monter à pass@16 pour 290 dollars, soit tout de même **5.7 fois moins cher**. La vérification formelle joue ici un rôle clé : on peut lancer plusieurs tentatives et garder celle qui compile, sans risque de sélectionner un faux positif.

<Callout type="info" title="Qu{"'"}est-ce que pass@k?">
pass@k signifie : "on lance le modèle k fois, et on garde le meilleur résultat". Normalement, sélectionner le "meilleur" résultat parmi k est un problème en soi (comment savoir lequel est bon?). Mais avec la vérification formelle, c'est gratuit : Lean vérifie automatiquement si chaque preuve est correcte. On lance k tentatives, on garde celle qui compile. C'est un avantage fondamental du paradigme "générer + prouver".
</Callout>

Leanstral bat aussi des modèles open-source généralistes beaucoup plus gros comme GLM5 (744B paramètres), Kimi-K2.5 (1T paramètres) et Qwen3.5 (397B), tout en étant considérablement plus petit et moins cher.

## Accès et déploiement

Leanstral est distribué sous **licence Apache 2.0** (totalement open-source), avec trois modes d'accès :

1. **Via Mistral Vibe** -- la CLI de Mistral. Tapez `/leanstral` pour démarrer un agent de preuve dans votre projet Lean
2. **API gratuite** -- un endpoint temporaire (`labs-leanstral-2603`) pour tester et donner du feedback à Mistral
3. **Auto-hébergé** -- téléchargez les poids et déployez sur votre propre infrastructure

<Callout type="tip" title="Essayer Leanstral">
Pour démarrer rapidement : inscrivez-vous sur [console.mistral.ai](https://console.mistral.ai/codestral/cli), installez Mistral Vibe, puis lancez `/leanstral` dans un projet contenant des fichiers `.lean`.
</Callout>

## Pourquoi c'est un changement de paradigme

Leanstral ne se contente pas d'être un bon modèle de plus. Il change la question fondamentale du code assisté par IA. Jusqu'ici, le workflow était : *générer du code, le relire, espérer qu'il est correct*. Avec la vérification formelle, ça devient : *générer du code, le prouver, avoir la garantie mathématique qu'il est correct*.

C'est particulièrement critique dans trois domaines :
- **Le code critique** (aérospatial, médical, finance, nucléaire) ou un bug peut coûter des vies ou des milliards
- **Les mathématiques** -- Leanstral peut aider les mathématiciens à formaliser et vérifier des preuves complexes
- **La confiance dans l'IA** -- si le code généré est prouvé correct, la question de la fiabilité de l'IA est résolue (au moins pour la partie formellement vérifiée)

L'annonce de Leanstral a coïncidé avec la sortie de Mistral Small 4 et l'entrée de Mistral dans la **Nvidia Nemotron Coalition** pour co-développer des modèles de base open-source.
