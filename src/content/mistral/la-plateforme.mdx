## La Plateforme : votre porte d'entrée à Mistral

**La Plateforme** est l'API de Mistral. Lancée en février 2024, elle permet à n'importe quel développeur d'accéder à tous les modèles Mistral via du code, avec un tier gratuit pour démarrer et une tarification au token pour scaler.

Le point crucial : l'API est **compatible avec le format OpenAI**. Si vous avez déjà du code qui appelle ChatGPT, passer à Mistral revient à changer deux lignes (l'URL de l'endpoint et la clé API). Le reste de votre code ne bouge pas.

### Modèles disponibles

| Modèle | Endpoint | Contexte | Multimodal |
|--------|----------|----------|------------|
| Mistral Small | `mistral-small-latest` | Variable | Non |
| Mistral Medium | `mistral-medium-latest` | Variable | Non |
| Mistral Large | `mistral-large-latest` | 256K | Oui |
| Codestral | `codestral-latest` | Variable | Non |
| Magistral | `magistral-latest` | Variable | Non |
| Mistral Embed | `mistral-embed` | 8K | Non |
| Pixtral | `pixtral-large-latest` | Variable | Oui |

Tous les modèles, une seule API, un seul compte. Les endpoints en `-latest` pointent toujours vers la version courante du modèle, ce qui vous évite de gérer les versions manuellement.

<Callout type="info" title="Compatibilité OpenAI : migration en 2 minutes">
L'API Mistral suit le même format qu'OpenAI (même structure de messages, mêmes paramètres). Migrer depuis ChatGPT ? Changez l'URL de base et la clé API, c'est tout. Les frameworks comme LangChain et LlamaIndex supportent Mistral nativement, donc la migration est encore plus simple si vous les utilisez.
</Callout>

### Tier gratuit : démarrez sans carte bancaire

Vous pouvez **tester gratuitement** tous les modèles d'entrée de gamme sans fournir de carte bancaire :

- Limites : moins de requêtes par jour qu'un tier payant
- Accès : modèles d'entrée de gamme (Small, embeddings)
- Pas de SLA (Service Level Agreement, c'est-à-dire de garantie de disponibilité), pas de support prioritaire

C'est parfait pour les prototypes, les étudiants, les projets personnels, et la découverte de l'écosystème.

**Démarrage en 2 minutes** : allez sur console.mistral.ai, créez un compte, générez une clé API, lancez votre première requête. C'est aussi simple que ça.

### Tarification : payez ce que vous consommez

Le modèle économique est simple : vous payez au token consommé, sans abonnement fixe.

**La logique tarifaire** :
- Les tokens d'entrée (votre prompt) coûtent moins cher que les tokens de sortie (la réponse générée). C'est logique : générer du texte est plus coûteux que le lire.
- Les modèles plus puissants coûtent plus cher par token (Large > Medium > Small)
- Le batch processing (traitement en lot) bénéficie de réductions de 25 à 50%

Les prix évoluent, mais Mistral est généralement compétitif par rapport à OpenAI et Anthropic. Et La Plateforme directe est presque toujours moins chère que de passer par les clouds publics (Azure, AWS, GCP) qui ajoutent leur marge.

### Rate limits : combien de requêtes par minute

- **Gratuit** : limites strictes (suffisantes pour le dev, pas pour la production)
- **Payant** : limites plus élevées, ajustables sur demande
- **Enterprise** : pratiquement sans limites

Les headers HTTP de chaque réponse indiquent votre consommation restante, ce qui facilite la gestion côté code.

### Fonctionnalités clés de l'API

**Chat Completions** : l'endpoint principal. Vous envoyez un historique de messages (system, user, assistant), et Mistral génère la suite de la conversation.

Paramètres courants que vous utiliserez :
- `temperature` : contrôle la créativité (0 = déterministe et factuel, 1 = créatif et varié)
- `max_tokens` : limite la longueur de la réponse
- `top_p` : nucleus sampling pour contrôler la diversité
- `tools` : définissez des fonctions que le modèle peut appeler (function calling)
- `response_format` : forcez le modèle à répondre en JSON structuré

**Streaming** : recevez la réponse token par token via SSE (Server-Sent Events, un protocole de communication serveur vers client en temps réel). Indispensable pour l'expérience utilisateur en temps réel : l'utilisateur voit la réponse apparaître progressivement au lieu d'attendre la fin de la génération.

**Embeddings** : convertissez du texte en vecteurs pour la recherche sémantique. L'endpoint Mistral Embed produit des vecteurs de 1024 dimensions.

**Fine-tuning** : créez des modèles personnalisés entraînés sur vos propres données, directement depuis l'API.



### SDKs officiels et communautaires

- **Python** : `pip install mistralai` (SDK officiel)
- **JavaScript/TypeScript** : `npm install @mistralai/mistralai` (SDK officiel)
- **Go, Rust, Java** : SDKs communautaires disponibles

<Callout type="warning" title="Gardez vos clés API secrètes">
Règle d'or : jamais de clé API dans du code public. Pas dans le HTML du frontend, pas dans un dépôt GitHub, pas dans un fichier de config commité. Utilisez des variables d'environnement. Faites tourner vos clés régulièrement. Une clé API compromise peut générer des milliers d'euros de consommation en quelques heures.
</Callout>

### Dashboard et monitoring

Le dashboard sur console.mistral.ai offre une vue complète de votre utilisation :
- **Consommation en temps réel** : tokens consommés, coûts par modèle, par jour
- **Historique des appels** : latence, taux d'erreur, volumes
- **Alertes configurables** : soyez prévenu si votre consommation dépasse un seuil

### Bonnes pratiques en production

**Retry avec backoff exponentiel** : si vous atteignez un rate limit (erreur 429), n'envoyez pas immédiatement une nouvelle requête. Attendez 1 seconde, puis 2, puis 4, puis 8. L'exponential backoff évite de saturer l'API.

**Caching intelligent** : si la même question revient régulièrement avec la même réponse attendue, mettez le résultat en cache. Ça réduit les coûts et la latence.

**Timeout adapté** : configurez le timeout en fonction du cas d'usage. Un chat court = timeout court (10 secondes). Une génération longue = timeout plus généreux (60 secondes).

**Fallback entre modèles** : si Large est trop lent ou indisponible, basculez automatiquement sur Small. La résilience de votre application ne doit pas dépendre d'un seul modèle.
