## Pourquoi l'observabilité LLM est différente du monitoring classique

Les applications LLM ne sont pas des services web classiques. Ce sont des **chaînes complexes** : récupération de documents (RAG), construction du prompt, appel au LLM, parsing de la réponse, appels d'outils, et potentiellement plusieurs itérations de tout ça. Quand une réponse est mauvaise, le problème peut venir de n'importe quel maillon de la chaîne.

**L'observabilité LLM** va bien au-delà du monitoring traditionnel (le service répond-il ? quelle est la latence ?). Elle permet de **comprendre le contenu** de chaque étape : quels documents ont été récupérés par le RAG ? Le prompt envoyé au LLM contenait-il les bonnes informations ? Combien de tokens ont été consommés ? La réponse est-elle pertinente par rapport au contexte fourni ?

C'est la différence entre savoir que ta voiture roule (monitoring) et comprendre pourquoi elle consomme trop ou prend le mauvais chemin (observabilité).

<Callout type="info" title={"Les trois piliers de l'observabilité LLM"}>
1. **Tracing** : suivre le parcours complet d'une requête à travers tous les composants
2. **Métriques** : mesurer quantitativement la performance (latence, coûts, tokens, taux d'erreur)
3. **Évaluation** : juger la qualité des réponses automatiquement (LLM-as-Judge) ou via feedback humain
</Callout>

## Le tracing : radiographier chaque requête

Une **trace** capture l'exécution complète d'une requête utilisateur, du début à la fin. Elle est composée de **spans** (étapes) organisées hiérarchiquement :

- **Requête utilisateur** (span racine)
  - Span **retrieval** : recherche de documents dans le vector store
  - Span **prompt building** : construction du prompt avec les documents et l'historique
  - Span **LLM call** : appel au modèle (le plus coûteux en temps)
  - Span **parsing** : extraction des informations de la réponse
  - Span **tool call** : appel à un outil externe (API, base de données)

### Ce que chaque span capture

- **Input/Output** : les données qui entrent et sortent de chaque étape. Pour un span LLM, c'est le prompt complet et la réponse complète.
- **Latence** : le temps d'exécution de chaque étape. Indispensable pour identifier les goulots.
- **Métadonnées** : modèle utilisé, température, max_tokens, et tout paramètre pertinent
- **Tokens** : nombre de tokens d'entrée et de sortie. C'est ce qui détermine le coût.
- **Coût estimé** : calculé à partir du nombre de tokens et du pricing du modèle
- **Erreurs** : stack traces, messages d'erreur, codes de statut HTTP

### Les questions auxquelles le tracing répond

Le tracing transforme des problèmes vagues en diagnostics précis :

- "Pourquoi la réponse est mauvaise ?" &rarr; Les documents RAG récupérés étaient non pertinents (le retrieval a échoué, pas le LLM)
- "Pourquoi la latence est élevée ?" &rarr; L'appel LLM prend 5 secondes parce que le prompt fait 15 000 tokens
- "Pourquoi les coûts explosent ?" &rarr; Le prompt système a été modifié et fait maintenant 10x plus de tokens
- "Pourquoi le modèle hallucine ?" &rarr; Le contexte RAG ne contient pas l'information demandée, le LLM invente

Sans tracing, chacun de ces problèmes nécessite des heures de debugging. Avec tracing, la cause est visible en quelques clics.

## LangSmith : la plateforme complète de LangChain

**LangSmith** est la plateforme d'observabilité la plus complète pour les applications LLM. Développée par l'équipe de LangChain, elle offre tracing, évaluation, monitoring et prompt management.

### Tracing de production

- **Auto-instrumentation** : si tu utilises LangChain, tout est tracé automatiquement sans code supplémentaire
- **SDK universel** : instrument n'importe quelle application (pas besoin de LangChain) via le SDK Python/JS
- **Visualisation hiérarchique** : naviguez dans les spans avec le contenu complet de chaque étape
- **Recherche avancée** : filtrez par métadonnées, latence, coûts, feedback utilisateur, tags

### Le workflow traces-vers-évaluation

C'est le pattern le plus puissant de LangSmith :
1. Identifie une trace problématique en production (réponse incorrecte, hallucination)
2. Ajoute-la à un **dataset d'évaluation** avec la correction attendue
3. Utilise ce dataset pour tester les futures versions de votre prompt ou de votre pipeline
4. Les bugs de production deviennent automatiquement des cas de test

Avec le temps, tu construis un dataset d'évaluation qui reflète les vrais problèmes de votre application, pas des cas artificiels.

<Callout type="tip" title={"Le tracing, c'est du debugging continu en production"}>
Ne voyez pas le tracing comme du simple monitoring. C'est un outil de debugging permanent. Chaque trace est une source de données pour améliorer votre système. Les meilleures équipes LLM passent du temps chaque semaine à analyser les traces problématiques et à en tirer des améliorations.
</Callout>

## Arize AI : spécialiste des embeddings et du RAG

**Arize AI** est une plateforme d'observabilité ML/LLM avec des forces uniques dans l'analyse des embeddings.

### Ce qui distingue Arize

- **Analyse d'embeddings** : visualisation UMAP/t-SNE des embeddings de requêtes et de documents. Tu vois littéralement les clusters de requêtes similaires et identifiez ceux qui ne matchent pas bien avec les documents.
- **RAG analysis** : évaluation automatique de la pertinence des documents récupérés, avec métriques détaillées par requête
- **Drift détection** : détecte les changements dans les distributions d'embeddings au fil du temps. Si les requêtes des utilisateurs évoluent, votre RAG doit s'adapter.

### Phoenix (open source)

Arize maintient **Phoenix**, un outil open source pour l'observabilité LLM locale. Il offre tracing, évaluation, analyse d'embeddings et une interface web. C'est un excellent point de départ pour les équipes qui veulent de l'observabilité sans engagement SaaS.

<Callout type="paper" title={"L'analyse d'embeddings pour débugger le RAG"}>
Projeter les embeddings de requêtes et de documents en 2D révèle visuellement les problèmes de retrieval. Tu vois des clusters de requêtes utilisateurs qui n'ont aucun document proche dans le vector store : ce sont des lacunes de votre knowledge base. Tu vois des documents qui ne sont jamais matchés : ce sont des contenus inutiles. C'est un outil de diagnostic puissant que les métriques seules ne peuvent pas offrir.
</Callout>

## Langfuse : l'alternative open source

**Langfuse** est une plateforme d'observabilité LLM **open source** qui a gagné en popularité grâce à sa simplicité et son modèle transparent.

### Fonctionnalités principales

- **Tracing** : via SDK Python/JS ou intégrations natives (LangChain, LlamaIndex, OpenAI SDK)
- **Scores** : système flexible pour attacher des évaluations aux traces (automatiques ou manuelles)
- **Prompt management** : versioning et déploiement de prompts directement depuis l'interface
- **Datasets** : construction de datasets d'évaluation à partir des traces de production
- **Analytics** : dashboards de coûts, latence, qualité, avec filtres et agrégations

### L'avantage open source

- **Self-hosted** : déploie sur ta propre infrastructure (Docker, Kubernetes). Tes données ne quittent jamais tes serveurs.
- **Contrôle total** : les traces contenant des données sensibles restent chez toi, pas chez un tiers
- **Extensible** : API ouverte pour intégrer avec tes outils internes

## Comparaison des plateformes

| Critère | LangSmith | Arize AI | Langfuse |
|---|---|---|---|
| **Type** | SaaS propriétaire | SaaS + Phoenix OSS | Open source + Cloud |
| **Tracing** | Excellent | Très bon | Très bon |
| **Évaluation** | Avancée (datasets, auto) | Avancée (focus RAG) | Bonne |
| **Analyse d'embeddings** | Non | Oui, point fort unique | Non |
| **Prompt management** | Oui | Non | Oui |
| **Self-hosted** | Non | Phoenix (limité) | Oui, pleinement |
| **Coût** | $$$ | $$$ | Gratuit (self-hosted) |

<Callout type="warning" title="Les traces contiennent des données sensibles">
Les traces capturent tes prompts complets et les réponses du LLM. Elles contiennent potentiellement des PII (données personnelles des utilisateurs), des informations business confidentielles, ou des données réglementées. Avant d'envoyer tes traces vers un SaaS cloud, vérifie la conformité RGPD et tes politiques de sécurité. Les solutions self-hosted (Langfuse, Phoenix) offrent un contrôle total sur les données.
</Callout>

## Implémentation pratique : par où commencer

### Ce qu'il faut tracer au minimum

- Chaque **appel LLM** : modèle utilisé, tokens consommés, latence, coût estimé
- Chaque **retrieval** : requête de recherche, documents retournés, scores de similarité
- Les **inputs/outputs** de chaque composant de la chaîne
- Le **feedback utilisateur** : thumbs up/down, corrections manuelles, signalements

### Les métriques clés à monitorer

- **Latence** : P50, P95, P99 (pas juste la moyenne, les outliers comptent)
- **Coûts** : par requête, par utilisateur, par jour. Avec alertes sur les anomalies.
- **Qualité** : score moyen des évaluations automatiques (LLM-as-Judge) et du feedback utilisateur
- **Taux d'erreur** : pourcentage de requêtes qui échouent techniquement
- **Pertinence du retrieval** : scores de similarité des documents RAG récupérés

### Architecture non-bloquante

Le tracing doit être **transparent pour l'utilisateur**. L'envoi des traces ne doit jamais ralentir l'application. Les SDKs modernes utilisent des envois asynchrones avec buffering : les traces sont accumulées en mémoire et envoyées par batch en arrière-plan. Si le backend d'observabilité tombe, l'application continue de fonctionner normalement, les traces sont simplement perdues.

L'observabilité LLM, c'est le monitoring classique adapté aux spécificités des applications à base de modèles de langage. C'est un investissement qui se rentabilise dès les premières traces analysées : chaque bug trouvé via le tracing est un bug qui n'aurait pas été trouvé autrement.
