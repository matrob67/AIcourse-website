## Le modèle qui meurt à petit feu

Vous déployez un modèle excellent. Les métriques sont au top. Tout le monde est content. Puis, semaine après semaine, les utilisateurs commencent à se plaindre. L'accuracy s'est écroulée. Mais personne ne l'a remarqué parce qu'il n'y avait pas de monitoring.

C'est le **data drift** : la distribution des données en production a changé par rapport aux données d'entraînement. Votre modèle a été entraîné sur un monde, mais il opère maintenant dans un monde différent. C'est comme un étudiant qui a révisé les maths de terminale et qui se retrouve à un exam d'université : le sujet a évolué, mais lui est resté au même niveau.

Selon des études internes de Google, les modèles perdent en moyenne 1 à 2% de performance par mois en production. Sans réentraînement régulier, c'est une chute de 10 à 20% en un an. Silencieusement.

<Callout type="info" title="Le vrai coût du drift non détecté">
Si vous découvrez le problème par les plaintes des utilisateurs, c'est déjà trop tard. Les dégâts sont faits : perte de confiance, perte de revenus, atteinte à la réputation. Détecter le drift **avant** les utilisateurs est la raison d'être du monitoring ML. C'est la différence entre prévenir un incendie et éteindre un incendie.
</Callout>

## Types de drift

### Data Drift (Covariate Shift)

Le **data drift** (ou covariate shift) désigne un changement dans la distribution des **données d'entrée** (les features) entre l'entraînement et la production. La relation entre les features et la cible reste la même, mais les features elles-mêmes changent.

**Exemples concrets** :
- Un modèle de prix immobilier entraîné sur des données de Paris reçoit soudainement des requêtes pour des biens en zone rurale : les surfaces, les prix au m2, et les caractéristiques sont complètement différents
- Les utilisateurs d'un site e-commerce changent de démographie au fil du temps (plus jeunes, plus internationaux)
- Un capteur IoT se décalibre progressivement, et les mesures dérivent de quelques pourcents chaque mois

### Concept Drift

Le **concept drift** est plus insidieux : c'est un changement dans la **relation entre les features et la cible**. Même si les données d'entrée ont la même distribution, la bonne réponse a changé.

**Exemples concrets** :
- Un modèle de détection de spam : les spammeurs adaptent constamment leurs techniques. Ce qui était du spam hier ne ressemble pas au spam d'aujourd'hui.
- Un modèle de prédiction de demande : le comportement d'achat change avec les saisons, les tendances, les crises économiques
- Un modèle de score de crédit : les critères de solvabilité évoluent avec la conjoncture économique. En période de crise, les mêmes revenus ne garantissent plus la même capacité de remboursement.

<Callout type="paper" title="La distinction cruciale entre data drift et concept drift">
Data drift et concept drift nécessitent des réponses différentes. Le data drift peut souvent être résolu en réentraînant sur des données plus récentes (le modèle apprend la nouvelle distribution). Le concept drift peut nécessiter un changement de features, d'architecture, ou même de formulation du problème. Identifier le type de drift est la première étape vers une correction efficace. C'est comme en médecine : le bon diagnostic avant le bon traitement.
</Callout>

### Prédiction Drift

Le **prédiction drift** est un changement dans la distribution des **prédictions** du modèle. C'est souvent un symptôme visible de data drift ou de concept drift sous-jacent. Si un modèle de classification binaire qui prédisait habituellement 50/50 se met soudainement à prédire 70% de positifs, quelque chose a changé dans les données ou dans le concept. C'est le signal d'alarme le plus facile à détecter.

### Label Drift (Prior Probability Shift)

Le **label drift** est un changement dans la distribution de la **variable cible** elle-même. Par exemple, la proportion de transactions frauduleuses passe de 1% à 3%. Le modèle, entraîné sur une base de 1%, est mal calibré pour une réalité de 3%.

## Détection du drift

### Méthodes statistiques

Plusieurs tests statistiques permettent de quantifier le drift :

- **Kolmogorov-Smirnov (KS) test** : teste si deux distributions continues sont statistiquement différentes. Simple, interprétable, et largement utilisé. Fonctionne bien pour les features numériques.
- **Chi-squared test** : l'équivalent pour les variables catégorielles. Teste si les fréquences observées diffèrent significativement des fréquences de référence.
- **Population Stability Index (PSI)** : mesure le décalage entre deux distributions. PSI < 0.1 = stable, 0.1-0.2 = changement modéré, > 0.2 = drift significatif. C'est la métrique la plus utilisée en production.
- **Jensen-Shannon Divergence** : mesure symétrique de la distance entre deux distributions, bornée entre 0 et 1. Plus facile à interpréter que la KL-divergence.
- **Wasserstein Distance** : la distance "earth mover" entre distributions. Intuitivement, c'est la quantité minimale de "terre" à déplacer pour transformer une distribution en une autre. Particulièrement sensible aux décalages de localisation.

### Fenêtres de comparaison

Le drift se détecte en comparant deux fenêtres de données :
- **Référence vs Production** : comparer les données de production actuelles avec le dataset d'entraînement. C'est la comparaison la plus naturelle.
- **Fenêtre glissante** : comparer la dernière semaine avec la semaine précédente. Détecte les changements graduels.
- **Page-Hinkley** : test séquentiel qui détecte les changements de moyenne au fil du temps. Adapté au monitoring en temps réel.

<Callout type="tip" title="Choisir la bonne taille de fenêtre">
La taille de la fenêtre de détection est un compromis classique : une fenêtre trop petite génère des faux positifs (le bruit statistique est confondu avec du drift), une fenêtre trop large détecte le drift trop tard (le mal est déjà fait). En pratique, commencez par comparer des fenêtres quotidiennes ou hebdomadaires avec les données de référence. Ajustez ensuite en fonction du taux de faux positifs.
</Callout>

## Métriques de monitoring

### Métriques de données (toujours disponibles)

- **Distribution des features** : histogrammes, quantiles, moyenne, écart-type
- **Valeurs manquantes** : pourcentage de nulls par feature (une augmentation soudaine indique un problème en amont)
- **Cardinalité** : nombre de valeurs uniques pour les variables catégorielles (une nouvelle catégorie non vue à l'entraînement ?)
- **Corrélations** : la matrice de corrélation entre features reste-t-elle stable ?

### Métriques du modèle (toujours disponibles)

- **Distribution des prédictions** : répartition des scores ou classes prédits
- **Confiance** : distribution des scores de confiance (softmax). Si le modèle devient soudainement très incertain sur beaucoup de requêtes, c'est un signal.
- **Latence** : temps de réponse du modèle (p50, p95, p99)
- **Throughput** : nombre de prédictions par seconde
- **Taux d'erreur** : pourcentage de requêtes échouées (timeout, erreurs serveur)

### Métriques de performance (quand les labels sont disponibles)

- **Accuracy, Précision, Recall, F1** : métriques de classification standard
- **MAE, RMSE** : métriques de régression
- **AUC-ROC** : capacité de discrimination du modèle
- **Calibration** : les probabilités prédites correspondent-elles aux fréquences réelles ? Un modèle qui prédit 80% de probabilité devrait avoir raison 80% du temps.

<Callout type="warning" title="Le grand défi : les labels retardés">
En production, les labels (la vérité terrain) ne sont souvent pas disponibles immédiatement. Pour un modèle de prédiction de churn, on ne sait si le client a vraiment churné qu'après 30 jours. Pour un modèle de défaut de paiement, il faut attendre l'échéance du prêt. Sans labels, on ne peut monitorer que les données d'entrée et les prédictions, pas les performances réelles. C'est pourquoi les métriques proxy (confiance, distribution des prédictions) sont si importantes.
</Callout>

## Outils de monitoring

### Evidently AI

**Evidently** est une bibliothèque open source Python, la plus populaire pour le monitoring ML :
- Génération de rapports de drift riches (HTML interactif ou JSON pour l'automatisation)
- Tests automatisés de qualité des données et des prédictions (intégrables dans un pipeline CI/CD)
- Dashboards de suivi via Grafana
- Gratuit et self-hosted

### NannyML

**NannyML** se spécialise dans un problème difficile : estimer les performances du modèle **sans avoir les labels**. Il utilise des techniques comme le CBPE (Confidence-Based Performance Estimation) pour estimer l'accuracy du modèle à partir de ses scores de confiance. C'est particulièrement utile quand les labels arrivent avec des semaines ou des mois de retard.

### Arize AI

**Arize** est une plateforme SaaS de monitoring ML complète :
- Détection automatique de drift sur toutes les features
- Root cause analysis : quand les performances se dégradent, Arize identifié quelle feature a drifté
- Embeddings drift pour les modèles NLP et vision (détecte les changements dans l'espace vectoriel)
- Intégration avec les frameworks de serving populaires

### WhyLabs

**WhyLabs** propose un monitoring léger basé sur le profiling statistique :
- Profils compacts des distributions via la bibliothèque open source whylogs (quelques Ko par profil, même pour des millions de lignes)
- Alertes automatiques sur les anomalies détectées
- Intégration avec les feature stores et les pipelines de données

## Stratégie de monitoring : par où commencer

Une stratégie de monitoring ML complète se construit en couches :

1. **Baseline** : établir les distributions de référence à partir des données d'entraînement. C'est votre point de comparaison.
2. **Alertes** : configurer des seuils sur les métriques clés (PSI > 0.1, latence p99 > 200ms, taux d'erreur > 1%). Commencez large, affinez au fil du temps.
3. **Dashboards** : visualiser les tendances en temps réel. Un dashboard que personne ne regarde ne sert à rien : intégrez-le dans votre routine quotidienne.
4. **Escalade** : définir des procédures claires quand un drift est détecté. Qui est alerté ? Quelle est la marche à suivre ? Réentraînement automatique, rollback, ou investigation manuelle ?
5. **Boucle de feedback** : collecter les labels de production dès que possible pour valider les performances réelles (et pas seulement les proxies)
6. **Automatisation** : déclencher automatiquement un réentraînement quand le drift dépasse un seuil. C'est le niveau ultime de maturité, mais commencez par les étapes précédentes.
