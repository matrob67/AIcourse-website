## Le problème : trop de providers, trop d'APIs incompatibles

En production, vous n'utilisez probablement pas un seul modèle. GPT-4 pour les requêtes complexes qui demandent du raisonnement, Claude pour l'analyse de documents longs, un modèle open source auto-hébergé pour les tâches basiques à haut volume. Chaque provider à sa propre API, ses propres formats de requête et réponse, ses propres limites de rate, son propre pricing.

Résultat dans votre code : des blocs `if provider == "openai"` partout, des adaptateurs différents pour chaque fournisseur, et une maintenance cauchemardesque à chaque changement d'API. Ajoutez un nouveau provider ? Il faut modifier dix services.

Un **gateway LLM** résout ça en se positionnant comme un proxy centralisé. Votre application parle à UNE seule API unifiée. Le gateway traduit, route, et gère la complexité multi-provider en coulisse.

<Callout type="info" title="Le gateway comme point de contrôle unique">
Au-delà de l'unification des APIs, le gateway est l'endroit idéal pour implémenter les politiques transversales : caching des réponses, rate limiting par utilisateur, fallback automatique quand un provider tombe, monitoring des coûts, budgets par équipe. Plutôt que de répliquer cette logique dans chaque microservice, centralisez-la une seule fois dans le gateway.
</Callout>

## LiteLLM : l'unification open source

**LiteLLM** est une bibliothèque Python + proxy qui expose plus de 100 modèles de différents providers derrière une API compatible OpenAI. C'est l'outil open source de référence pour cette tâche.

### L'idée : une seule ligne de code, n'importe quel modèle

Peu importe le provider, vous appelez la même fonction `completion()` avec le format OpenAI. LiteLLM gère la traduction vers l'API native du provider.

```python
from litellm import completion

# Même API pour OpenAI, Anthropic, Google, Mistral, Cohere...
response = completion(
    model="claude-3.5-sonnet",  # ou "gpt-4", "gemini-pro", "mistral-large"
    messages=[{"role": "user", "content": "Explique-moi les transformers"}],
    temperature=0.7
)
```

Changer de modèle = changer une chaîne de caractères. Pas de refactoring, pas de nouvel adaptateur.

### Le proxy : la version production

En mode proxy, LiteLLM tourne comme un serveur HTTP. Vos applications envoient des requêtes à ce serveur (format OpenAI), et il route vers les providers configurés.

**Fonctionnalités du proxy** :
- **Load balancing** : distribuez les requêtes entre plusieurs déploiements du même modèle ou entre modèles équivalents
- **Fallback automatique** : si OpenAI renvoie une erreur 429 (rate limit) ou 500, la requête est automatiquement renvoyée vers Anthropic ou un autre provider
- **Rate limiting** : limitez les requêtes par clé API interne, par utilisateur, ou par provider
- **Budget management** : fixez des plafonds de dépenses par équipe ou par projet
- **Logging** : intégration native avec LangSmith, Langfuse, Helicone pour l'observabilité

### Architecture en production

**Application** &rarr; **LiteLLM Proxy** &rarr; **Provider APIs** (OpenAI, Anthropic, Mistral, modèles self-hosted)

Le proxy gère les clés API réelles des providers. Les applications n'ont qu'une clé interne LiteLLM. Si vous changez de provider ou ajoutez un fallback, aucune application ne doit être modifiée.

<Callout type="tip" title="LiteLLM, parfait pour les organisations multi-équipes">
Imaginez une entreprise avec 5 équipes qui utilisent des LLM. Sans gateway : chaque équipe gère ses propres clés API, son propre budget, ses propres fallbacks. Avec LiteLLM : chaque équipe à une clé interne. Le proxy gère les clés réelles des providers, répartit les budgets, agrège les logs. Centralisation sans friction.
</Callout>

### Stratégies de load balancing

Le choix de la stratégie dépend de votre objectif :

- **Round-robin** : distribution uniforme, le plus simple. Chaque requête va au provider suivant dans la liste.
- **Least busy** : route vers le déploiement le moins chargé. Optimal pour équilibrer la charge.
- **Latency-based** : mesure la latence de chaque provider et route vers le plus rapide. Minimise le temps de réponse.
- **Cost-based** : route vers le provider le moins cher qui peut traiter la requête. Minimise les coûts.

### Fallback et retries en cascade

Scénario concret d'un appel avec fallback :

1. **Tentative 1** : envoi vers OpenAI GPT-4
2. Erreur 429 (rate limit dépassé) &rarr; **Tentative 2** : fallback vers Anthropic Claude
3. Si succès, terminé. Si erreur à nouveau &rarr; **Tentative 3** : fallback vers Mistral Large
4. Si tout échoue &rarr; erreur retournée au client avec détails

Ce mécanisme est transparent pour l'application appelante. Elle envoie une requête, elle reçoit une réponse. Les retries et fallbacks sont gérés par le gateway.

## Portkey : le gateway premium avec intelligence

**Portkey** est une plateforme SaaS qui va au-delà du simple proxy pour offrir du routing intelligent et des fonctionnalités avancées.

### Ce que Portkey ajoute par rapport à LiteLLM

- **Routing conditionnel** : "utilisateur premium" &rarr; GPT-4, "utilisateur gratuit" &rarr; modèle open source. Les règles de routage sont configurables sans code.
- **A/B testing natif** : répartissez le trafic entre deux modèles (50/50 ou tout autre ratio) pour comparer leurs performances en conditions réelles
- **Canary deployment** : déployez progressivement un nouveau modèle (1% &rarr; 10% &rarr; 50% &rarr; 100%) avec surveillance automatique
- **Prompt management** : versionnez et déployez vos prompts indépendamment du code applicatif
- **Guardrails intégrés** : validations automatiques sur les entrées et sorties (toxicité, PII, format)
- **Caching sémantique** : cache basé sur la similarité sémantique, pas l'exacte match
- **Virtual keys** : abstraction des clés API réelles pour une gestion fine des accès

### Routing intelligent en pratique

Au lieu de router statiquement vers un seul modèle, Portkey permet des règles sophistiquées :
- Utilisateur premium + requête complexe &rarr; GPT-4o
- Utilisateur premium + requête simple &rarr; GPT-4o-mini
- Utilisateur gratuit &rarr; Llama 3 auto-hébergé
- Mode A/B test : 50% vers Claude, 50% vers GPT-4o pour comparer la satisfaction

<Callout type="paper" title="Le gateway comme outil de gouvernance">
Pour les grandes organisations, le gateway est un levier de gouvernance puissant. Il permet d'imposer des politiques uniformes : "aucun appel vers OpenAI pour les données sensibles", "budget max $200/jour par équipe", "tous les appels LLM sont logués et auditables". C'est le point de contrôle central qui manque quand chaque équipe gère ses propres intégrations LLM.
</Callout>

## Autres solutions à connaître

### Helicone
- Proxy léger avec un focus sur l'observabilité et l'analytics de coûts
- Intégration ultra-simple : changez juste la base URL de vos appels OpenAI
- Excellent pour le monitoring des dépenses

### Martian
- Router intelligent basé sur la prédiction de qualité des réponses
- Route automatiquement chaque requête vers le modèle optimal en termes de rapport qualité/coût
- Pas de configuration manuelle des règles de routage

### OpenRouter
- Marketplace de modèles avec une seule clé API pour des dizaines de providers
- Pricing unifié et transparent, pratique pour comparer les coûts
- Idéal pour le prototypage multi-modèle

## Comparaison rapide

| Critère | LiteLLM | Portkey | Helicone |
|---|---|---|---|
| **Type** | Open source + Cloud | SaaS premium | SaaS |
| **Self-hosted** | Oui | Non | Non |
| **Load balancing** | Basique (round-robin, etc.) | Avancé (conditionnel) | Basique |
| **Fallback** | Oui, configurable | Oui, avec monitoring | Non |
| **Caching** | Simple (exact match) | Sémantique | Oui |
| **Prompt management** | Non | Oui, avec versioning | Non |
| **Observabilité** | Via intégrations tierces | Intégrée et complète | Excellent, c'est le focus |
| **Coût** | Gratuit (self-hosted) | Freemium | Freemium |

<Callout type="warning" title="La latence ajoutée par le gateway">
Le gateway ajoute un saut réseau supplémentaire. En pratique, c'est typiquement 5-20 ms de latence additionnelle. Comparé au temps de génération d'un LLM (100 ms à plusieurs secondes), c'est négligeable dans la grande majorité des cas. Mais pour les applications ultra-sensibles à la latence (trading, temps réel critique), mesurez l'impact avant de déployer.
</Callout>

## Architecture de production recommandée

1. **Applications** (backend, chatbot, agents)
2. &darr; **Gateway** (LiteLLM self-hosted ou Portkey)
3. &darr; **Providers** (OpenAI, Anthropic, modèles self-hosted sur vLLM)
4. &darr; **Observabilité** (LangSmith, Langfuse, Helicone)
5. &darr; **Alerting** (coûts anormaux, latence élevée, taux d'erreur)

## Bonnes pratiques

- **Toujours configurer des fallbacks** : aucun provider ne garantit 100% de disponibilité. Un fallback évite que votre app tombe quand un provider à un incident.
- **Monitorer les coûts en temps réel** : les surprises de facturation arrivent vite, surtout avec des prompts longs ou un trafic inattendu
- **Activer le caching** : même un cache simple (exact match) peut réduire significativement les coûts pour les requêtes répétitives
- **Centraliser les clés API** : le gateway gère les clés des providers, les applications n'ont qu'une clé interne. Rotation et révocation simplifiées.
- **Versionner la configuration** : la config du gateway (routes, fallbacks, budgets) est du code. Versionnez-la dans Git, reviewez les changements.
- **Tester le failover régulièrement** : simulez des pannes de provider pour vérifier que vos fallbacks fonctionnent réellement

Un gateway bien configuré, c'est la différence entre une facture LLM qui explose sans visibilité et un budget maîtrisé avec des garde-fous automatiques.
