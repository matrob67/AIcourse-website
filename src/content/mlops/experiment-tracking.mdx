## Le problème : vous avez testé combien de configurations déjà ?

Un projet ML = 100s d'expériences. Learning rate différent ? Architecture différente ? Dataset différent ? Hyperparamètres variés ?

Sans un système structuré, vous vous perdez. "Quel modèle a obtenu les meilleur résultats ? Avec quoi ? Sur quelles données ?" → réponse : "euh... je sais pas."

Le **experiment tracking** enregistre automatiquement, pour chaque entraînement :

- Les **hyperparamètres** (learning rate, batch size, etc.)
- Les **métriques** (loss, accuracy, perplexity over time)
- Les **artefacts** (checkpoints, visualisations, logs)
- L'**environnement** (versions libs, code source)
- Les **données** (version dataset, stats)

<Callout type="info" title="Le spreadsheet trap">
Beaucoup de gens commencent par un tableur. 10 expériences ça marche. 1000 ? C'est l'enfer. Les outils de tracking offrent recherche, filtrage, comparaison visuelle, reproduction auto. C'est un vrai plus.
</Callout>



## MLflow : le standard open source

**MLflow** créé par Databricks, c'est l'outil le plus adopté. Flexible, pas de vendor lock-in.

### Les 4 composants

- **Tracking** : enregistrez params, métriques, artefacts
- **Models** : format standardisé pour empacker un modèle
- **Model Registry** : gestion du cycle de vie (staging, prod, archived)
- **Projects** : packaging du code ML reproductible

### Forces

- Open source = deployable on-premise
- Agnostique : PyTorch, TensorFlow, scikit-learn, HuggingFace, tout
- Simple : quelques lignes de code
- Standard de facto : large adoption

### Limites

- Interface web basique (fonctionnelle mais pas jolie)
- Collaboration d'équipe nécessite un serveur dédié
- Visuels avancés limités

<Callout type="tip" title="MLflow + HuggingFace">
HuggingFace Transformers a une intégration native MLflow. Ajoutez un callback et boum, tout est auto-tracké (hyperparamètres, métriques, checkpoints) sans code custom.
</Callout>

## Weights & Biases : l'interface qui tue

**W&B** c'est la plateforme SaaS pour la visualisation et la collaboration. C'est vraiment beau.

### Ce qu'il fait

- **Experiment Tracking** : dashboards interactifs, comparaisons visuelles, zoom, filtrage
- **Sweeps** : recherche d'hyperparamètres distribuée avec Bayesian optimization
- **Artifacts** : versioning de datasets et modèles
- **Tables** : exploration interactive de données tabulaires
- **Reports** : rapports collaboratifs avec graphiques et analyses
- **Prompts** : tracking spécifique pour les chaînes LLM

### Forces

- Interface exception : vraiment belle, intuitive
- Collaboration : partage de projets, commentaires, rapports d'équipe
- Intégrations : PyTorch, TensorFlow, HuggingFace, LangChain
- Communauté : très utilisé en recherche, des papers partagent les logs W&B publics

### Limites

- SaaS = données sur les serveurs W&B (self-hosted existe mais cher)
- Payant (gratuit perso, ~$50/mois/user pour équipes)
- Vendor lock-in : les features avancées sont propriétaires

<Callout type="paper" title="W&B en recherche">
C'est devenu standard en recherche ML. Les papers incluent des liens W&B publics. La communauté explore les résultats détaillés et reproduit les expériences.
</Callout>

## Neptune : l'équilibre

**Neptune** entre MLflow et W&B. Pas aussi jolie que W&B, mais plus flexible que MLflow.

### Ce qu'il fait

- **Experiment Tracking** : structure flexible, namespaces hiérarchiques
- **Model Registry** : gestion du cycle de vie
- **Comparaison d'expériences** : interface côte à côte très détaillée
- **Self-hosted** : déployable on-premise

### Forces

- Interface de comparaison ergonomique
- Structure des métadonnées flexible (pas de schéma imposé)
- Bon rapport qualité/prix
- Self-hosted disponible

### Limites

- Communauté plus petite
- Moins de features LLM-spécifiques
- Moins visible en recherche

## Comparaison : qui choisir

| Critère | MLflow | W&B | Neptune |
|---|---|---|---|
| **Open source** | Oui | Non | Non |
| **Self-hosted** | Oui (gratuit) | Payant | Payant |
| **Interface** | Basique | Excellente | Bonne |
| **Collaboration** | Limitée | Excellente | Bonne |
| **Coût** | Gratuit | $$$ | $$ |
| **LLM support** | Basique | W&B Prompts | Basique |

<Callout type="warning" title="Les données sensibles">
Les traces contiennent vos prompts et réponses, potentiellement avec des infos sensibles (PII). Avant d'envoyer vers le cloud : vérifiez la conformité RGPD. Self-hosted (Langfuse, Flyte) offre plus de contrôle.
</Callout>

## Bonnes pratiques

- **Logger automatiquement** : utilisez les callbacks natifs, pas du logging manuel
- **Versionner le code** : enregistrez le git commit hash avec chaque expérience
- **Tagger** : utilisez des tags pour catégoriser (baseline, ablation, final)
- **Comparer systématiquement** : définissez une métrique principale, comparez toujours au baseline
- **Documenter les échecs** : les expériences échouées apprennent autant que les succès
- **Partager avec l'équipe** : le tracking a de la valeur que s'il est accessible à tous

Le tracking c'est pas glamour mais c'est essentiel. C'est la différence entre "je fais du ML à l'aveugle" et "je sais ce que je fais".
