## Le problème : vous avez testé combien de configurations déjà ?

Un projet ML sérieux, c'est facilement des centaines d'expériences. Learning rate de 1e-3 ou 3e-4 ? Architecture avec 6 ou 12 couches ? Dataset v2 ou v3 ? Batch size 32 ou 128 ? Chaque combinaison est une expérience. Et chaque expérience produit des courbes de loss, des métriques d'évaluation, des checkpoints de modèle.

Sans un système structuré, on se perd très vite. "Quel modèle avait donné les meilleurs résultats ? Avec quels hyperparamètres ? Sur quelles données ?" - la réponse honnête est souvent "aucune idée, j'ai écrasé les résultats la semaine dernière".

Le **experiment tracking** enregistre automatiquement, pour chaque run d'entraînement :

- Les **hyperparamètres** : learning rate, batch size, warmup steps, architecture, etc.
- Les **métriques** : loss, accuracy, perplexity, F1, et leur évolution au fil de l'entraînement
- Les **artefacts** : checkpoints du modèle, visualisations, matrices de confusion, logs
- L'**environnement** : versions des bibliothèques, commit Git du code, hardware utilisé
- Les **données** : version du dataset, statistiques de base (taille, distribution)

<Callout type="info" title="Le piège du tableur Excel">
Beaucoup commencent par un tableur Google Sheets pour tracker leurs expériences. Avec 10 runs, c'est gérable. Avec 100, c'est pénible. Avec 1 000, c'est inutilisable. Les outils de tracking offrent la recherche par paramètre, le filtrage par métrique, la comparaison visuelle de courbes, et la reproduction automatique d'une expérience passée. C'est un saut qualitatif.
</Callout>

## MLflow : le standard open source

**MLflow**, créé par Databricks, est l'outil de tracking le plus adopté au monde. Son avantage principal : c'est open source, flexible, et sans vendor lock-in.

### Les 4 composants de MLflow

- **Tracking** : le cœur du produit. Enregistrez params, métriques et artefacts pour chaque expérience. Interface web pour explorer, comparer, filtrer.
- **Models** : un format standardisé pour empaqueter un modèle avec ses dépendances. Un modèle MLflow se déploie sur n'importe quelle plateforme compatible.
- **Model Registry** : gestion du cycle de vie des modèles. Un modèle passe de "staging" à "production" à "archived", avec historique complet des transitions.
- **Projects** : packaging reproductible du code ML (environnement conda/docker + point d'entrée).

### Forces concrètes

- **Open source** : déployable sur votre propre infrastructure, zéro coût de licence, vos données restent chez vous
- **Agnostique** : fonctionne avec PyTorch, TensorFlow, scikit-learn, HuggingFace, XGBoost, et tout le reste
- **Simple à intégrer** : quelques lignes de code suffisent pour commencer à tracker
- **Standard de facto** : la plus large adoption dans l'industrie, ce qui facilite le recrutement et la documentation

### Limites honnêtes

- L'interface web est fonctionnelle mais visuellement basique, surtout comparée à W&B
- La collaboration multi-utilisateurs nécessite de déployer un serveur MLflow dédié (pas juste un fichier local)
- Les visualisations avancées (comparaisons de courbes, dashboards custom) sont limitées

<Callout type="tip" title="MLflow + HuggingFace : le combo sans friction">
HuggingFace Transformers a une intégration native avec MLflow. Ajoutez un callback MLflow à votre Trainer et c'est fait : hyperparamètres, métriques par epoch, learning rate schedule, tout est auto-tracké sans écrire une seule ligne de logging custom. C'est le chemin de moindre résistance pour commencer.
</Callout>

## Weights & Biases : quand l'interface fait la différence

**W&B** (Weights & Biases) est la plateforme SaaS qui a séduit la communauté ML par la qualité de son interface. C'est vraiment beau et agréable à utiliser.

### Les fonctionnalités qui font la différence

- **Experiment Tracking** : dashboards interactifs avec zoom, filtrage, comparaison de courbes côte à côte. Vous voyez en un coup d'oeil quelle configuration performe le mieux.
- **Sweeps** : recherche d'hyperparamètres distribuée avec optimisation bayésienne intégrée. Définissez l'espace de recherche, W&B explore intelligemment.
- **Artifacts** : versioning de datasets et modèles avec lignage automatique (quel artefact a produit quel autre)
- **Tables** : exploration interactive de données tabulaires, parfait pour analyser les erreurs du modèle
- **Reports** : rapports collaboratifs mélangeant graphiques, analyses et commentaires. Idéal pour partager les résultats avec l'équipe.
- **Prompts** : tracking spécialisé pour les chaînes LLM (prompts, réponses, évaluations)

### Forces

- Interface exceptionnelle : intuitive, réactive, visuellement soignée. C'est le point fort incontestable.
- Collaboration native : partage de projets, commentaires sur les runs, rapports d'équipe
- Intégrations profondes : PyTorch, TensorFlow, HuggingFace, LangChain, Keras, et plus
- Communauté active : très utilisé en recherche académique, de nombreux papiers partagent leurs logs W&B publics

### Limites

- SaaS par défaut : vos données (prompts, métriques, artefacts) sont sur les serveurs W&B. La version self-hosted existe mais coûte cher.
- Payant pour les équipes : gratuit pour l'usage personnel, mais ~$50/mois/utilisateur pour les features d'équipe
- Vendor lock-in : les features avancées (Sweeps, Reports, Artifacts) sont propriétaires

<Callout type="paper" title="W&B est devenu un standard en recherche ML">
Il est devenu courant de voir dans les papiers de recherche des liens vers des projets W&B publics. La communauté peut explorer les résultats détaillés, comparer les runs, et reproduire les expériences. C'est un vrai plus pour la science ouverte.
</Callout>

## Neptune : l'équilibre entre les deux

**Neptune** se positionne entre MLflow (open source, basique) et W&B (SaaS, premium). Pas aussi joli que W&B, mais plus complet que MLflow.

### Ce qu'il fait bien

- **Experiment Tracking** : structure flexible avec namespaces hiérarchiques. Vous organisez vos métadonnées comme vous voulez, sans schéma imposé.
- **Model Registry** : gestion complète du cycle de vie des modèles
- **Comparaison d'expériences** : interface côte à côte particulièrement ergonomique pour comparer des dizaines de runs
- **Self-hosted** : déployable on-premise, important pour les entreprises avec des contraintes de données

### Forces

- Interface de comparaison ergonomique et détaillée
- Structure des métadonnées flexible (pas de schéma rigide à respecter)
- Bon rapport qualité/prix par rapport à W&B
- Option self-hosted pour garder le contrôle des données

### Limites

- Communauté plus petite que MLflow ou W&B
- Moins de features spécifiques aux LLM
- Moins visible dans la recherche académique

## Comparaison : qui choisir selon votre contexte

| Critère | MLflow | W&B | Neptune |
|---|---|---|---|
| **Open source** | Oui, totalement | Non | Non |
| **Self-hosted** | Oui (gratuit) | Oui (payant) | Oui (payant) |
| **Interface** | Fonctionnelle | Excellente | Bonne |
| **Collaboration** | Basique | Excellente | Bonne |
| **Coût** | Gratuit | $$$ | $$ |
| **LLM support** | Basique | W&B Prompts | Basique |

<Callout type="warning" title="Attention aux données sensibles dans les traces">
Les traces d'expériences contiennent potentiellement des informations sensibles : prompts avec des données utilisateurs, réponses contenant des PII, métriques business confidentielles. Avant d'envoyer tout ça vers un SaaS cloud, vérifiez la conformité RGPD. Les solutions self-hosted (MLflow, Langfuse) offrent un contrôle total sur vos données.
</Callout>

## Bonnes pratiques pour un tracking efficace

- **Logger automatiquement** : utilisez les callbacks natifs des frameworks (HuggingFace, PyTorch Lightning) plutôt que du logging manuel. Moins de code, moins d'oublis.
- **Versionner le code** : enregistrez le commit Git hash avec chaque expérience. Sans ça, vous ne pouvez pas reproduire.
- **Tagger intelligemment** : utilisez des tags pour catégoriser vos runs (baseline, ablation, final, debug). Ça facilite la recherche plus tard.
- **Comparer systématiquement** : définissez une métrique principale et comparez toujours au baseline. Sans point de référence, les chiffres ne veulent rien dire.
- **Documenter les échecs** : les expériences ratées sont aussi instructives que les succès. "Learning rate 1e-2 diverge après 100 steps" est une information précieuse pour toute l'équipe.
- **Rendre accessible** : le tracking n'a de valeur que si toute l'équipe peut y accéder. Un tracker que personne ne consulte est un tracker inutile.

Le tracking d'expériences n'est pas glamour, mais c'est la fondation d'un travail ML rigoureux. C'est ce qui sépare "je fais du ML en tâtonnant" de "je sais exactement ce que j'ai essayé, ce qui a marché, et pourquoi".
