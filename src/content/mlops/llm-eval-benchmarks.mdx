## Pourquoi des benchmarks

Évaluer un LLM est difficile : plusieurs bonnes réponses possibles. "Qualité" est subjective et dépend du contexte.

Les benchmarks = **tâches standardisées** pour comparer objectivement. Suive les progrès, choisir le modèle adapté.

<Callout type="info" title="Le paradoxe">
Un LLM peut exceller sur tous les benchmarks et être nul pour votre cas. Inversement, un modèle modeste sur benchmarks peut être excellent après fine-tuning. Les benchmarks mesurent capacités générales, pas votre cas spécifique.
</Callout>



## Benchmarks de connaissances

### MMLU (Massive Multitask Language Understanding)

Le benchmark **le plus cité**. 14 042 questions à choix multiples sur 57 sujets (maths, physique, histoire, droit, médecine, etc.).

Le score MMLU c'est le "chiffre magique" annoncé avec chaque nouveau modèle. GPT-4 ~86%, Claude 3.5 Sonnet ~88%, meilleurs modèles >90%.

**MMLU-Pro** = version plus difficile avec 10 choix (au lieu de 4) et problèmes plus complexes.

### ARC (AI2 Reasoning Challenge)

Raisonnement scientifique au niveau collège/lycée. ARC-Easy et ARC-Challenge. Ce dernier = multi-étapes.

### HellaSwag

Bon sense et compréhension de la langue naturelle (complétion de phrases). Meilleurs modèles >95%. Quasiment saturé.

## Benchmarks de code

### HumanEval

OpenAI. 164 problèmes de programmation Python. Le modèle génère une fonction complète.

Métrique = **pass@k** : % de problèmes résolus en k tentatives.

Pass@1 = première réponse correcte. Pass@10 = au moins une parmi 10 correcte. L'écart indique la consistance.

<Callout type="paper" title="Pass@1 vs Pass@10">
Un modèle avec pass@1=70% et pass@10=95% est capable mais inconsistant. Il trouve la réponse mais pas toujours la première fois.
</Callout>

### MBPP (Mostly Basic Python Problems)

974 problèmes Python niveau débutant à intermédiaire. Plus large que HumanEval mais plus facile.

### SWE-bench

Vrais bugs dans des projets open source. Le modèle reçoit une issue GitHub et doit produire un patch. **Le benchmark le plus réaliste** pour les capacités de codage.

## Benchmarks de mathématiques

### GSM8K

8 500 problèmes de maths niveau école/collège (plusieurs étapes). Standard pour le raisonnement basique.

### MATH

12 500 problèmes niveau lycée à licence par difficulté (1-5). Beaucoup plus dur que GSM8K.

### AIME / AMC

Compétitions de maths américaines. Les meilleurs modèles font bien sur AMC mais craignent sur AIME.

## Benchmarks conversationnels

### MT-Bench

80 questions en 8 catégories (écriture, raisonnement, maths, code). Chaque question a un follow-up pour tester la cohérence multi-tour. GPT-4 évalue sur 1-10.

### LMSYS Chatbot Arena

**Chatbot Arena** = évaluation humaine continue. Des utilisateurs anonymes posent des questions à deux modèles anonymes et votent le meilleur.

Le classement **Elo résultant** c'est considéré comme **l'une des évaluations les plus fiables**. C'est les vraies préférences humaines.

<Callout type="tip" title="Le Elo c'est gold">
Le système Elo capture les préférences humaines réelles. Il's mis à jour continuellement. C'est souvent le classement regardé en premier pour comparer modèles.
</Callout>

## Benchmarks spécialisés

- **IFEval** : suivi d'instructions précises
- **TruthfulQA** : réponses véridiques vs réponses populaires fausses
- **BBH** : raisonnement avancé difficile
- **GPQA** : questions niveau doctorat (même les experts en galère)

## Limites des benchmarks

### Contamination des données

Si les questions du benchmark apparaissent dans les données d'entraînement, le modèle les a "mémorisées", pas compris. Certains benchmarks historiques sont devenus quasi-inutiles.

### Saturation

HellaSwag, ARC-Easy sont **saturés** (>95% pour les meilleurs modèles). Impossible de comparer. De nouveaux benchmarks plus durs arrivent régulièrement.

### Goodhart's Law

Quand un benchmark devient cible d'optimisation, il cesse d'être une bonne mesure. Les modèles sont parfois optimisés **spécifiquement pour les benchmarks** vs performance réelle.

<Callout type="warning" title="Ne fiez-vous pas à un seul">
Aucun benchmark unique capture toutes les capacités. Excellent maths = peut être nul créativité. Évaluez sur plusieurs benchmarks pertinents pour **votre** cas d'usage.
</Callout>

## En pratique

Pour choisir un LLM :

1. Consulter les classements : Chatbot Arena Elo, Open LLM Leaderboard
2. Identifier benchmarks pertinents : code → HumanEval/SWE-bench, maths → GSM8K/MATH, général → MMLU
3. Créer votre propre benchmark : dataset spécifique à votre cas
4. Évaluer rapport qualité/coûts : 2x moins cher mais 5% moins bon = peut être meilleur choix
5. Tester en conditions réelles : A/B testing avec vraies utilisateurs

Les benchmarks = point de départ, pas la réponse complète.
