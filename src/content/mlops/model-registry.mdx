## Le chaos avant un Model Registry

Vous avez entraîné 100 modèles en 6 mois. Lequel est en production ? Lequel était là avant ? Comment revenir en arrière si le nouveau modèle fait n'importe quoi ? Quel dataset a produit ce modèle ? Quels hyperparamètres ?

Sans un registry, c'est le chaos. Les modèles sont éparpillés sur des disques locaux, des buckets S3, des machines différentes. `model_v3_final_FINAL_v2.pt` : personne ne sait ce que c'est, même pas celui qui l'a entraîné il y a 3 mois.

Un **Model Registry**, c'est une source de vérité unique pour tous vos modèles. C'est comme Git pour le code, mais pour les modèles ML. Chaque modèle est versionné, documenté, tracé. Vous savez toujours exactement quel modèle est en production, d'où il vient, et comment le remplacer.

<Callout type="info" title="Le cycle de vie d'un modèle">
Développement (expérimentation) → Staging (tests pré-production) → Production (live, avec les vrais utilisateurs) → Archived (retiré, mais conservé pour référence). Le registry gère ces transitions et maintient un historique complet. À tout moment, vous savez où en est chaque modèle.
</Callout>

## Les fonctionnalités fondamentales

### Versioning

Chaque modèle à des versions numérotées. V1 = BERT-base entraîné sur 10K exemples. V2 = BERT-base avec plus de données. V3 = RoBERTa avec une nouvelle architecture. Chaque version est un artefact immuable : une fois enregistré, il ne change plus. C'est comme un tag Git : un instantané figé dans le temps.

### Metadata complète

Pour chaque version du modèle, le registry stocke : le commit hash du code, la version du dataset d'entraînement, tous les hyperparamètres, les métriques sur le test set (accuracy, F1, latence), l'environnement (versions des librairies). **Tout**. Parce que sans ces informations, reproduire un résultat ou comprendre pourquoi un modèle se comporte ainsi devient un casse-tête.

### Les stades du modèle

Un modèle passe par des stades bien définis : None (pas encore prêt) → Staging (en cours de test pré-prod) → Production (live avec les utilisateurs) → Archived (retiré). Chaque transition est tracée : qui l'a promue, quand, et pourquoi. C'est un audit trail complet.

### Reproductibilité

Vous trouvez un modèle en production qui fait des prédictions bizarres. Vous devez pouvoir remonter à exactement quel code, quel dataset, quels hyperparamètres l'ont produit. Sans cette traçabilité, le debugging en production est un cauchemar. Le registry est le fil d'Ariane qui vous ramène toujours à l'origine.

## MLflow Model Registry

**MLflow** est l'outil de model registry le plus utilisé. Il s'intègre avec MLflow Tracking (là où vous avez loggé vos expériences d'entraînement), ce qui crée un pipeline naturel : vous entraînez, vous loggez, vous enregistrez le meilleur modèle dans le registry.

### Le workflow typique

1. Vous entraînez un modèle via MLflow → ça crée un **run** avec toutes les métriques
2. Vous enregistrez le modèle dans le registry → ça crée une **version** liée au run
3. Vous passez la version en "Staging" → elle est marquée pour les tests pré-production
4. Vous testez sur votre environnement de staging → tout va bien
5. Vous passez en "Production" → le modèle est live
6. Problème en prod ? → vous revenez à la version précédente en quelques clics, sans panique

### Le format MLflow Model

MLflow définit un format standardisé pour les modèles, appelé **MLmodel**. Ce format encapsule tout ce qu'il faut pour réutiliser le modèle :
- Les poids du modèle dans le format natif (PyTorch, TensorFlow, ONNX, etc.)
- Un fichier `MLmodel` décrivant les interfaces supportées (appelées "flavors")
- Un `conda.yaml` ou `requirements.txt` pour recréer l'environnement exact
- Une **signature** définissant le schéma d'entrée/sortie du modèle (quelles colonnes en entrée, quel format en sortie)

<Callout type="tip" title="Les Flavors MLflow : la portabilité en action">
Les "flavors" sont un concept puissant de MLflow : un même modèle peut être chargé de plusieurs manières. Un modèle PyTorch est enregistré à la fois comme `pytorch` (chargeable en tant que module PyTorch natif) et comme `python_function` (chargeable comme une simple fonction Python avec `predict()`). Cela permet au système de serving de charger n'importe quel modèle de manière uniforme, qu'il soit PyTorch, TensorFlow, scikit-learn ou autre. C'est l'abstraction qui rend le déploiement agnostique au framework.
</Callout>

### Déploiement depuis le Registry

MLflow permet de déployer directement un modèle depuis le registry, sans copier-coller de fichiers :
- **Inférence locale** : chargement du modèle via l'API Python (`mlflow.pyfunc.load_model()`)
- **REST API** : `mlflow models serve` lance un serveur d'inférence HTTP en une commande
- **Docker** : `mlflow models build-docker` crée une image conteneur prête à déployer
- **Cloud** : déploiement direct vers SageMaker, Azure ML, ou Databricks

### Forces et limites

**Forces** : open source et gratuit, intégration native avec tout l'écosystème MLflow, gestion des stades et transitions, format standardisé qui facilite le déploiement, API Python et REST complètes

**Limites** : l'interface web est fonctionnelle mais basique pour la gestion des modèles, pas de contrôle d'accès granulaire sans la version Databricks payante, les modèles très volumineux (dizaines de Go) nécessitent un stockage adapté (S3, GCS)

## Hugging Face Hub

### Le GitHub des modèles ML

Le **Hugging Face Hub** est une plateforme communautaire qui combine model registry, partage de datasets, et hébergement d'applications de démonstration. C'est devenu le **GitHub des modèles ML**, avec plus de 500 000 modèles publics. Si MLflow est le registry d'entreprise, le Hub est la place publique mondiale du ML.

### Fonctionnalités

- **Model Cards** : documentation structurée décrivant le modèle, ses capacités, ses limites et ses biais. C'est devenu un standard de l'industrie.
- **Versioning Git** : chaque modèle est un dépôt Git (avec Git LFS pour les fichiers volumineux). Vous avez l'historique complet des modifications.
- **Inference API** : testez l'inférence directement depuis la page du modèle, sans écrire une ligne de code
- **Spaces** : déploiement d'applications de démo avec Gradio ou Streamlit, hébergées gratuitement
- **Discussions** : commentaires, questions et pull requests sur les modèles (exactement comme sur GitHub)
- **Collections** : regroupement thématique de modèles (par exemple, tous les modèles de traduction français-anglais)
- **Leaderboards** : classements de modèles sur des benchmarks standardisés

<Callout type="paper" title="Hub privé pour les entreprises">
Hugging Face propose un Hub privé (Enterprise Hub) pour les organisations. Les modèles restent privés, avec gestion fine des accès, audit logs, et conformité SOC2. C'est une option populaire pour les entreprises qui veulent le workflow et la simplicité Hugging Face sans exposer leurs modèles au public.
</Callout>

### Intégration avec l'écosystème HuggingFace

Le Hub s'intègre directement avec les bibliothèques HuggingFace, ce qui le rend extrêmement pratique :
- **Transformers** : `model.push_to_hub()` pour publier, `AutoModel.from_pretrained("user/model")` pour charger
- **Datasets** : `load_dataset("user/dataset")` pour charger un dataset en une ligne
- **PEFT** : partage d'adaptateurs LoRA (des petits fichiers qui modifient le comportement d'un gros modèle)
- **Safetensors** : format sécurisé pour les poids de modèles (voir l'encadré sécurité ci-dessous)

### Forces et limites

**Forces** : communauté massive et active, Model Cards comme norme de documentation, facilité d'utilisation exceptionnelle, Inference API gratuite pour tester, versioning Git natif

**Limites** : pas de gestion formelle des stades (staging/production) comme MLflow, pas d'intégration native avec les pipelines CI/CD d'entreprise, orienté partage communautaire plus que gestion de cycle de vie, coût du Hub Enterprise pour les grandes organisations

## Comparaison

| Critère | MLflow Registry | HuggingFace Hub |
|---|---|---|
| Type | Registry d'entreprise | Plateforme communautaire |
| Stades (staging/prod) | Oui, natif | Non natif |
| Versioning | Séquentiel (v1, v2, v3) | Git (branches, commits) |
| Documentation | Basique (tags, description) | Model Cards riches et structurées |
| Communauté | N/A | 500K+ modèles publics |
| Déploiement | Multi-cloud (SageMaker, Azure) | Inference API + Spaces |
| Hébergement | On-premise ou cloud (vous gérez) | Cloud SaaS / Enterprise |
| Coût | Gratuit (open source) | Gratuit (public) / Payant (privé) |

<Callout type="warning" title="Sécurité des modèles : attention aux fichiers pickle">
Les modèles ML peuvent contenir du code malveillant, notamment les fichiers pickle (utilisés par défaut par PyTorch). Charger un fichier pickle revient à exécuter du code arbitraire : un modèle malveillant pourrait supprimer vos fichiers ou exfiltrer vos données. Le format **safetensors** de HuggingFace résout ce problème : il stocke uniquement les tenseurs (les chiffres) sans aucun code exécutable. Privilégiez toujours safetensors pour le stockage et le partage de modèles.
</Callout>

## En pratique : combiner les deux

Pour une équipe ML en entreprise, l'approche la plus courante combine les forces des deux outils :
- **MLflow Model Registry** pour la gestion interne du cycle de vie (staging, production, rollback, audit trail)
- **Hugging Face Hub** (privé ou public) pour le partage, la documentation riche (Model Cards), et la collaboration
- Des **webhooks et pipelines CI/CD** pour automatiser les transitions entre les deux systèmes (un modèle validé dans MLflow est automatiquement poussé sur le Hub privé, par exemple)

Cette combinaison donne le meilleur des deux mondes : la rigueur du cycle de vie MLflow et la richesse communautaire du Hub.
