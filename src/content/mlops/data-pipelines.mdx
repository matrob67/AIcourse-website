## Le défi : des données qui ne tiennent nulle part

LLaMA a été entraîné sur 15 **trillions** de tokens. LAION-5B contient 5 milliards d'images. Même un "petit" dataset de fine-tuning peut peser des dizaines de Go. Votre RAM, aussi généreuse soit-elle, ne peut pas contenir tout ça. Il faut distribuer le traitement sur plusieurs machines.

Un pipeline de données ML typique ressemble à ça : ingestion des données brutes, nettoyage, filtrage qualité, transformations (normalisation, augmentation), tokenization, et enfin création des batches d'entraînement. Chaque étape doit pouvoir traiter des centaines de Go, voire des To de données, sans planter.

<Callout type="info" ’>
Le preprocessing d'un dataset LLM peut prendre des jours, voire des semaines de calcul. Chaque jour gagné sur le pipeline = un jour de plus pour itérer sur l'entraînement. Optimiser le pipeline de données n'est pas un luxe, c'est un investissement direct en vélocité de recherche.
</Callout>

## Apache Spark : le géant éprouvé du big data

**Apache Spark** existe depuis 2009 et reste le standard incontesté pour le traitement de données massives. Databricks (la version commerciale), utilisé par les plus grandes entreprises du monde, est construit dessus.

### Architecture : lazy évaluation et optimisation automatique

Spark partitionne vos données sur les nœuds du cluster. Quand vous enchaînez des transformations (filtrage, jointures, agrégations), Spark ne les exécute pas immédiatement. Il construit un plan d'exécution logique. C'est seulement quand vous demandez le résultat (écriture, collecte) que l'optimizer **Catalyst** analyse le plan complet, le réordonne, fusionne les étapes, et l'exécute de la manière la plus efficace possible.

C'est comme un chef cuisinier qui lit toute la recette avant de commencer, plutôt que de faire chaque étape une par une.

### Spark pour le ML en pratique

```python
# Lecture depuis le cloud, automatiquement distribuée
data = spark.read.parquet("s3://bucket/huge-dataset/")

# Chaîne de transformations (lazy, pas encore exécuté)
data = data.filter(...).repartition(...).groupBy(...).agg(...)

# L'exécution se déclenche ici
data.write.parquet("output/")
```

PySpark lit nativement Parquet, CSV, JSON, ORC depuis S3, GCS, Azure Blob ou HDFS. Déduplication, tokenization, feature engineering, jointures massives : tout se fait avec l'API DataFrame familière.

<Callout type="tip" title="Spark, le choix des grands datasets LLM">
RedPajama, FineWeb, et la plupart des grands datasets ouverts pour LLM ont été préparés avec Spark. C'est le seul outil qui scale confortablement de 100 Go à 5 To et au-delà sans configuration exotique. Quand vous traitez des trillions de tokens, Spark est votre allié.
</Callout>

### MLlib : le ML intégré à Spark

**MLlib** propose des algorithmes classiques (régression, clustering, classification, collaborative filtering) directement sur les DataFrames distribuées. C'est pratique pour le feature engineering et les modèles classiques. Mais pour le deep learning, le pattern habituel est : Spark pour le preprocessing massif, puis PyTorch/JAX pour l'entraînement sur GPU.

### Forces et limites

**Bon pour** : traitement batch massif (To à Po), analytics SQL sur data lakes, écosystème mature avec des centaines de connecteurs

**Pas bon pour** : streaming temps réel (Spark Streaming existe mais n'est pas son point fort), pas d'accélération GPU native, overhead de la JVM pour les petits jobs

## Dask : le distribué Python-native

**Dask** offre du calcul distribué avec une API qui mime Pandas et NumPy. Si vous connaissez Pandas, vous connaissez Dask.

### L'idée : distribuer ce que vous savez déjà faire

Si votre DataFrame Pandas fait 100 Go et ne tient pas en mémoire, Dask le découpe en partitions qui tiennent chacune en mémoire, les répartit sur les workers d'un cluster, et exécute vos opérations Pandas partition par partition. Votre code reste quasi-identique.

### Les collections Dask

- **DataFrame** : API Pandas distribuée, la plus utilisée
- **Array** : API NumPy distribuée, pour les tableaux numériques massifs
- **Bag** : pour les données non-structurées (textes bruts, JSON imbriqués)
- **ML** : wrappers pour scikit-learn distribué (GridSearchCV sur cluster, par exemple)

<Callout type="paper" ’>
Dask excelle quand vos données tiennent sur une grosse machine ou un petit cluster (jusqu'à ~1 To). L'API est plus pythonique, le déploiement plus simple, l'overhead plus faible. Au-delà de 1 To, ou si vous avez besoin de SQL avancé et de l'optimizer Catalyst, Spark prend l'avantage.
</Callout>

### Forces et limites

**Bon pour** : data scientists familiers de Pandas/NumPy, datasets moyens (100 Go - 1 To), prototypage rapide avant passage à Spark

**Pas bon pour** : requêtes SQL complexes, datasets au-delà du pétaoctet, pas de support GPU natif

## Ray Data : le streaming pour les pipelines ML

**Ray Data** est le module de traitement de données du framework Ray. Contrairement à Spark et Dask qui matérialisent les données intermédiaires, Ray Data est conçu pour le **streaming** vers l'entraînement.

### Le streaming : la différence fondamentale

Avec un pipeline classique (Spark), vous prétraitez la totalité de vos 5 To de données, les écrivez en Parquet sur disque, puis les lisez pour l'entraînement. Deux passes complètes sur les données, et des To de stockage intermédiaire.

Avec Ray Data, le streaming fonctionne autrement :
1. Lit un bloc de données brutes
2. Le transforme (nettoyage, tokenization)
3. L'envoie directement aux GPU d'entraînement
4. Passe au bloc suivant, en parallèle

Zéro matérialisation intermédiaire. Le preprocessing et l'entraînement tournent simultanément. C'est comme une chaîne de montage continue plutôt qu'un processus en deux étapes séparées.

<Callout type="tip" title="Le streaming, idéal pour les gros LLM">
Pour un entraînement LLM sur des trillions de tokens, matérialiser tout le dataset tokenisé en Parquet d'abord peut coûter des To de stockage et des jours de preprocessing. Ray Data élimine cette étape en streamant directement vers l'entraînement. Moins d'infra, plus rapide.
</Callout>

### Fonctionnalités spécifiquement ML

- **Preprocessing GPU** : augmentation d'images et tokenization directement sur GPU, pas seulement CPU
- **Intégration entraînement** : les données sont consommées directement par Ray Train, sans étape intermédiaire
- **Workers hétérogènes** : CPU workers pour le preprocessing, GPU workers pour l'entraînement, dans le même cluster Ray

### Forces et limites

**Bon pour** : pipelines de données LLM, preprocessing GPU, streaming vers l'entraînement, intégration avec l'écosystème Ray

**Pas bon pour** : analytics SQL (Spark fait ça bien mieux), communauté et documentation moins matures que Spark

## Comparaison synthétique

| Critère | Spark | Dask | Ray Data |
|---|---|---|---|
| **Maturité** | Très mature (15+ ans) | Élevée | Moyenne |
| **Langage** | Scala/Python/Java | Python natif | Python natif |
| **GPU** | Limité (RAPIDS) | Limité | Natif |
| **Streaming** | Spark Streaming | Dask Streams | Natif, streaming-first |
| **Scale max** | PB+ | TB+ | TB+ |
| **SQL** | Excellent (Catalyst) | Basique | Non supporté |
| **LLM focus** | Preprocessing batch | Usage général | Excellent (streaming) |

<Callout type="warning" ’>
Si vos données tiennent en mémoire d'une seule machine (< 100 Go), Pandas ou Polars suffit amplement. Vous économisez en complexité d'infrastructure, en temps de debugging distribué, et en coûts de cluster. Le distribué n'est justifié que quand c'est vraiment nécessaire.
</Callout>

## Mention spéciale : Polars

**Polars** mérite d'être mentionné. Écrit en Rust, il est 5-10x plus rapide que Pandas sur une seule machine. Il supporte le lazy évaluation (comme Spark) et le streaming (traitement par chunks).

Pour les datasets qui tiennent sur un serveur costaud (jusqu'à ~1 To avec les machines modernes), Polars est souvent la meilleure option : performances quasi-distribuées sans la complexité d'un cluster.

**L'échelle de décision pour choisir votre outil** :
1. **< 10 Go** : Pandas, l'outil que tout le monde connaît
2. **10-100 Go** : Polars, la vitesse sans la complexité
3. **100 Go - 1 To** : Dask ou Polars selon le cas
4. **1-100 To** : Spark, le standard éprouvé
5. **100 To+** : Spark avec optimisations custom

Le bon outil, c'est celui qui résout votre problème à votre échelle, sans ajouter de complexité inutile.
