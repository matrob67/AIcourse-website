## De GPT-3.5 brut a ChatGPT : l'alignement en action

OpenAI a utilisé le RLHF pour transformer GPT-3.5, un modèle brut qui complétait du texte sans se soucier de l'utilisateur, en ChatGPT, un assistant utile et (relativement) sûr. Anthropic a fait de même pour Claude. C'est le processus qui transforme un modèle instruit en un modèle **aligné** avec ce que les humains veulent réellement : des réponses utiles, honnêtes, et inoffensives.

Voyons comment ça fonctionne en pratique, avec les outils que vous pouvez utiliser dès aujourd'hui.

<Callout type="info" title="Les trois étapes du RLHF">
1. **SFT** (Supervised Fine-Tuning) : apprendre a suivre des instructions en s'entraînant sur de bonnes réponses
2. **Reward Model** : apprendre ce que les humains préfèrent en comparant des paires de réponses
3. **RL (PPO)** : optimiser le modèle pour maximiser les récompenses du reward model

Chaque étape affine progressivement le comportement. A la fin du pipeline : un modèle qui est a la fois utile, prudent et honnête.
</Callout>

## Étape 1 : SFT (Supervised Fine-Tuning)

C'est du fine-tuning classique sur instructions. Vous prenez votre modèle de base et vous l'entraînez sur des paires (instruction, bonne réponse). C'est le point de départ indispensable : sans SFT, le modèle ne sait même pas qu'il est censé répondre a des questions.

Avec TRL (Transformer Reinforcement Learning), la librairie de Hugging Face :

```python
from trl import SFTTrainer, SFTConfig

config = SFTConfig(
    output_dir="./sft_model",
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,
    learning_rate=2e-5,
    num_train_epochs=3,
)

trainer = SFTTrainer(
    model=model,
    args=config,
    train_dataset=dataset,
)
trainer.train()
```

Résultat : un modèle capable de suivre des instructions. Pas encore aligné, mais au moins coopératif.

## Étape 2 : entraîner le Reward Model

Le problème fondamental : qu'est-ce qu'une "bonne" réponse ? Les humains le savent intuitivement, mais il faut encoder cette intuition dans un modèle.

La solution : un **reward model** entraîné sur des **paires de comparaison**. On montre deux réponses a un annotateur humain, et il indique laquelle il préfère. Pas de score absolu, juste "A est mieux que B". C'est bien plus facile pour un humain que de donner une note sur 10.

### Format des données de préférence

```json
{
  "prompt": "Explique la relativité générale simplement.",
  "chosen": "La relativité générale est la théorie d'Einstein...",
  "rejected": "E=mc2 est une équation de physique..."
}
```

### La loss Bradley-Terry

$$\mathcal{L} = -\log\left(\sigma\left(r(\text{chosen}) - r(\text{rejected})\right)\right)$$

Le principe est simple : le reward model attribue un score $r$ a chaque réponse. Si $r(\text{chosen}) > r(\text{rejected})$, la loss diminue. Petit a petit, le modèle apprend a distinguer les bonnes réponses des mauvaises selon les critères humains.

<Callout type="paper" title="La qualité des annotations est le goulot d'étranglement">
Si l'accord entre annotateurs humains tombe en dessous de 70%, le reward model qui en résulte sera médiocre. L'annotation est le maillon le plus critique du pipeline : coûteux, sujet aux biais, mais absolument essentiel. Un reward model entraîné sur des annotations incohérentes donnera un modèle final incohérent.
</Callout>

## Étape 3 : optimisation PPO

A ce stade, vous avez deux ingrédients :
- Un modèle SFT (la "policy" qu'on veut améliorer)
- Un reward model (le "juge" qui évalue les réponses)

PPO (Proximal Policy Optimization) les combine : le modèle génère des réponses, le reward model les évalue avec un score, et PPO ajuste les poids du modèle pour produire des réponses mieux notées. C'est une boucle d'amélioration continue.

### La pénalité KL : le garde-fou indispensable

Sans contrôle, le modèle va inévitablement "hacker" le reward model. Il va trouver des patterns qui maximisent le score sans vraiment améliorer la qualité — par exemple, générer des réponses absurdement longues parce que le reward model donne légèrement plus de points aux réponses détaillées.

La solution est une pénalité KL qui empêche le modèle de trop s'éloigner de sa version SFT initiale :

$$\text{objective} = \mathbb{E}[\text{reward}] - \beta \times \text{KL}(\pi_\theta \| \pi_{\text{ref}})$$

Le terme KL mesure a quel point le modèle a divergé du modèle de référence. $\beta$ contrôle la force de cette contrainte : trop bas, le modèle hacke le reward model ; trop haut, le modèle n'apprend rien car il est trop contraint.

<Callout type="warning" title="Le reward hacking est un vrai danger">
Sans pénalité KL, le modèle dégénère rapidement. Il apprend a exploiter les failles du reward model au lieu de véritablement s'améliorer. Un exemple classique : le modèle répète des phrases-clés que le reward model aime, produisant du contenu qui "sonne bien" mais ne dit rien. Réglez $\beta$ avec soin — c'est l'hyperparamètre le plus critique du RLHF.
</Callout>

## TRL : le framework de référence pour expérimenter

TRL (Transformer Reinforcement Learning) de Hugging Face rend le RLHF accessible aux praticiens :

- **SFTTrainer** : instruction tuning en quelques lignes
- **RewardTrainer** : entraînement du reward model sur des données de préférence
- **PPOTrainer** : l'optimisation RL complète
- Intégration native avec PEFT (LoRA, QLoRA) pour réduire la mémoire nécessaire
- Support multi-GPU et DeepSpeed pour passer a l'échelle

C'est l'outil de référence pour l'expérimentation et les modèles de taille modeste (7-13B paramètres).

## OpenRLHF : quand il faut passer a l'échelle

TRL fonctionne très bien pour des modèles de 7 a 13 milliards de paramètres. Mais pour les gros modèles (34B, 70B et au-dela), il faut une infrastructure plus sophistiquée.

**OpenRLHF** adopte une architecture distribuée basée sur Ray, avec des rôles séparés sur des GPU différents :

- **Actor** (le modèle qu'on entraîne), **Critic** (la value head), **Reward Model**, et **Reference Model** tournent chacun sur leurs propres GPU
- Support de modèles jusqu'a 70B+ paramètres
- Optimisations mémoire avancées (offloading, gradient checkpointing)
- Supporte PPO, DPO, GRPO, et d'autres algorithmes

<Callout type="tip" title="TRL vs OpenRLHF : comment choisir">
**TRL** : pour l'expérimentation, modèles 7-13B. Facile a prendre en main, parfaitement intégré a l'écosystème Hugging Face.
**OpenRLHF** : pour la production, modèles 34B-70B+. Plus complexe a configurer, mais conçu pour le scaling massif.
</Callout>

## Les défis pratiques du RLHF

### L'instabilité de PPO sur les LLM

PPO appliqué aux LLM est notoirement capricieux. Les hyperparamètres (learning rate, $\beta$ KL, clip ratio, nombre d'époques PPO) demandent un tuning très soigneux. Un learning rate légèrement trop élevé peut faire diverger le modèle en quelques steps. C'est un exercice de funambule.

### Le coût mémoire est brutal

Le pipeline RLHF complet nécessite **quatre modèles en mémoire simultanément** :
1. La policy (le modèle qu'on entraîne)
2. Le modèle de référence SFT (gelé, pour calculer la KL)
3. Le reward model (gelé, pour évaluer)
4. La value head (pour estimer les avantages)

Pour un modèle de 7B en FP16 : comptez environ 56 Go de VRAM rien que pour les poids, avant même de compter les activations et les gradients.

### La génération est le bottleneck

PPO demande de **générer** des réponses complètes a chaque step (pas juste un forward pass). La génération auto-régressive, token par token, est lente par nature. Résultat : chaque step PPO coûte bien plus cher en temps qu'un step SFT classique.

## Les alternatives a PPO : simplifier le pipeline

La complexité du RLHF classique a motivé le développement d'alternatives plus simples :

- **DPO** (Direct Preference Optimization) : élimine le reward model ET l'étape RL. Entraîne directement sur les préférences humaines
- **GRPO** (Group Relative Policy Optimization) : utilisé par DeepSeek, simplifie le calcul des avantages
- **KTO** (Kahneman-Tversky Optimization) : nécessite juste des labels bon/mauvais, pas de paires comparatives
- **ORPO** (Odds Ratio Preference Optimization) : combine SFT et alignement en une seule passe

<Callout type="warning" title="DPO gagne du terrain sur PPO">
DPO est plus simple a implémenter, plus stable a entraîner, et consomme beaucoup moins de mémoire (pas besoin de quatre modèles en VRAM). De plus en plus de praticiens l'adoptent comme méthode d'alignement par défaut. Le RLHF classique avec PPO reste la référence théorique, mais DPO domine en pratique pour les équipes avec des ressources limitées.
</Callout>

**L'essentiel** : le RLHF est le processus qui transforme un modèle instruit en un modèle aligné avec les préférences humaines. C'est complexe, coûteux, et demande un soin considérable dans chaque étape. Mais c'est précisément ce qui fait la différence entre un LLM qui complète du texte et un assistant qui vous est réellement utile.
