## L'idée clé

En Data Parallelism, chaque GPU a le modèle entier. Trop gros ? **Tensor Parallelism** : divisez les matrices de poids entre GPU.

Si une couche attention pèse 12 GB, divisez-la sur 2 GPU = 6 GB chacun.



## Comment ça marche : découpe de matrices

Pour une multiplication `Y = X × W` :

**Par colonnes** : partitionnez W en colonnes, chaque GPU calcule son morceau de Y en parallèle. Zéro communication !

```
W = [W_1 | W_2 | ... | W_P]
Y = X × W = [X×W_1 | X×W_2 | ... | X×W_P]
```

**Par lignes** : partitionnez W en lignes, besoin d'agréger (all-reduce) les résultats.

```
W = [W_1; W_2; ...; W_P]
Y = sum(X[...] × W_i)  (chaque GPU fait un morceau)
```

<Callout type="info" title="Choix architectural">
Dans un Transformer, TP s'applique aux GEMM (attention, MLP). Les normes et embeddings sont répliqués. La stratégie : TP sur les opérations coûteuses, réplication sur les cheap.
</Callout>

## TP dans l'attention multi-tête

L'attention multi-tête est faite de H têtes indépendantes. Distribution naturelle :

- GPU 0 calcule H/2 têtes
- GPU 1 calcule H/2 têtes

Chaque GPU traite ses têtes **indépendamment** (zéro communication intra-tête).

Après : un all-reduce pour combiner les projections de sortie.

Très efficace !

## TP dans la couche MLP

Typique : `Y = dropout(GELU(X×W1)×W2)`

Stratégie :
- W1 : partition par colonnes (GELU est élément-par-élément, compatible)
- W2 : partition par lignes, all-reduce pour agréger

```
GPU 0 : GELU(X×W1_0)  → Z_0
GPU 1 : GELU(X×W1_1)  → Z_1
...
Chaque GPU calcule Z_i × W2_i, puis all-reduce pour Y final
```

<Callout type="paper" title="Megatron-LM">
L'architecture par Megatron-LM place stratégiquement les all-reduce pour minimiser la communication. Une couche Transformer entière = 2 all-reduce (forward) + 2 (backward). Indépendant de la taille du modèle !
</Callout>

## Le coût : communications

À chaque all-reduce, vous communiquez un tenseur de taille `B × L × d` (batch × séquence × dimension).

Pour P GPU, volume total ~ `2 × (P-1)/P × B × L × d` (facteur 2 pour reduce-scatter + all-gather).

**Critique** : les communications sont sur le chemin critique. Pas de recouvrement. Le modèle **attend** les communications.

<Callout type="warning" title="TP demande NVLink">
TP ne doit être utilisé qu'entre GPU du même noeud connectés par NVLink (900 GB/s sur H100). Entre noeuds via InfiniBand (100-400 Gb/s), c'est un goulot : la latence et la bande passante ralentissent le calcul.
</Callout>

## Degré de TP

Combien de GPU pour le TP ?

Typiquement : 2, 4 ou 8 (nombre de GPU connectés par NVLink au sein d'un noeud DGX).

| Degré TP | Overhead comm. | Usage |
|---|---|---|
| 1 (pas TP) | 0% | Modèles < 13B |
| 2 | 3-5% | 13-30B |
| 4 | 8-12% | 30-70B |
| 8 | 15-25% | 70B+ |

Au-delà de 8, les matrices par GPU deviennent trop petites pour utiliser les Tensor Cores efficacement.

<Callout type="tip" title="Stratégie pratique">
Utilisez le **plus petit** degré TP qui laisse le modèle tenir en mémoire. Complétez avec **Pipeline Parallelism** (distribuer couches sur noeuds) et **Data Parallelism** (batch distribution). Typiquement : TP=8 au sein du noeud, pipeline entre noeuds, DDP pour batch.
</Callout>

## Sequence Parallelism (bonus optimization)

Certaines opérations (LayerNorm, dropout, connexion résiduelle) ne se prêtent pas au TP. Elles sont répliquées sur tous les GPU du TP, gaspillant calcul/mémoire.

**Sequence Parallelism** : partitionnez ces opérations le long de la dimension de séquence.

```
TP : partition par dimension cachée (GEMM)
SP : partition par séquence (norm, dropout)
Transitions : all-gather et reduce-scatter
```

Gain : mémoire et calcul utilisés plus efficacement.

## Combinaisons multiples

TP rare tout seul. Généralement :

- **TP + Data Parallelism** : chaque groupe TP traite un micro-batch différent
- **TP + Pipeline Parallelism** : couches distribuées en stages, chaque stage utilise TP
- **TP + ZeRO** : partitionnement optimiseur entre groupes TP

Ça s'appelle **parallélisme 3D** (TP × PP × DP). Le standard pour les LLM énormes.

<Callout type="info" title="Scalabilité massale">
Pour entraîner un modèle 175B (GPT-3), aucune single technique ne suffit. On empile TP (au sein du noeud) + PP (entre noeuds) + DP (batch distribution). Orchestration complexe mais nécessaire.
</Callout>

## Conclusion

Tensor Parallelism découpe les couches pour tenir sur plusieurs GPU. Essentiel pour les gros modèles. Mais demande une bande passante inter-GPU élevée. C'est pourquoi TP se limite au sein d'un noeud NVLink, et on combine avec PP et DP pour le scaling multi-noeud.
