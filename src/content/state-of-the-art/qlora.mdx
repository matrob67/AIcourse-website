## Le problème : fine-tuner un modèle de 65B paramètres

Même avec LoRA (Low-Rank Adaptation, qui n'entraîne que quelques millions de paramètres), le fine-tuning d'un modèle de 65 milliards de paramètres nécessite de charger **tous les poids** en mémoire GPU. En FP16 (16 bits par paramètre), 65B paramètres occupent environ 130 Go de VRAM. Aucun GPU seul ne peut gérer ça.

C'est là que QLoRA entre en jeu. L'idée est simple mais puissante : stocker le modèle de base dans un format **ultra-compressé** (4 bits par paramètre au lieu de 16), tout en gardant les adaptateurs LoRA en pleine précision pour ne pas sacrifier la qualité de l'apprentissage. Le modèle de base est figé et compressé, seuls les petits adaptateurs LoRA sont entraînés en BF16.

<ArxivLink id="2305.14314" title="QLoRA: Efficient Finetuning of Quantized LLMs (Dettmers et al., 2023)" />

## Les trois innovations techniques de QLoRA

### 1. NormalFloat 4-bit (NF4)

Le format NF4 est un type de données 4-bit spécialement conçu pour les poids de réseaux de neurones. L'observation clé qui le motive : les poids pré-entraînés suivent approximativement une **distribution normale** (une gaussienne centrée sur zéro). La plupart des poids sont proches de zéro, avec quelques valeurs extrêmes rares.

<Callout type="paper" title="Quantiles optimaux">
En quantification classique (INT4), les 16 niveaux de quantification sont répartis uniformément sur la plage de valeurs. C'est du gaspillage : beaucoup de niveaux tombent dans des zones peu peuplées. NF4 place les 16 niveaux aux **quantiles** de la distribution normale, ce qui signifie que chaque niveau couvre une proportion égale de la distribution. Résultat : l'erreur de quantification est minimisée pour des données normalement distribuées.
</Callout>

Concrètement, NF4 atteint une qualité de représentation significativement meilleure que INT4 ou FP4 classique, avec exactement la même empreinte mémoire (4 bits par poids). C'est de la précision gratuite, obtenue simplement en choisissant mieux les niveaux de quantification.

### 2. Double Quantification

Toute quantification par blocs nécessite des **constantes de scaling** : pour chaque bloc de 64 poids, on stocke une constante FP32 (4 octets) qui permet de reconvertir les valeurs 4-bit en valeurs réelles. Avec 64 éléments à 4 bits (32 octets par bloc), ces constantes représentent un overhead de 12,5%.

L'astuce de QLoRA : quantifier les constantes elles-mêmes. Au lieu de stocker chaque constante en FP32 (4 octets), on les quantifie en FP8 (1 octet). L'overhead tombe à environ 3%.

Ça semble marginal, mais sur un modèle de 65B paramètres, cette optimisation économise plusieurs gigaoctets de VRAM. À cette échelle, chaque Go compte.

### 3. Paged Optimizers

Pendant le fine-tuning, l'optimiseur Adam stocke deux états par paramètre (momentum et variance), ce qui peut consommer beaucoup de mémoire. QLoRA utilise un système de **pagination GPU-CPU** : quand la VRAM est insuffisante, les états de l'optimiseur sont automatiquement déplacés en RAM CPU, puis rechargés sur le GPU quand nécessaire.

<Callout type="info" title="Gestion de mémoire transparente">
Cette technique exploite la fonctionnalité Unified Memory de CUDA pour gérer automatiquement les transferts page par page entre GPU et CPU. Pour le développeur, c'est totalement transparent : pas de code supplémentaire, pas de configuration manuelle. Le système gère les allers-retours tout seul.
</Callout>

## Impact concret sur la mémoire

Voici l'empreinte mémoire comparée pour un modèle LLaMA 65B :

| Méthode | VRAM nécessaire | GPU minimum |
|---|---|---|
| Fine-tuning FP32 | ~780 Go | Cluster multi-GPU |
| Fine-tuning FP16 | ~130 Go | 2x A100 80Go |
| LoRA FP16 | ~130 Go + adaptateurs | 2x A100 80Go |
| QLoRA NF4 | ~33 Go | 1x A100 40Go |
| QLoRA NF4 (7B) | ~6 Go | 1x RTX 3090/4090 |

Remarquez un point important : LoRA seul ne réduit PAS la mémoire nécessaire pour charger le modèle. Il réduit le nombre de paramètres entraînés (et donc les gradients/états optimizer), mais le modèle de base doit toujours tenir en mémoire en FP16. C'est la quantification 4-bit de QLoRA qui divise l'empreinte du modèle de base par 4.

Le modèle de base est stocké en NF4 (4-bit), mais les adaptateurs LoRA sont entraînés en **BF16** pleine précision. Les gradients sont calculés en BF16, rétropropagés à travers les couches quantifiées, et mis à jour uniquement sur les paramètres LoRA.


## Qualité du fine-tuning : la surprise

<Callout type="paper" title="Résultats remarquables">
Le paper démontre que QLoRA NF4 atteint des performances comparables au fine-tuning FP16 complet sur plusieurs benchmarks. L'écart est souvent inférieur à 1%, ce qui est surprenant étant donné la réduction massive de mémoire (4x moins).
</Callout>

Pour prouver leur point, les auteurs ont créé **Guanaco**, une famille de modèles fine-tunés avec QLoRA sur le dataset OASST1. Le Guanaco 65B atteint 99,3% des performances de ChatGPT selon une évaluation humaine sur le benchmark Vicuna, tout en ayant été fine-tuné sur un seul GPU 48 Go en 24 heures. C'est cette démonstration qui a convaincu la communauté.

## Implémentation pratique

QLoRA est parfaitement intégré dans l'écosystème Hugging Face via `bitsandbytes` et `PEFT` :

```python
from transformers import AutoModelForCausalLM, BitsAndBytesConfig
from peft import LoraConfig, get_peft_model

# Configuration de la quantification 4-bit
bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.bfloat16,
    bnb_4bit_use_double_quant=True,
)

# Chargement du modèle quantifié
model = AutoModelForCausalLM.from_pretrained(
    "mistralai/Mistral-7B-v0.1",
    quantization_config=bnb_config,
    device_map="auto",
)

# Application de LoRA
lora_config = LoraConfig(r=16, lora_alpha=32, target_modules="all-linear")
model = get_peft_model(model, lora_config)
```

Quatre paramètres clés dans la config `bitsandbytes` : `load_in_4bit` active la quantification, `nf4` choisit le format NormalFloat, `bfloat16` pour le dtype de calcul, et `double_quant` active la double quantification. Avec ces quatre lignes, tu divises la mémoire par 4.

## Conseils pratiques

### Choix du compute dtype
- **BF16** (recommandé) : meilleure stabilité numérique grâce à sa plage dynamique étendue, adapté aux GPU Ampere et plus récents (RTX 3000+, A100, H100)
- **FP16** : alternative sur les GPU plus anciens qui ne supportent pas BF16

### Rang LoRA avec QLoRA
Avec QLoRA, tu peux te permettre un **rang plus élevé** qu'avec LoRA classique. Pourquoi ? Parce que l'empreinte mémoire des adaptateurs est dérisoire comparée au modèle de base en 4-bit. Un r=32 ou r=64 ajoute quelques dizaines de Mo, alors que le modèle de base en occupe plusieurs Go. Autant en profiter pour maximiser la capacité d'adaptation.

<Callout type="warning" title="Limitations de la quantification 4-bit">
La quantification NF4 introduit une petite perte d'information **irréversible** dans le modèle de base. Pour des tâches nécessitant une très haute précision numérique (mathématiques complexes, code très technique), un fine-tuning en 8-bit ou FP16 peut donner de meilleurs résultats. Évaluez toujours sur votre tâche spécifique avant de conclure.
</Callout>

## Impact sur l'écosystème

QLoRA a été un véritable catalyseur pour la démocratisation du fine-tuning de LLM. Avant QLoRA, adapter un modèle de 7B+ paramètres nécessitait un GPU professionnel (A100 minimum). Après QLoRA, une RTX 3090 ou 4090 suffit pour fine-tuner un 7B, et un A100 suffit pour un 65B.

La conséquence : des milliers de praticiens et chercheurs qui n'avaient pas accès à des clusters GPU ont pu commencer à fine-tuner des LLM. La grande majorité des modèles fine-tunés de la communauté open-source utilisent aujourd'hui QLoRA comme méthode de base. C'est devenu le point d'entrée par défaut du fine-tuning.
