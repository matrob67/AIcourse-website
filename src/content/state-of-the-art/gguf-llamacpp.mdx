## La révolution locale

En mars 2023, Georgi Gerganov a créé llama.cpp : un projet C/C++ pur qui fait tourner les LLM **sans CUDA, sans Python, sans GPU NVIDIA**. Boom ! Soudain, tout le monde pouvait exécuter des modèles de 7B sur un MacBook Air.

<Callout type="info" title="L'impact réel">
llama.cpp a démocratisé les LLM. Au lieu de payer OpenAI ou louer un cloud GPU, vous exécutez **sur votre machine**. Zéro dépendance cloud, données privées. Cela a créé tout un écosystème (Ollama, LM Studio, Jan, GPT4All).
</Callout>

## Le format GGUF

GGUF (remplaçant de GGML) est un format binaire qui contient **tout** en un seul fichier :

1. **Header** : identifiant, version, nombre de tenseurs
2. **Métadonnées** : architecture, hyperparamètres, tokenizer, template de chat
3. **Tenseurs** : les poids du modèle

Avantage : plus besoin de fichiers séparés de config, tokenizer, poids. Tout dans un fichier.



## Types de quantification GGUF

### Les Q-quants (anciens)

- **Q4_0/Q4_1** : 4-bit, le standard initial
- **Q5_0/Q5_1** : 5-bit, meilleure qualité
- **Q8_0** : 8-bit, presque lossless

### Les K-Quants (recommandés)

Ces formats utilisent des **précisions variables** selon l'importance de chaque couche :

- **Q2_K** : 2-3 bits, ultra-compressé, dégradation visible
- **Q3_K_M** : 3 bits, bon compromis pour espace très limité
- **Q4_K_M** : **4 bits, le choix par défaut**
- **Q5_K_M** : 5 bits, haute qualité
- **Q6_K** : 6 bits, quasi lossless (presque pas de perte)

<Callout type="tip" title="Q4_K_M c'est le sweet spot">
Q4_K_M vous donne ~95-97% des performances de FP16 en 4.5 bits effectifs de taille. C'est le format à choisir par défaut. Q5_K_M si vous en avez le temps, Q3_K_M si vous êtes vraiment limité en espace.
</Callout>

## Backends supportés

llama.cpp fonctionne partout :

| Backend | Matériel | Performance |
|---|---|---|
| CPU (AVX2) | Tout PC x86 moderne | Bonne |
| Metal | Mac M1-M4 | Excellente |
| CUDA | GPU NVIDIA | Excellente |
| Vulkan | GPU AMD, Intel | Bonne |
| CANN | NPU Ascend | Bon |

### Apple Silicon (M1-M4)

Metal (GPU intégré au Mac) fait tourner les LLM **vite**. Un M2 Max avec 32 Go de RAM unifié peut exécuter un modèle 30B en Q4_K_M à **10-20 tokens/seconde**. Assez pour un chat fluide.

## Estimation mémoire

Pour un modèle de N milliards de paramètres :

```
Q4_K_M : ~N × 0.6 Go
Exemple: 7B → ~4.5 Go
```

Ajoutez le contexte KV cache (variable selon la longueur).

## Comment quantifier

### Étape 1 : Convertir Hugging Face en GGUF

```bash
python convert_hf_to_gguf.py ./model_hf \
    --outtype f16 \
    --outfile model.f16.gguf
```

Cela crée un gros fichier intermédiaire FP16.

### Étape 2 : Quantifier

```bash
./llama-quantize model.f16.gguf model.Q4_K_M.gguf Q4_K_M
```

Boom ! Un fichier 4.5 Go à la place de 14 Go.

## Fonctionnalités avancées

### Offloading partial GPU

Si le modèle ne tient pas entièrement en VRAM GPU, une partie reste en RAM CPU. Plus lent mais ça marche.

### Prompt caching

Sauvegarder le KV cache d'un prompt système et le réutiliser. Si vous avez 100 requêtes avec le même système prompt, vous le calculez une fois.

### Grammars (GBNF)

Contraindre la génération à un format (JSON, XML, etc.). Le modèle suit la structure au lieu de générer du charabia.

<Callout type="warning" title="Qualité vs GPTQ/AWQ">
La quantification GGUF utilise RTN (simple arrondi), pas les méthodes data-aware comme GPTQ/AWQ. Pour la meilleure qualité sur GPU NVIDIA, AWQ peut offrir mieux. GGUF gagne par sa flexibilité multi-plateforme.
</Callout>
