## La révolution de l'inférence locale

En mars 2023, Georgi Gerganov a créé llama.cpp : un projet en C/C++ pur qui fait tourner des LLM **sans CUDA, sans Python, sans GPU NVIDIA**. Du jour au lendemain, tout le monde pouvait exécuter un modèle de 7 milliards de paramètres sur un MacBook Air. Pas besoin de cloud, pas besoin de carte graphique hors de prix.

<Callout type="info" title={"L'impact réel de llama.cpp"}>
llama.cpp a littéralement démocratisé les LLM. Au lieu de payer OpenAI par token ou de louer un GPU cloud à l'heure, tu exécutes le modèle **sur ta propre machine**. Zéro dépendance cloud, données 100% privées. Ce projet a engendré tout un écosystème d'applications : Ollama, LM Studio, Jan, GPT4All, et bien d'autres.
</Callout>

## Le format GGUF : tout en un fichier

GGUF (qui remplace l'ancien format GGML) est un format binaire conçu pour être **auto-suffisant**. Un seul fichier contient absolument tout ce dont tu as besoin pour faire de l'inférence :

1. **Header** : identifiant du format, version, nombre de tenseurs
2. **Métadonnées** : architecture du modèle, hyperparamètres, tokenizer complet, template de chat
3. **Tenseurs** : les poids du modèle eux-mêmes

L'avantage est énorme : plus besoin de jongler entre un fichier de config, un fichier de tokenizer, des fichiers de poids séparés. Tu télécharges un fichier, tu le lances. C'est tout.



## Les types de quantification GGUF

C'est ici que les choses deviennent intéressantes. GGUF propose plusieurs niveaux de compression, chacun avec un compromis différent entre taille et qualité.

### Les Q-quants (ancienne génération)

- **Q4_0/Q4_1** : 4 bits, le standard initial. Fonctionnel mais rudimentaire.
- **Q5_0/Q5_1** : 5 bits, meilleure qualité mais plus gros.
- **Q8_0** : 8 bits, quasi sans perte mais prend beaucoup de place.

### Les K-Quants (recommandés aujourd'hui)

La vraie innovation. Ces formats utilisent des **précisions variables** : les couches importantes du modèle reçoivent plus de bits, les couches moins critiques en reçoivent moins. C'est de la quantification "intelligente" :

- **Q2_K** : 2-3 bits, ultra-compressé. Le modèle tient dans très peu de RAM, mais la qualité souffre visiblement. Pour les cas désespérés d'espace.
- **Q3_K_M** : ~3.3 bits. Bon compromis quand tu as vraiment peu d'espace. Utilisable, mais tu sentiras la dégradation.
- **Q4_K_M** : ~4.5 bits. **C'est le choix par défaut recommandé.** Le sweet spot entre taille et qualité.
- **Q5_K_M** : ~5.5 bits. Haute qualité, pour ceux qui ont un peu plus de mémoire.
- **Q6_K** : ~6.5 bits. Quasi sans perte, difficile de distinguer du FP16 en pratique.

<Callout type="tip" title="Q4_K_M : le choix par défaut">
Q4_K_M offre environ 95-97% des performances du modèle original FP16, avec un fichier 3 à 4 fois plus petit. C'est le format à choisir en premier. Monte à Q5_K_M si tu veux plus de qualité, descends à Q3_K_M uniquement si tu es vraiment contraint en mémoire.
</Callout>

## Compatibilité matérielle : llama.cpp tourne partout

Un des grands atouts de llama.cpp est son support multi-plateforme :

| Backend | Matériel | Performance |
|---|---|---|
| CPU (AVX2) | Tout PC x86 moderne | Bonne |
| Metal | Mac M1-M4 | Excellente |
| CUDA | GPU NVIDIA | Excellente |
| Vulkan | GPU AMD, Intel | Bonne |
| CANN | NPU Ascend | Bon |

### Le cas Apple Silicon (M1-M4)

Le Metal backend (le GPU intégré des Mac) fait tourner les LLM remarquablement vite. Un Mac M2 Max avec 32 Go de RAM unifiée peut exécuter un modèle 30B en Q4_K_M à **10-20 tokens par seconde**. C'est largement suffisant pour un chat fluide et agréable. La RAM unifiée des Mac est un avantage énorme ici : pas de transfert CPU-GPU, tout est dans le même espace mémoire.

## Estimer la mémoire nécessaire

Une règle simple pour un modèle de N milliards de paramètres :

**Q4_K_M** : environ $N \times 0.6$ Go (exemple : 7B donne environ 4.5 Go)

A cela, il faut ajouter le KV cache qui dépend de la longueur du contexte. Pour un contexte de 8K tokens, comptez 1 à 2 Go supplémentaires selon le modèle.

## Comment quantifier un modèle

### Étape 1 : Convertir depuis le format Hugging Face

```bash
python convert_hf_to_gguf.py ./model_hf \
    --outtype f16 \
    --outfile model.f16.gguf
```

Ça crée un fichier intermédiaire en FP16 (pleine précision). C'est gros mais c'est la base.

### Étape 2 : Quantifier au format souhaité

```bash
./llama-quantize model.f16.gguf model.Q4_K_M.gguf Q4_K_M
```

Et voilà : un fichier de 4.5 Go qui remplace 14 Go. La compression est impressionnante.

## Fonctionnalités avancées

### Offloading partiel GPU

Si ton modèle ne tient pas entièrement dans la VRAM du GPU, llama.cpp peut répartir : une partie des couches sur le GPU (rapide), le reste en RAM CPU (plus lent). C'est moins performant que tout-GPU, mais ça permet d'utiliser un GPU avec peu de VRAM sans abandonner complètement l'accélération matérielle.

### Prompt caching

Tu peux sauvegarder le KV cache d'un system prompt et le réutiliser pour toutes les requêtes suivantes. Si tu as 100 requêtes avec le même system prompt de 2000 tokens, tu le calcules une seule fois. Gain de temps énorme pour les déploiements avec un prompt fixe.

### Grammars (GBNF)

Tu peux contraindre la génération à respecter un format spécifique (JSON, XML, code...) grâce à des grammaires formelles. Le modèle génère token par token en suivant la structure imposée, au lieu de produire du texte libre qui pourrait ne pas parser.

<Callout type="warning" title="Qualité : GGUF vs GPTQ/AWQ">
La quantification GGUF utilise RTN (Round-To-Nearest), un simple arrondi. C'est moins sophistiqué que les méthodes data-aware comme GPTQ ou AWQ qui analysent les données pour optimiser la quantification. Sur GPU NVIDIA en production, AWQ peut offrir une meilleure qualité. Mais GGUF gagne haut la main en flexibilité multi-plateforme.
</Callout>

## L'écosystème GGUF

### Ollama
L'interface la plus simple qui existe :
```bash
ollama run mistral
```
Une seule commande. Le modèle est téléchargé, quantifié, et lancé. Parfait pour les débutants et les tests rapides.

### LM Studio
Une application desktop avec interface graphique. Tu navigues dans un catalogue de modèles, tu cliques, ça tourne. Idéal pour ceux qui ne veulent pas toucher au terminal.

### Jan, GPT4All
D'autres clients desktop pour l'inférence locale, chacun avec sa propre approche de l'expérience utilisateur.

### Hugging Face
Des milliers de modèles pré-quantifiés au format GGUF sont disponibles sur le Hub. Des contributeurs comme TheBloke (maintenant bartowski) quantifient systématiquement les nouveaux modèles dans toutes les variantes.

## En résumé

GGUF + llama.cpp ont rendu les LLM **véritablement locaux et accessibles**. Zéro dépendance, zéro API, données privées. C'est la porte d'entrée idéale pour expérimenter avec les LLM, et un outil de production viable pour les cas où la vie privée ou la latence réseau sont des contraintes.
