## Le problème : certaines questions exigent de connecter les points

Imaginez qu'on vous pose : "Quels sont les fournisseurs des entreprises qui ont eu des problèmes légaux en 2024 ?"

Avec un RAG classique, vous cherchez des passages similaires à cette question dans votre base vectorielle. Mais le problème, c'est que la réponse n'existe dans **aucun document unique**. Il faut **combiner** des informations éparpillées :
- Un document parle de l'entreprise X et de ses problèmes légaux
- Un autre document liste les fournisseurs de l'entreprise X
- Aucun document ne dit directement "voici les fournisseurs de l'entreprise X qui a eu des problèmes légaux"

Le RAG vectoriel cherche des passages qui **ressemblent** à la question. Mais ici, il faut **naviguer entre des entités connectées**. C'est exactement ce que fait **Graph RAG** : il construit un graphe de connaissances qui capture les **relations** entre entités, puis utilise ce graphe pour répondre.

<Callout type="info" ’>
Au lieu de juste chercher des passages, créez un graphe : des nœuds (entreprises, personnes, concepts) reliés par des arêtes ("fournit à", "travaille pour", "a eu un problème"). Le retrieval devient une navigation intelligente dans ce réseau de relations.
</Callout>

## Comment ça marche

### Étape 1 : Construire le graphe depuis les documents

C'est la phase la plus coûteuse. Pour chaque document :
1. Un LLM extrait les **entités** : personnes, entreprises, lieux, concepts, dates
2. Le même LLM identifié les **relations** entre ces entités : "X fournit Y", "Z travaille chez W"
3. On crée les nœuds et arêtes dans un graphe (typiquement Neo4j ou NetworkX)

**Entreprise A** &rarr; *fournit a* &rarr; **Entreprise B** &rarr; *problème legal* &rarr; **Incident 2024**

Le résultat : une structure navigable qui encode explicitement les connexions que le RAG vectoriel devrait deviner.

### Étape 2 : Chercher intelligemment

Quand quelqu'un demande "fournisseurs d'entreprises avec des problèmes légaux" :
1. Identifiez les entités mentionnées dans la question
2. Trouvez les nœuds correspondants dans le graphe
3. Traversez les arêtes pertinentes (fournisseur → entreprise → problème légal)
4. Collectez les informations le long du chemin
5. Retournez ce contexte enrichi au LLM pour générer la réponse

C'est du **raisonnement multi-hop** : on saute d'entité en entité en suivant les relations. Quelque chose que la similarité cosinus ne sait pas faire.



## L'approche Microsoft : hiérarchie et communautés

Microsoft a publié une implémentation de Graph RAG qui ajoute deux idées malignes :

**Community Détection** : les graphes réels sont souvent immenses. L'algorithme de Leiden détecte des **communautés** — des clusters de nœuds fortement interconnectés. Chaque communauté reçoit un résumé généré par un LLM. C'est comme créer automatiquement des "chapitres" dans votre base de connaissances.

Deux modes de recherche en découlent :
- **Local Search** : pour les questions ciblées ("Que sait-on sur l'entreprise X ?") — on explore le voisinage direct du nœud
- **Global Search** : pour les questions de synthèse ("Quels sont les thèmes majeurs de ce corpus ?") — on agrège les résumés des communautés

<Callout type="paper" title="Global Search excelle à la synthèse">
Le RAG vectoriel classique examine les 5-10 passages les plus proches de la question. Graph RAG avec Global Search voit la big picture en synthétisant des thèmes entiers à partir des résumés de communautés. C'est la différence entre lire 5 pages prises au hasard dans un livre et lire la table des matières avec les résumés de chaque chapitre.
</Callout>

## Code simple avec LangChain + Neo4j

```python
from langchain_community.graphs import Neo4jGraph
from langchain.chains import GraphCypherQAChain

graph = Neo4jGraph(url="bolt://localhost:7687",
                   username="neo4j",
                   password="password")

chain = GraphCypherQAChain.from_llm(
    llm=ChatOpenAI(model="gpt-4"),
    graph=graph,
    verbose=True,
)
result = chain.invoke(
    "Fournisseurs liés à des entreprises sanctionnées ?"
)
```

Ce qui se passe sous le capot : le LLM traduit la question en requête Cypher (le SQL des graphes), Neo4j exécute la requête, et le LLM formule la réponse à partir des résultats.

## Le vrai coût : construction et maintenance

L'extraction d'entités et de relations nécessite de passer **chaque document** au LLM. C'est le prix d'entrée, et il n'est pas négligeable :

| Taille | Docs | Coût (GPT-4) | Temps |
|---|---|---|---|
| Petit | 100 | ~5$ | 30 min |
| Moyen | 1 000 | ~50$ | 5h |
| Grand | 10 000 | ~500$ | 50h |

Et ce n'est pas un coût unique : quand les documents changent, le graphe doit être mis à jour. Nouveaux nœuds, nouvelles arêtes, résumés de communautés à recalculer. C'est un investissement continu.

<Callout type="warning" title="Maintenance continue">
Le graphe de connaissances n'est pas un "build once, use forever". Les documents évoluent, les relations changent, de nouvelles entités apparaissent. Prévoyez un pipeline de mise à jour automatique, ou le graphe deviendra rapidement obsolète.
</Callout>

## Quand l'utiliser

**Graph RAG est justifié si :**
- Vos questions sont **multi-hop** (traverser plusieurs relations pour répondre)
- Vous avez besoin de **synthèse globale** sur un large corpus
- Le domaine est relationnel par nature : compliance, audit, supply chain, organigrammes

**Un RAG vectoriel classique suffit si :**
- Les questions sont factuelles et directes ("Quelle est la politique de remboursement ?")
- Le corpus n'a pas de relations complexes entre entités
- Le budget est serré (le RAG vectoriel coûte une fraction du Graph RAG)

<Callout type="tip" title="Approche hybride souvent meilleure">
En pratique, un routeur intelligent décide au cas par cas : question directe et factuelle → RAG vectoriel (rapide et peu coûteux). Question relationnelle ou de synthèse → Graph RAG (plus lent mais plus complet). C'est le meilleur des deux mondes.
</Callout>

Graph RAG est l'évolution naturelle du RAG pour les cas d'usage complexes. Le coût d'entrée est réel, mais pour les questions qui exigent de naviguer entre entités connectées, aucune autre approche n'offre la même qualité de réponse.
