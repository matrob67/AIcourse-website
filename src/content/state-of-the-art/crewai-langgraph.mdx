## Deux philosophies : équipe vs graphe

Quand on veut faire travailler plusieurs agents IA ensemble, deux grandes approches dominent. On peut les résumer par une analogie simple :

**CrewAI**, c'est comme monter une **équipe de projet**. Tu définis des rôles (chercheur, rédacteur, analyste), tu leur donnes des missions, et un chef d'équipe coordonne le tout. C'est intuitif : on pense en termes de "qui fait quoi".

**LangGraph**, c'est comme dessiner un **organigramme de décision** (un graphe). Chaque étape est un nœud, chaque flèche est une transition. L'état du système circule d'un nœud à l'autre. On pense en termes de "quand est-ce qu'on fait quoi, et que se passe-t-il si tel condition est remplie".

<Callout type="info" title="Choisir entre les deux">
CrewAI si tu veux prototyper rapidement avec des rôles clairs et une logique simple.
LangGraph si tu as besoin d'un contrôle fin du flux, avec des branchements conditionnels et une gestion d'état rigoureuse.
</Callout>

## CrewAI : l'équipe intuitive

CrewAI repose sur trois abstractions qui parlent d'elles-mêmes :

- **Agent** : un personnage avec un rôle, un objectif, une histoire (backstory), et des outils à sa disposition
- **Task** : une tâche spécifique assignée à un agent, avec un output attendu
- **Crew** : l'équipe qui orchestre le travail de tous les agents

L'idée est naturelle : tu construis ton équipe comme un manager construirait une task force. Le chercheur cherche, le rédacteur rédige, et CrewAI s'assure que les résultats passent d'un agent à l'autre dans le bon ordre.

```python
from crewai import Agent, Task, Crew

researcher = Agent(
    role="Chercheur",
    goal="Trouver infos précises",
    backstory="Expert en recherche doc",
    tools=[search_tool, scrape_tool],
)

writer = Agent(
    role="Rédacteur",
    goal="Articles clairs et engageants",
    backstory="Journaliste 10 ans expérience",
)

research_task = Task(
    description="Chercher les avancées IA génératives",
    agent=researcher,
    expected_output="Résumé structuré des avancées",
)

writing_task = Task(
    description="Rédiger article",
    agent=writer,
    context=[research_task],  # Dépend de research_task
    expected_output="Article blog 800 mots",
)

crew = Crew(
    agents=[researcher, writer],
    tasks=[research_task, writing_task]
)
result = crew.kickoff()
```

**Modes d'exécution** :
- **Sequential** : les tâches s'exécutent dans l'ordre, chacune recevant le résultat de la précédente. C'est le mode par défaut, simple et prévisible.
- **Hierarchical** : un "manager agent" est créé automatiquement. Il reçoit les tâches et décide quel agent mobiliser, un peu comme un chef de projet qui délègue intelligemment.

**Les forces de CrewAI** :
- API lisible et intuitive, on comprend le code en le lisant
- La métaphore d'équipe rend le design naturel pour les non-techniques
- Intégration facile d'outils (recherche web, scraping, API...)
- Support de la délégation entre agents

## LangGraph : le contrôle total par graphe d'états

LangGraph prend une approche radicalement différente. Au lieu de penser en personnes et tâches, on modélise le workflow comme un **graphe orienté** :

- **State** : un dictionnaire typé qui représente l'état courant du système (la question posée, les documents trouvés, la réponse générée...)
- **Nodes** : des fonctions Python qui transforment l'état. Chaque nœud lit une partie de l'état, fait son travail, et met à jour l'état.
- **Edges** : les transitions entre nœuds, qui peuvent être conditionnelles (si les documents sont pertinents, va vers "generate" ; sinon, va vers "web_search")

```python
from langgraph.graph import StateGraph, END
from typing import TypedDict, List

class RAGState(TypedDict):
    question: str
    documents: List[str]
    answer: str
    needs_web_search: bool

graph = StateGraph(RAGState)

# Nœuds
graph.add_node("retrieve", retrieve_from_vectordb)
graph.add_node("grade", grade_relevance)
graph.add_node("web_search", search_web)
graph.add_node("generate", generate_answer)

# Edges
graph.set_entry_point("retrieve")
graph.add_edge("retrieve", "grade")
graph.add_conditional_edges("grade", decide_route, {
    "relevant": "generate",
    "not_relevant": "web_search",
})
graph.add_edge("web_search", "generate")
graph.add_edge("generate", END)

app = graph.compile()
```

Ce code implémente un RAG adaptatif : on cherche dans la base, on évalue la pertinence, et si les résultats ne sont pas bons, on fait une recherche web avant de générer. Ce genre de logique conditionnelle est trivial avec LangGraph, mais pénible à implémenter avec CrewAI.

**Les forces de LangGraph** :
- Contrôle total sur chaque transition et condition
- Excellent debugging grâce à la visualisation du graphe
- Persistance native (checkpointing) : reprenez un workflow interrompu exactement là où il s'est arrêté
- Streaming token par token intégré
- Human-in-the-loop avancé : pausez le graphe à n'importe quel nœud pour demander une validation humaine

<Callout type="tip" title="LangGraph pour la production">
LangGraph offre une prévisibilité que les frameworks conversationnels ne donnent pas. Un graphe est déterministe et testable : pour un état donné, le chemin suivi est toujours le même. C'est exactement ce dont on a besoin en production.
</Callout>



## Comparaison directe

| Aspect | CrewAI | LangGraph |
|---|---|---|
| Paradigme | Équipe par rôles | Graphe d'états |
| Abstraction | Haute | Basse |
| Contrôle flux | Séquentiel/hiérarchique | Graphe arbitraire |
| Courbe d'apprentissage | Faible | Moyenne |
| Debugging | Modéré | Excellent (viz) |
| Persistance | Limitée | Native |
| Human-in-the-loop | Basique | Avancé |
| Production-ready | En progression | Oui |

## Quand choisir quoi

**CrewAI est idéal pour** :
- Le prototypage rapide de systèmes multi-agents
- Les cas d'usage avec des rôles bien définis (recherche, analyse, rédaction)
- Les équipes peu techniques qui veulent un code lisible
- Les pipelines linéaires : A fait son travail, passe à B, qui passe à C

**LangGraph est idéal pour** :
- Les workflows complexes avec des branchements conditionnels et des boucles
- La production avec des exigences de fiabilité et de monitoring
- Les systèmes RAG avancés avec routing intelligent entre sources
- Les cas où l'humain doit intervenir à des moments précis du workflow

<Callout type="warning" title="Multi-agent coûte cher">
Quel que soit le framework choisi, les systèmes multi-agents sont intrinsèquement moins fiables et plus lents qu'un agent unique bien outillé. Avant de créer une armée d'agents, demande-toi : un seul agent avec les bons outils ne suffirait-il pas ? La complexité organisationnelle à un coût réel en tokens, en latence et en débugage.
</Callout>

## L'évolution de l'écosystème

L'écosystème des frameworks agents converge progressivement. On voit apparaître des solutions qui cherchent à combiner la simplicité de conception de CrewAI avec le contrôle d'exécution de LangGraph. Par exemple, **Smolagents** de Hugging Face propose une approche code-first plus légère, et des plateformes no-code permettent de construire des graphes d'agents visuellement.

Le meilleur choix dépend de tes besoins concrets : si tu veux aller vite et que ton workflow est simple, CrewAI. Si tu veux du contrôle, de la testabilité et de la robustesse en production, LangGraph. Et dans certains cas, combiner les deux est la meilleure stratégie.
