## Qu'est-ce que c'est, concrètement ?

Un LLM ne sait faire qu'une chose : générer du texte. Mais dans la vraie vie, tu veux qu'il fasse des trucs concrets : chercher dans une base de données, lancer un calcul, envoyer un email, réserver un vol. Comment on passe du texte à l'action ?

C'est là qu'intervient le **function calling**. Le principe est simple : tu décris au LLM les fonctions disponibles (nom, description, paramètres attendus), et quand il juge pertinent d'en appeler une, il génère un JSON structuré qui dit "j'appelle cette fonction avec ces paramètres". Ton application côté serveur reçoit ce JSON, exécute la fonction correspondante, renvoie le résultat au LLM, et celui-ci l'intègre dans sa réponse finale.

<Callout type="info" title="Qui exécute vraiment ?">
Le LLM n'exécute jamais rien lui-même. Il génère une **spécification d'appel** : `{"name": "get_weather", "arguments": {"city": "Paris"}}`. C'est ton application qui l'exécute, récupère le résultat, et le renvoie au LLM. Le LLM est le cerveau qui décide, ton code est les mains qui agissent.
</Callout>

## Anatomie d'un function call

### Décrire une fonction au LLM

La qualité de la description détermine directement la qualité d'utilisation par le LLM. Pense à cette description comme la documentation d'une API publique : claire, précise, avec des exemples implicites dans les noms de paramètres.

```json
{
  "name": "get_weather",
  "description": "Obtenir la météo actuelle d'une ville. Retourne la température et les conditions.",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "Nom de la ville (ex: Paris, Tokyo)"
      },
      "unit": {
        "type": "string",
        "enum": ["celsius", "fahrenheit"],
        "description": "Unité de température souhaitée"
      }
    },
    "required": ["city"]
  }
}
```

### Le flux complet en 4 étapes

1. **User** : "Quel temps fait-il à Paris ?"
2. **LLM** produit : `{"name": "get_weather", "arguments": {"city": "Paris", "unit": "celsius"}}`
3. **Ton app** exécute la fonction, récupère : `{"temp": 18, "condition": "nuageux"}`
4. **LLM** reçoit le résultat et répond : "Il fait 18 degres à Paris, temps nuageux."

Le LLM n'a jamais accédé à Internet. Il a juste compris que la question nécessitait des données météo, a formulé la bonne requête, et a intégré la réponse dans une phrase naturelle. Tout le "travail réel" est fait par ton code.



## Les règles d'or de la conception

**Noms explicites** : `search_company_database` est infiniment meilleur que `query` ou `tool1`. Le LLM se fie au nom pour comprendre quand utiliser la fonction.

**Descriptions détaillées** : c'est ta seule chance d'expliquer au LLM ce que la fonction fait, quand l'utiliser, et ce qu'elle retourne. Ne lésine pas.

**Paramètres bien typés** : strings, numbers, booleans, enums. Plus c'est contraint, moins le LLM fera d'erreurs. Un `enum: ["celsius", "fahrenheit"]` est mieux qu'un `type: string` avec "unité de température".

**Retours informatifs** : le résultat doit aider le LLM à comprendre ce qui s'est passé.
```json
// Bon : le LLM comprend immédiatement
{"status": "success", "results": [...], "count": 42}

// Mauvais : le LLM ne sait pas si c'est vide ou erreur
{"data": null}
```

<Callout type="tip" title="La règle des 3 à 7 outils">
Un LLM fonctionne nettement mieux avec 3 à 7 outils bien conçus qu'avec 50 outils vagues. Trop d'outils = confusion, appels au mauvais outil, paramètres mélangés. Au-delà de 10 outils, pense à un système de routing ou une hiérarchie de sélection.
</Callout>

## Patterns avancés

### Appels parallèles

Les modèles modernes peuvent appeler plusieurs fonctions en parallèle quand elles sont indépendantes :

> **User :** "Météo à Paris et Tokyo ?"
> **LLM** &rarr; `[get_weather("Paris"), get_weather("Tokyo")]` -- les deux en parallèle

Ton app exécute les deux, renvoie les résultats, et le LLM compose une réponse unifiée. C'est beaucoup plus rapide qu'attendre chaque appel séquentiellement.

### Appels séquentiels (chaînés)

Parfois le résultat d'un appel est nécessaire pour le suivant :

1. `search_user("Jean")` &rarr; `{id: 123}`
2. `get_orders(user_id=123)` &rarr; `[...]`

Le LLM gère naturellement cette dépendance : il attend le résultat du premier appel avant de formuler le second.

### Structured output via function calling

Une astuce peu connue : utiliser le function calling pour **forcer un output structuré**, même si la "fonction" n'exécute rien. Tu définis une fausse fonction dont les paramètres correspondent à la structure que tu veux extraire :

```json
{
  "name": "extract_entities",
  "parameters": {
    "properties": {
      "persons": {"type": "array", "items": {"type": "string"}},
      "organizations": {"type": "array"}
    }
  }
}
```

Le LLM remplit les paramètres = tu obtiens un JSON structuré fiable. C'est plus robuste que de demander "renvoie-moi un JSON".

## Les erreurs courantes (et comment s'en remettre)

1. **Paramètres de mauvais type** : le LLM passe une string quand un number est attendu
2. **Hallucination de fonction** : le LLM invente une fonction qui n'existe pas dans ta liste
3. **Mauvais outil choisi** : le LLM sélectionne le mauvais outil pour la tâche
4. **Paramètres manquants** : oubli d'un paramètre marqué comme requis

**Stratégies de recovery** :
- **Renvoie l'erreur au LLM** : dans la majorité des cas, il se corrige tout seul au second essai
- **Validation côté serveur** : vérifie les types et les contraintes avant d'exécuter
- **Fallback** : si un outil échoue, propose une approche alternative

<Callout type="warning" title="La sécurité avant tout">
Le function calling donne au LLM un **pouvoir d'action réel** sur ton système. Valide toujours les paramètres, limite les permissions au strict nécessaire, sandboxe les exécutions. Un LLM manipulé par injection de prompt pourrait tenter d'appeler des fonctions dangereuses avec des paramètres malveillants.
</Callout>

## Les bonnes pratiques en résumé

1. Commence avec peu d'outils (3-5), ajoute progressivement
2. Teste chaque outil avec des inputs variés, valides ET malformés
3. Documente tes outils comme une API publique : noms clairs, descriptions complètes
4. Logue tous les appels pour le debug et l'audit
5. Mets des timeouts sur chaque exécution
6. Limite le nombre d'appels par conversation pour éviter les boucles infinies

Le function calling est la brique fondamentale des agents modernes. Bien conçu, c'est un mécanisme simple et robuste qui transforme un LLM de "générateur de texte" en "acteur capable d'interagir avec le monde réel".
