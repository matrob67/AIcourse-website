## Le cauchemar de l'intégration

Pour comprendre pourquoi MCP existe, il faut d'abord comprendre le problème qu'il résout.

Imaginez le monde de l'informatique avant USB. Votre souris utilisait un connecteur série. Votre clavier, un PS/2. Votre imprimante, un port parallèle. Votre scanner, du SCSI. Chaque périphérique avait son propre connecteur physique, son propre driver, son propre protocole. Brancher un nouvel appareil était une aventure.

C'est exactement la situation des agents IA aujourd'hui. Vous voulez connecter votre agent à PostgreSQL ? Il vous faut un connecteur custom. À Slack ? Un autre connecteur. À Google Drive ? Encore un autre. Et si vous avez 50 outils différents ? Vous écrivez 50 intégrations différentes, chacune avec ses propres particularités, ses propres bugs, et sa propre maintenance.

**MCP est le "standard USB" des agents IA**. Un seul protocole ouvert. N'importe quel agent compatible MCP peut se connecter à n'importe quel outil compatible MCP. Sans code d'intégration spécifique.

<Callout type="info" title="L'analogie USB, en détail">
USB a standardisé la couche physique (le connecteur) ET la couche logique (le protocole de communication). MCP fait la même chose pour les agents IA : il standardise le format des messages (JSON-RPC) ET la sémantique des interactions (comment exposer un outil, comment lire une ressource, comment gérer les permissions). Le résultat : un écosystème où les pièces s'emboîtent sans friction.
</Callout>

## Architecture : les trois pièces du puzzle

MCP suit une architecture client-serveur simple mais bien pensée :

1. **MCP Host** : votre application IA --- Claude Desktop, un IDE comme Cursor, ou un agent custom que vous avez développé. C'est le "cerveau" qui décide quand et comment utiliser les outils
2. **MCP Client** : le composant technique côté host qui gère la communication (ouverture de connexion, envoi de requêtes, réception de réponses). En général, vous n'y touchez pas directement
3. **MCP Server** : le service qui expose des ressources et des outils à l'agent. C'est ici que la magie opère --- un serveur MCP PostgreSQL sait lire une base de données, un serveur MCP GitHub sait naviguer dans les repos

### Le protocole sous-jacent

MCP utilise **JSON-RPC 2.0** --- un standard internet éprouvé pour les appels de fonctions à distance --- sur différents transports :

- **Stdio** : communication via entrée/sortie standard (idéal pour les serveurs locaux)
- **HTTP/SSE** : communication web avec Server-Sent Events (pour les serveurs distants)

Chaque serveur MCP expose trois types de capacités :

- **Resources** : des données que le LLM peut lire. Fichiers locaux, résultats de requêtes SQL, contenu d'APIs, documents Google Drive
- **Tools** : des fonctions que le LLM peut appeler pour agir. Créer un ticket Jira, envoyer un message Slack, exécuter une commande
- **Prompts** : des templates de prompts pré-configurés et optimisés pour des tâches spécifiques

### Le flux de communication en action

Voici ce qui se passe concrètement quand vous demandez à Claude d'analyser vos données :

1. **Vous** &rarr; **Claude** : "Analyse les ventes du dernier trimestre dans ma base de données"
2. **Claude** reconnaît qu'il a besoin de données et formule une requête
3. **Client MCP** &rarr; JSON-RPC &rarr; **Serveur MCP PostgreSQL**
4. Le serveur exécute la requête SQL et renvoie les résultats
5. **Claude** reçoit les données et produit son analyse
6. **Claude** &rarr; **Vous** : "Voici l'analyse : les ventes ont augmenté de 15%..."

Tout ce processus est transparent. Vous ne voyez que le résultat final.

## L'écosystème des serveurs MCP

### Serveurs de données

- **Filesystem** : lire et naviguer dans vos fichiers locaux
- **PostgreSQL / SQLite / MySQL** : accéder à vos bases de données
- **Google Drive** : lire vos documents cloud
- **GitHub** : accéder aux repositories, issues, pull requests

### Serveurs d'outils

- **Brave Search** : faire des recherches web en temps réel
- **Puppeteer** : naviguer sur le web de manière automatisée (scraping, tests)
- **Memory** : mémoire persistante pour que l'agent se souvienne entre les sessions
- **Sequential Thinking** : outils de raisonnement structuré pour les problèmes complexes

### Serveurs custom

N'importe quel développeur peut créer son propre serveur MCP pour exposer ses données et outils internes. Et c'est accessible : les SDKs officiels rendent la tâche simple.

<Callout type="tip" title="Créer votre propre serveur MCP">
Les SDKs officiels (Python, TypeScript/JavaScript) permettent de créer un serveur fonctionnel en 30 lignes de code. Un registre communautaire regroupe des centaines de serveurs prêts à l'emploi. Pour la plupart des outils populaires (Slack, Notion, Jira), quelqu'un a déjà fait le travail.
</Callout>



## Exemple concret : un serveur MCP minimaliste en Python

```python
from mcp.server import Server, stdio_transport
from mcp.types import Tool, TextContent

server = Server("mon-serveur")

@server.tool("interroger_database")
async def query_db(requete: str) -> list[TextContent]:
    """Exécute une requête SQL sur la base de données."""
    resultats = await db.execute(requete)
    return [TextContent(type="text", text=str(resultats))]

@server.tool("obtenir_meteo")
async def get_weather(ville: str) -> list[TextContent]:
    """Obtient la météo actuelle pour une ville."""
    meteo = await weather_api.get(ville)
    return [TextContent(type="text", text=f"{ville}: {meteo.temp} degrés")]

async def main():
    async with stdio_transport() as transport:
        await server.run(transport)
```

Ce serveur expose deux outils : interroger une base de données et obtenir la météo. Une fois lancé, n'importe quel host MCP (Claude Desktop, Cursor, un agent custom) peut les utiliser. Le LLM voit les descriptions des outils, comprend quand les utiliser, et les appelle automatiquement quand c'est pertinent.

## La sécurité : qui peut faire quoi ?

La sécurité est un aspect critique de MCP --- un agent qui a accès à votre base de données et à votre email est un agent qu'il faut protéger sérieusement.

### Modèle de sécurité en couches

MCP implémente plusieurs niveaux de protection :

- **Confirmation utilisateur** : avant d'appeler un outil potentiellement dangereux (supprimer un fichier, envoyer un email), l'host demande l'approbation explicite de l'utilisateur
- **Scoping** : chaque serveur n'expose que les données autorisées --- un serveur filesystem peut être restreint à un seul dossier, pas à tout le disque
- **Isolation des processus** : chaque serveur tourne dans son propre processus. Si l'un plante ou est compromis, les autres continuent de fonctionner normalement
- **Audit** : chaque interaction est loggée, permettant de tracer ce que l'agent a fait et pourquoi

<Callout type="warning" title="Attention aux injections via MCP">
Comme tout système d'agent, les données retournées par les serveurs MCP peuvent contenir du texte malveillant --- c'est le vecteur d'attaque appelé "prompt injection indirecte". Un attaquant pourrait insérer des instructions dans un document que le serveur renvoie à l'agent. Les hosts MCP doivent traiter toutes les réponses comme du contenu non-fiable et les valider avant de les présenter au LLM.
</Callout>

## L'adoption : qui utilise MCP aujourd'hui ?

### Les hosts compatibles

- **Claude Desktop** : support natif, c'est le host de référence
- **Cursor** : IDE IA avec support MCP pour les outils de développement
- **Zed** : éditeur de code nouvelle génération avec intégration MCP

### L'écosystème communautaire

Des centaines de serveurs MCP couvrent maintenant les principaux besoins :

- **Bases de données** : PostgreSQL, MySQL, MongoDB, Supabase, Redis
- **Cloud** : AWS, GCP, Azure
- **Outils de dev** : GitHub, GitLab, Jira, Linear
- **Communication** : Slack, Discord, Email
- **Productivité** : Notion, Google Workspace, Todoist

## Pourquoi MCP compte pour l'avenir de l'IA

MCP n'est pas juste un outil pratique --- c'est une tentative de standardiser l'écosystème des agents IA avant que la fragmentation ne devienne irréversible. Les avantages d'un standard adopté largement sont considérables :

- **Fin du code redondant** : plus besoin d'écrire 50 intégrations différentes. Si un serveur MCP existe pour votre outil, vous l'utilisez tel quel
- **Réutilisation massive** : un serveur MCP PostgreSQL créé par une équipe est utilisable par des milliers d'autres
- **Indépendance fournisseur** : le protocole est ouvert et décentralisé. Pas de vendor lock-in

<Callout type="paper" title="Un standard ouvert, pas un produit Anthropic">
MCP est un protocole ouvert avec une spécification publique. N'importe qui peut implémenter un client ou un serveur compatible, sans licence ni permission d'Anthropic. Cette ouverture est délibérée et cruciale : pour qu'un standard devienne universel, il ne peut pas être contrôlé par une seule entreprise.
</Callout>

**Le grand enjeu** : l'adoption par les autres fournisseurs de modèles. OpenAI, Google et Meta ont commencé à montrer de l'intérêt. Si MCP atteint une masse critique d'adoption, il pourrait devenir le standard de facto de l'IA agentique --- comme HTTP l'est devenu pour le web. La course est lancée.
