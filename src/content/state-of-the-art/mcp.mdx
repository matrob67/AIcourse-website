## Le cauchemar de l'intégration

Imaginez que vous ayez 10 applications différentes chez vous : votre frigo a ses propres boutons, votre machine à laver ses propres boutons, votre four ses propres boutons. Chaque appareil a une télécommande complètement différente. C'est le chaos actuel des agents IA.

Chaque application implémente ses propres connecteurs pour accéder aux bases de données, APIs, ou outils. Le résultat : une fragmentation énorme. Connecter un agent IA à une base de données PostgreSQL requiert du code custom. Connecter le même agent à une autre base de données (MySQL) requiert du code différent. Et si vous avez 50 outils différents ? Vous écrivez 50 intégrations différentes.

**MCP est le "standard USB" des agents IA**. Un seul protocole. N'importe quel agent peut se connecter à n'importe quel outil.

<Callout type="info" title="L'analogie USB">
Avant USB, chaque périphérique avait son connecteur unique : souris série, clavier PS/2, imprimante parallèle, scanner SCSI. USB a standardisé tout ça. Maintenant, n'importe quel périphérique USB fonctionne avec n'importe quel ordinateur USB. MCP fait la même chose pour les agents IA et les sources de données.
</Callout>

## Architecture : comment ça marche

### Les trois pièces du puzzle

1. **MCP Host** : votre application IA (Claude Desktop, un IDE, un agent custom)
2. **MCP Client** : le pont côté host (gère la communication)
3. **MCP Server** : le service qui expose des ressources et outils

### Le protocole sous-jacent

MCP utilise **JSON-RPC 2.0** (un standard internet pour appeler des fonctions à distance) sur différents transports :

- Stdio (communication via entrée/sortie standard)
- HTTP/SSE (communication web)

Le serveur expose trois types de choses :

- **Resources** : des données que le LLM peut lire (fichiers, données de base de données, contenu d'APIs)
- **Tools** : des fonctions que le LLM peut appeler (créer quelque chose, envoyer un message, exécuter une action)
- **Prompts** : des templates de prompts pré-configurés et optimisés

### Flux de communication visuel

```
Vous → [Claude] → "J'ai besoin de données de la base de données"
        ↓
    [Client MCP] → JSON-RPC → [Serveur MCP: PostgreSQL]
        ↓                            ↓
    [Client MCP] ← résultats ← [Exécute la requête SQL]
        ↓
    [Claude] → "Voici l'analyse des données..."
```

## Types de serveurs MCP (l'écosystème)

### Serveurs de données

- **Filesystem** : lire vos fichiers locaux
- **PostgreSQL/SQLite** : accéder à vos bases de données
- **Google Drive** : lire vos documents cloud
- **GitHub** : accéder aux repositories et issues

### Serveurs d'outils

- **Brave Search** : faire des recherches web
- **Puppeteer** : naviguer sur le web de manière automatisée
- **Memory** : mémoire persistante pour l'agent
- **Sequential Thinking** : outils de raisonnement structuré

### Serveurs custom

N'importe quel développeur peut créer un serveur MCP pour ses propres données et outils. C'est accessible.

<Callout type="tip" title="Créer votre propre serveur MCP">
Les SDKs officiels (Python, TypeScript/JavaScript) rendent c'est très facile. Un serveur basique s'écrit en 30 lignes de code. Il y a un registre communautaire de serveurs prêts à copier-coller.
</Callout>



## Un exemple concret : serveur MCP minimaliste

### Serveur Python

```python
from mcp.server import Server, stdio_transport
from mcp.types import Tool, TextContent

server = Server("mon-serveur")

@server.tool("interroger_database")
async def query_db(requete: str) -> list[TextContent]:
    """Exécute une requête SQL sur la base de données."""
    resultats = await db.execute(requete)
    return [TextContent(type="text", text=str(resultats))]

@server.tool("obtenir_meteo")
async def get_weather(ville: str) -> list[TextContent]:
    """Obtient la météo actuelle pour une ville."""
    meteo = await weather_api.get(ville)
    return [TextContent(type="text", text=f"{ville}: {meteo.temp} degrés")]

async def main():
    async with stdio_transport() as transport:
        await server.run(transport)
```

Voilà. Vous exposez deux outils : interroger une base de données et obtenir la météo. Claude peut maintenant les utiliser.

## Sécurité : qui peut faire quoi

### Modèle de sécurité en couches

MCP implémente plusieurs niveaux de protection :

- **Confirmation utilisateur** : avant d'appeler un outil dangereux, demander l'approbation
- **Scoping** : les serveurs n'exposent que les données autorisées (pas d'accès au disque entier)
- **Isolation** : chaque serveur tourne dans son propre processus (si l'un plante, les autres survivent)
- **Audit** : chaque interaction est loggée

<Callout type="warning" title="Attention aux injections malveillantes">
Comme tout système d'agent, les données retournées par les serveurs MCP peuvent contenir du texte malveillant (une attaque appelée "prompt injection"). Les hosts MCP doivent traiter les réponses comme du contenu non-fiable et les valider avant de les donner au LLM.
</Callout>

## L'adoption actuelle

### Qui supporte MCP

- **Claude Desktop** : support natif
- **Cursor** : support pour les outils de développement
- **Zed** : éditeur de code avec support MCP
- **Continue** : extension VS Code avec MCP
- **Cline** : agent de codage avec MCP

### L'écosystème communautaire

Des centaines de serveurs MCP existent maintenant :

- Bases de données (PostgreSQL, MySQL, MongoDB, Supabase)
- Cloud (AWS, GCP, Azure)
- Outils de dev (GitHub, GitLab, Jira)
- Communication (Slack, Discord, Email)
- Productivité (Notion, Google Workspace, Linear)

## Pourquoi c'est important

MCP représente une tentative sérieuse de standardiser l'écosystème des agents IA. En créant un protocole ouvert et interopérable :

- **Moins de code redondant** : vous n'écrivez pas 50 intégrations différentes
- **Réutilisation** : si quelqu'un a créé un serveur MCP pour PostgreSQL, vous l'utilisez directement
- **Pas de dépendance fournisseur** : l'écosystème est décentralisé et open

<Callout type="paper" title="Un standard ouvert">
MCP est un protocole ouvert avec une spécification publique. N'importe qui peut implémenter un client ou un serveur compatible, sans demander la permission à Anthropic. Cette ouverture est cruciale pour éviter qu'une seule entreprise contrôle tout l'écosystème.
</Callout>

**Le grand enjeu** : l'adoption par d'autres fournisseurs de modèles (Google, OpenAI, Meta) et par les développeurs. Si MCP atteint une "masse critique" d'adoption, il pourrait devenir le standard de facto, comme HTTP l'est devenu pour le web. Si quelques autres fournisseurs le soutiennent, c'est une vraie révolution. Si c'est juste Anthropic, ça reste un outil utile mais limité.
