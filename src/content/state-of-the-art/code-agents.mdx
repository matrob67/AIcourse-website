## Pourquoi le code est le terrain idéal pour les agents IA

Si les agents IA brillent quelque part, c'est dans le code. Et la raison est fondamentale : le code offre un **feedback objectif et immédiat**.

- Ca compile ou ca compile pas
- Les tests passent ou ils passent pas
- Le linter se plaint ou il se plaint pas

Pas d'ambiguïté, pas de subjectivité. Contrairement à la rédaction (est-ce que ce texte est "bon" ?) ou l'analyse business (est-ce que cette conclusion est "correcte" ?), le code donne une réponse binaire. L'agent peut donc **vérifier son propre travail** automatiquement, itérer, corriger, et re-vérifier. C'est cette boucle de feedback serrée qui rend les agents de code si efficaces.

<Callout type="info" title="Le feedback objectif, c'est la clé">
Un agent qui rédige un email ne peut pas vérifier si l'email est "bon". Un agent qui écrit du code peut lancer les tests et savoir immédiatement. Cette différence fondamentale explique pourquoi les agents de code sont en avance sur tous les autres types d'agents.
</Callout>



## Les systèmes pionniers

### Devin (Cognition Labs, 2024)

Présenté comme le "premier ingénieur IA autonome", Devin a marqué les esprits en montrant un agent capable de résoudre des issues GitHub de bout en bout. Il dispose d'une interface complète (terminal, éditeur de code, navigateur web), peut planifier sur le long terme et décomposer des problèmes complexes en sous-tâches. Son score initial sur SWE-bench (13.86%) a établi un premier record, vite dépassé depuis.

### Claude Code (Anthropic)

Un CLI qui transforme ton terminal en environnement de développement assisté par IA. Claude Code a un accès direct au système de fichiers et au terminal, comprend des codebases entiers grâce à son large contexte, et fonctionne en boucle ReAct : il lit des fichiers, écrit du code, exécute des commandes, observe les résultats, et itère. Il peut aussi créer des commits, lancer des tests, et gérer des branches Git.

### Cursor

Un fork de VS Code qui intègre l'IA au coeur de l'IDE :
- **Tab completion** : suggestions contextuelles qui comprennent ton code
- **Cmd+K** : édite du code par instruction en langage naturel ("rends cette fonction async")
- **Chat** : conversation avec l'ensemble de ton codebase (indexé localement)
- **Composer** : modifications coordonnées sur plusieurs fichiers simultanément
- Support de plusieurs modèles (GPT-4, Claude, Mistral)

## L'architecture typique d'un agent de code

### Les outils à disposition

Un bon agent de code dispose d'une boite à outils complète :

1. **File operations** : lire, écrire, éditer des fichiers avec précision
2. **Terminal** : exécuter des commandes shell, installer des dépendances, compiler
3. **Search** : grep, find, recherche sémantique pour naviguer dans le codebase
4. **Browser** : consulter la documentation, Stack Overflow, les API docs
5. **Git** : créer des branches, faire des commits, ouvrir des PRs

### La boucle de développement

L'agent suit un cycle itératif qui ressemble beaucoup à celui d'un développeur humain :

1. **Comprendre** : lire l'issue ou la demande, explorer le code existant pour comprendre l'architecture
2. **Planifier** : identifier les fichiers à modifier, anticiper les impacts
3. **Implémenter** : écrire ou modifier le code
4. **Tester** : exécuter les tests, compiler, vérifier
5. **Itérer** : si des erreurs apparaissent, les analyser et corriger
6. **Finaliser** : commit, PR, mise à jour de la documentation si nécessaire

<Callout type="tip" title="Le contexte est roi">
La qualité du code produit dépend principalement de la **compréhension du contexte** : l'architecture du projet, les conventions de code, les dépendances, les patterns utilisés. Un agent qui investit du temps à explorer le codebase avant d'écrire du code produit de bien meilleurs résultats qu'un agent qui fonce.
</Callout>

## Stratégies de génération de code

**Génération complète** : l'agent crée un fichier entier from scratch. Adapté aux nouvelles fonctionnalités ou aux nouveaux modules.

**Edition ciblée** : l'agent modifie des parties spécifiques d'un fichier existant. Idéal pour les corrections de bugs et les petites améliorations.

**Diff-based** : l'agent génère un patch (diff) au lieu du fichier complet. Plus économe en tokens, et plus facile à reviewer pour l'humain.

**Code-first agents** (comme Smolagents de Hugging Face) : au lieu de générer du JSON décrivant des appels d'outils, le LLM écrit directement du code Python qui est exécuté. C'est plus expressif et naturel : un `for` loop avec un `if` condition est beaucoup plus lisible qu'une chaîne de tool calls JSON imbriqués.

## Benchmarks : où en sont-ils ?

### SWE-bench

Le benchmark de référence : résoudre des issues GitHub réelles dans des projets open-source populaires (Django, scikit-learn, sympy...). Ca teste la capacité de l'agent à comprendre un bug décrit en langage naturel, trouver les fichiers concernés, et implémenter un fix correct.

| Agent | Score (%) |
|---|---|
| GPT-4 direct (sans outils) | ~1.7 |
| Claude 3.5 Sonnet + tools | ~49 |
| OpenAI Codex Agent | ~69 |
| Claude Code | ~72 |

La progression est spectaculaire : de 1.7% avec un LLM brut à 72% avec un agent outillé. L'architecture agent (boucle, outils, itération) fait toute la différence.

### HumanEval et MBPP

Ces benchmarks testent la génération de fonctions individuelles. Moins réalistes que SWE-bench, mais utiles pour comparer les capacités de base de génération de code des différents modèles.

<Callout type="paper" title="L'écart se réduit vite">
Les meilleurs agents résolvent maintenant des bugs réels dans de vrais projets open-source. Mais les problèmes complexes (refactoring architectural, choix de design patterns, migrations de frameworks) restent largement hors de portée.
</Callout>

## Les défis qui restent

**Compréhension à grande échelle** : les codebases industriels font des centaines de milliers de lignes. L'agent doit naviguer intelligemment sans tout lire, en utilisant la recherche, les imports et les dépendances comme guides.

**Tests et validation** : un agent fiable doit non seulement écrire du code, mais aussi écrire et exécuter des tests. Sans tests, le risque de régressions silencieuses est majeur. Le code "a l'air correct" mais casse quelque chose ailleurs.

**Sécurité** : donner à un agent l'accès au terminal et au système de fichiers crée des risques réels. Sandboxing strict, permissions limitées, et revue humaine sont indispensables.

<Callout type="warning" title="La revue humaine reste indispensable">
Même les meilleurs agents de code produisent régulièrement du code incorrect, suboptimal, ou qui ne respecte pas les conventions du projet. La revue humaine reste essentielle, surtout pour du code qui part en production. Les agents accélèrent le développement, mais ils ne remplacent pas le jugement d'un développeur expérimenté.
</Callout>

## L'avenir des agents de code

Les agents de code sont probablement l'application IA la plus mature et la plus impactante aujourd'hui. La direction est claire : gérer des tâches toujours plus complexes, du refactoring massif aux migrations de systèmes entiers, en passant par la collaboration avec des équipes humaines sur des projets de grande envergure.

Aujourd'hui, les agents de code ont les capacités d'un développeur junior consciencieux. Demain, ils seront probablement des assistants de niveau senior. Mais dans tous les cas, la revue humaine et le jugement d'ingénierie resteront au coeur du processus.
