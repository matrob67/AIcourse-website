## Le problème de gaspillage mémoire

Quand tu déploies un LLM en production, chaque requête utilisateur a besoin d'espace mémoire pour stocker son **KV cache** (les clés et valeurs d'attention calculées). Le problème : on ne sait pas à l'avance combien de tokens la réponse fera. 50 ? 500 ? 2000 ?

Les systèmes naïfs pré-allouent un bloc de mémoire **contigu** pour la longueur maximale possible de chaque requête. C'est comme réserver une table de 20 couverts au restaurant pour chaque client, même si la plupart viennent seuls. Résultat : **60 à 80% de la mémoire GPU est gaspillée** en espace vide réservé mais jamais utilisé.



<Callout type="info" title="Une idée vieille de 60 ans">
Ce problème de fragmentation mémoire, les systèmes d'exploitation l'ont résolu dans les années 1960 avec la **mémoire virtuelle paginée**. Au lieu d'allouer des blocs contigus de taille variable, on découpe la mémoire en petites **pages** de taille fixe qui peuvent être placées n'importe ou. PagedAttention applique exactement cette même idée au KV cache des LLM.
</Callout>

## PagedAttention : le principe

### Le modèle de paging

Le KV cache est divisé en **blocs (pages)** de taille fixe, typiquement 16 tokens chacun. Chaque requête n'occupe que les blocs dont elle a effectivement besoin, et ces blocs peuvent être dispersés n'importe ou dans la mémoire GPU :

- **Requête A** (120 tokens) : Bloc 5, Bloc 12, Bloc 3
- **Requête B** (30 tokens) : Bloc 7, Bloc 1
- **Requête C** (200 tokens) : Bloc 9, Bloc 4, Bloc 15, Bloc 2

Chaque requête possède une **table de pages** qui fait le lien entre les positions logiques (token 0-15 dans le bloc 0, tokens 16-31 dans le bloc 1...) et les blocs physiques en mémoire. Les blocs n'ont pas besoin d'être contigus : la table de pages s'occupe de la traduction.

### Allocation à la demande

- Un nouveau bloc n'est alloué que quand le bloc précédent est **plein** (les 16 slots occupés)
- Quand une requête se termine, ses blocs sont **immédiatement libérés** et réutilisables pour d'autres requêtes
- Le gaspillage se limite au dernier bloc partiellement rempli : moins de 4% en moyenne

### Le kernel d'attention modifié

L'opération d'attention doit être adaptée pour lire les clés et valeurs depuis des blocs potentiellement dispersés en mémoire. Un **block table** stocké en GPU indique ou trouver chaque bloc logique. Grâce à la taille fixe des blocs (16 tokens), les lectures mémoire au sein de chaque bloc restent efficaces et coalescées.

## Les résultats concrets

PagedAttention réduit le gaspillage de KV cache de **60-80% a moins de 4%**. En pratique, ça signifie :

- **2 à 4 fois plus de requêtes simultanées** avec la même quantité de VRAM
- **Réduction proportionnelle des coûts d'inférence** (même performance, moins de GPU)
- **Utilisation mémoire stable et prévisible** même avec des longueurs de réponse très variées

## vLLM : le framework qui a tout lancé

vLLM est le framework open-source créé par les auteurs de PagedAttention. Il est devenu le **standard de facto** pour le serving de LLM en production grâce à son efficacité et sa simplicité d'utilisation.

### Fonctionnalités principales

- **PagedAttention** pour la gestion mémoire optimale
- **Continuous batching** pour le scheduling dynamique des requêtes
- **Speculative decoding** intégré pour accélérer la génération
- **Tensor parallelism** et **pipeline parallelism** pour le multi-GPU
- API compatible OpenAI (remplacement direct)

### Démarrage en 3 lignes

```python
from vllm import LLM, SamplingParams

llm = LLM(model="mistralai/Mistral-7B-v0.1")
outputs = llm.generate(["Explique la gravité"],
                       SamplingParams(max_tokens=256))
```

### Mode serveur

```bash
vllm serve mistralai/Mistral-7B-v0.1 \
    --host 0.0.0.0 --port 8000 \
    --tensor-parallel-size 2
```

Ça lance un serveur avec une API REST compatible OpenAI. Tu peux remplacer tes appels à l'API OpenAI par des appels à ce serveur local sans changer ton code client.

## Optimisations avancées

### Prefix Caching

Quand plusieurs requêtes partagent le **même préfixe** (un system prompt identique, des exemples few-shot communs, un contexte RAG partagé), les blocs de KV cache correspondant à ce préfixe sont **partagés** entre toutes les requêtes :

- **System prompt** (calculé une seule fois, partagé)
  - Requête 1 : blocs spécifiques
  - Requête 2 : blocs spécifiques
  - Requête 3 : blocs spécifiques

Au lieu de recalculer et stocker le KV cache du system prompt pour chaque requête, on le calcule une fois et on le réutilise. Les économies sont massives sur les systèmes RAG ou le contexte partagé est souvent bien plus long que la réponse.

### Copy-on-Write

Pour le beam search ou le sampling multiple (générer N réponses différentes pour le même prompt), les séquences partagent initialement les mêmes blocs de KV cache. Un bloc n'est dupliqué que quand une séquence diverge et a besoin de modifier ce bloc. C'est le même mécanisme que les systèmes d'exploitation utilisent pour la gestion de la mémoire des processus.

<Callout type="tip" title="RAG + Prefix Caching = combo optimal">
Si tu utilises un long contexte RAG identique pour 100 requêtes, le prefix caching évite de recalculer et stocker son KV cache 100 fois. Pour un contexte RAG de 4000 tokens avec un modèle 7B, ça économise environ 200 Go de mémoire GPU et des milliers de TFLOP de calcul.
</Callout>

## Benchmarks de performance

### Résultats réels (A100 80 Go, Mistral-7B)

| Métrique | Sans PagedAttention | Avec PagedAttention |
|---|---|---|
| Max requêtes concurrentes | ~40 | ~120 |
| Throughput (tokens/s) | 1200 | 3500 |
| VRAM gaspillée | ~50% | ~3% |
| Stabilité de la latence | Variable | Stable |

PagedAttention multiplie le throughput par environ 3x, principalement parce qu'on peut servir 3 fois plus de requêtes simultanément avec la même mémoire.

### Comparaison des frameworks de serving

| Framework | Technologie | Throughput relatif |
|---|---|---|
| HF Transformers | Baseline | 1x |
| vLLM | PagedAttention | 8-15x |
| TGI | FlashAttention | 6-12x |
| SGLang | RadixAttention | 8-15x |

vLLM et SGLang offrent des performances comparables. SGLang utilise une version améliorée du concept de PagedAttention.

## Évolution : RadixAttention (SGLang)

**RadixAttention** généralise PagedAttention en organisant les préfixes dans un **arbre Radix** (trie). Les avantages :

- Matching de préfixe plus efficace (recherche en O(longueur du préfixe) au lieu de O(nombre de requêtes))
- Meilleure gestion des workloads hétérogènes avec des préfixes partiellement partagés
- Partage plus fin que le simple prefix caching : deux requêtes qui partagent 80% de leur contexte bénéficient du partage, même si les 20% restants diffèrent

## Les limites à connaître

Même avec PagedAttention, la VRAM GPU reste finie. Au-delà d'une certaine charge, le système doit faire du **swapping** : déplacer des KV caches en RAM CPU pour faire de la place. Ce swapping dégrade les performances d'environ 50% pour les requêtes affectées. Il faut monitorer la fragmentation des blocs et dimensionner correctement le système pour éviter cette situation.

<Callout type="warning" title="PagedAttention ne fait pas de miracles">
PagedAttention multiplie l'efficacité mémoire par 3-4x, mais c'est une ressource finie. Au-delà de la capacité maximale, le système swape et devient très lent. Dimensionne ton infrastructure en fonction de la charge attendue, pas de la charge théorique maximale.
</Callout>
