## Le modèle qui se juge lui-même

L'intuition de départ est simple : si un modèle peut générer une réponse incorrecte, peut-il aussi la **reconnaître** comme incorrecte quand on lui demande de vérifier ? La réponse est souvent oui, et c'est moins paradoxal qu'il n'y paraît.

Générer une bonne réponse et vérifier une réponse sont deux tâches de **difficulté différente**. Penser à une preuve mathématique est difficile. Vérifier qu'une preuve est correcte, c'est généralement plus facile. Écrire un programme sans bug est dur. Lire un programme et repérer le bug, c'est souvent plus accessible.

La **self-vérification** exploite cette asymétrie : le modèle devient son propre contrôleur qualité.

<Callout type="info" title="L{"'"}asymétrie fondamentale">
Générer une bonne réponse = difficile. Vérifier si une réponse est bonne = plus facile.
Un modèle qui échoue à générer correctement du premier coup peut quand même détecter l'erreur quand on lui montre sa propre réponse. C'est comme relire sa copie d'examen : on voit des fautes qu'on n'aurait pas évitées en écrivant.
</Callout>

## Techniques de self-vérification

### 1. Self-Refine : amélioration itérative

Le principe est un dialogue du modèle avec lui-même :

1. Générez une réponse initiale
2. Demandez : "Critique cette réponse. Quelles sont les erreurs potentielles ?"
3. Demandez : "Améliore la réponse en tenant compte de cette critique"
4. Répétez les étapes 2-3 jusqu'à ce que la critique ne trouve plus de problème

C'est comme un auteur qui relit et révise son manuscrit en plusieurs passes. Chaque passe attrape des erreurs que la précédente avait manquées.

### 2. Self-Debugging : le cas spécial du code

Pour le code, la self-vérification a un avantage énorme : on peut **exécuter** le code et obtenir un feedback objectif.

1. Le modèle génère le code
2. On l'exécute contre les tests
3. Si erreur : on montre le message d'erreur au modèle
4. Le modèle analyse l'erreur et propose une correction
5. On re-teste

Le feedback n'est pas subjectif ici : le test passe ou ne passe pas. C'est ce qui rend le self-debugging si efficace par rapport aux autres formes de self-vérification.

### 3. Vérification par inversion

Une technique élégante : au lieu de vérifier directement si la réponse est correcte, on résout le problème **à l'envers** :

> **Problème :** $2x + 3 = 11$, trouver $x$
> **Réponse proposée :** $x = 4$
> **Vérification :** $2(4) + 3 = 8 + 3 = 11$ &check;

Au lieu de re-dériver la solution, on substitue et on vérifie que ça colle. Ce pattern est généralisable bien au-delà des maths.

### 4. Constitutional AI (Self-Critique)

Le modèle évalue sa propre réponse contre des **principes explicites** :

- "Est-ce que cette réponse est utile ?"
- "Est-ce que cette réponse est honnête et factuelle ?"
- "Est-ce que cette réponse est sûre et sans danger ?"

Le modèle se juge selon une "constitution" de principes. C'est la base de l'alignement chez Anthropic.

<Callout type="tip" title="Le pattern d{"'"}inversion : à retenir">
Pour les maths : substituez la réponse dans l'équation originale.
Pour les traductions : retraduisez dans la langue source et comparez.
Pour les résumés : vérifiez que chaque point clé du résumé se retrouve dans le texte original.

Ce pattern d'inversion est généralisable et remarquablement efficace. Le coût est minime (un appel supplémentaire) et la fiabilité augmente significativement.
</Callout>

## Réflexion : apprendre de ses erreurs avec mémoire

Le framework Réflexion (Shinn et al., 2023) pousse la self-vérification plus loin en ajoutant une **mémoire d'échecs** :

1. Le modèle tente de résoudre la tâche
2. En cas d'échec : il génère une réflexion ("J'ai échoué parce que j'ai confondu X et Y")
3. Cette réflexion est stockée en mémoire
4. À la tentative suivante : les réflexions précédentes sont incluses dans le prompt
5. Le modèle évite les erreurs déjà commises

Les résultats sont spectaculaires :
- HumanEval (code) : 67% --> 91%
- AlfWorld (agent) : 63% --> 97%

La mémoire d'erreur transforme chaque échec en apprentissage. Au lieu de refaire les mêmes erreurs, le modèle accumule de l'expérience, comme un développeur qui tient un journal de bugs.

## Les limites réelles : ne pas surestimer

### Le modèle peut-il vraiment se corriger ?

La réponse honnête : **parfois oui, souvent non**.

Des études rigoureuses montrent que la self-correction sans feedback externe est **limitée** :

| Type d'erreur | Détection par self-vérification |
|---|---|
| Erreurs factuelles évidentes | Souvent détectées |
| Erreurs subtiles de logique | Fréquemment manquées |
| Hallucinations sur des sujets inconnus | Non détectées |
| Biais systématiques | Renforcés au lieu d'être corrigés |

<Callout type="paper" title="Sans feedback externe, l{"'"}amélioration est limitée">
Huang et al. (2023) ont montré que sans feedback objectif externe (exécution de code, calcul symbolique, vérification factuelle), les LLM **ne s'améliorent pas** significativement par self-correction. Le modèle tend à maintenir sa réponse initiale, ou pire, à la dégrader en deuxième intention. La self-vérification fonctionne surtout quand il y a un ancrage objectif.
</Callout>

### Le biais de confirmation

Comme un humain qui défend son opinion plutôt que de chercher ses propres erreurs, le LLM a un **biais de confirmation**. "Ma réponse est correcte" est plus probable dans l'espace des tokens que "je me suis trompé". Le modèle tend à justifier sa réponse initiale plutôt qu'à la remettre en question genuinement.

### Le coût cumulatif

Trois itérations de critique-amélioration consomment **4 à 5 fois plus de tokens** que la réponse initiale. C'est un investissement significatif qui ne se justifie que si le gain de qualité est réel.

## Bonnes pratiques pour une self-vérification efficace

### 1. Combiner avec du feedback externe

La self-vérification fonctionne **beaucoup mieux** avec un ancrage objectif :

- **Code** : exécution de tests unitaires
- **Maths** : vérification par substitution ou calcul symbolique
- **Faits** : vérification par recherche web
- **Format** : validation JSON/XML automatique

```python
response = model.generate(prompt)
critique = model.generate(f"Critique : {response}")
if execution_test(response) == PASS:
    return response  # Pas besoin d'améliorer
else:
    improved = model.generate(f"Corrige basé sur : {critique}")
    return improved
```

### 2. Limiter les itérations

2 à 3 itérations maximum. Au-delà, le modèle tourne en rond, "corrige" des choses qui n'avaient pas besoin de l'être, ou dégrade la réponse originale. Plus n'est pas toujours mieux.

### 3. Utiliser un "juge" différent

Pour la critique, utilisez un prompt radicalement différent, ou mieux, un modèle différent. Ça réduit le biais de confirmation : le "juge" n'a pas les mêmes angles morts que le "générateur".

<Callout type="warning" title="Un multiplicateur, pas une solution miracle">
La self-vérification corrige les erreurs que le modèle est capable de "reconnaître" : erreurs évidentes de calcul, incohérences logiques flagrantes. Mais elle ne corrige pas les hallucinations profondes ni les biais dont le modèle n'est pas conscient.

C'est un multiplicateur de fiabilité sur les erreurs "bêtes", pas une garantie de correction universelle.
</Callout>

## L'évolution : la self-vérification internalisée

Les modèles de raisonnement modernes (o1, R1) ont internalisé la self-vérification dans leur chaîne de pensée. Au lieu de demander explicitement au modèle "vérifie ta réponse", le modèle se vérifie **spontanément** en réfléchissant. Il revient sur ses étapes, teste des alternatives, détecte ses propres incohérences, le tout pendant le processus de génération.

L'auto-correction n'est plus juste une astuce de prompting. C'est une capacité qui peut être entraînée et intégrée directement dans le modèle, un signe de la maturation du domaine.
