## Le défi : servir un LLM rapidement

Faire tourner un modèle en local, c'est facile : `model.generate()` et c'est parti. Mais en **production**, c'est une autre histoire. 100 utilisateurs envoient des requêtes en même temps, chacun veut une réponse en moins de 2 secondes, et votre GPU à une mémoire limitée.

Les frameworks de serving résolvent ce problème en gérant intelligemment la **mémoire GPU**, la **file d'attente** des requêtes, le **batching dynamique** (regrouper plusieurs requêtes pour les traiter en parallèle), et l'**accélération** via des kernels optimisés. Quatre solutions dominent le marché.

<Callout type="info" title="Le choix dépend de votre contexte">
vLLM pour la flexibilité et la communauté. TGI si vous êtes dans l'écosystème Hugging Face. Triton si vous servez plusieurs modèles hétérogènes. SGLang si vous construisez des agents complexes.
</Callout>

## vLLM : le standard de facto

vLLM, né à UC Berkeley, est devenu le framework de serving le plus populaire grâce à une innovation clé : **PagedAttention**.

Le problème qu'il résout : pendant l'inférence, chaque requête maintient un **KV cache** (les clés et valeurs d'attention calculées pour les tokens précédents). Normalement, ce cache est un bloc de mémoire continu. Si un utilisateur arrive avec un contexte plus long que prévu, il faut tout réallouer — c'est du gaspillage massif, comme réserver une salle de 100 places pour une réunion de 10 personnes.

PagedAttention découpe le KV cache en **petites pages** de mémoire (comme un système d'exploitation gère la RAM). Chaque requête prend exactement la mémoire dont elle a besoin, pas plus. Résultat : gestion mémoire **2 à 4x plus efficace**, ce qui permet de servir plus de requêtes simultanées.

**Bonus** : le prefix caching — si 10 utilisateurs posent des questions sur le même document système, le KV cache de ce préfixe commun est calculé une seule fois et partagé.

```bash
vllm serve mistralai/Mistral-7B-v0.1 \
    --tensor-parallel-size 2 \
    --max-model-len 32768 \
    --enable-prefix-caching
```

## TGI : la simplicité Hugging Face

TGI (Text Génération Inference), c'est le serveur officiel de Hugging Face, écrit en Rust pour la performance et la fiabilité.

Son atout principal : la **simplicité de déploiement**. Vous avez un modèle Hugging Face ? Une seule commande Docker et vous avez une API de production. Zéro configuration, zéro prise de tête.

```bash
docker run --gpus all -p 8080:80 \
    ghcr.io/huggingface/text-generation-inference:latest \
    --model-id mistralai/Mistral-7B-v0.1 \
    --num-shard 2
```

TGI gère automatiquement la quantification, le sharding multi-GPU, le continuous batching, et expose une API compatible OpenAI. C'est le chemin le plus court entre un modèle sur le Hub et une API en production.

<Callout type="tip" title="Pour un premier déploiement">
TGI = le chemin le plus rapide du notebook Jupyter à l'API de production. Il ne couvre pas 100% des cas avancés, mais pour 80% des déploiements, TGI suffit amplement. Commencez par là.
</Callout>

## NVIDIA Triton : la plateforme enterprise

Triton n'est pas spécifiquement conçu pour les LLM — c'est la plateforme de serving **généraliste** de NVIDIA. Vous pouvez servir sur le même serveur : un LLM (via TensorRT-LLM), des modèles d'embeddings, des rerankers, des classifieurs d'images, des modèles de détection. Tout cohabite.

**Forces** : A/B testing natif, canary deployments, métriques Prometheus, versioning des modèles, support multi-backend (TensorFlow, PyTorch, ONNX, TensorRT).
**Faiblesse** : configuration complexe (XML/protobuf), courbe d'apprentissage raide, pas optimisé spécifiquement pour l'expérience développeur LLM.

Utilisez Triton si vous avez un **écosystème complet** de modèles ML à servir et une équipe MLOps dédiée. Sinon, c'est overkill.

## SGLang : le challenger pour les agents

SGLang est le nouveau venu, créé à Berkeley également, et optimisé pour les **workflows structurés et agentiques**.

Son innovation : **RadixAttention**, une gestion du KV cache basée sur un arbre Radix. Concrètement, quand un agent fait 10 appels successifs au LLM avec des préfixes qui se chevauchent, RadixAttention réutilise intelligemment les calculs déjà faits. Pour les workflows multi-tours, c'est souvent plus efficace que le PagedAttention de vLLM.

Autre atout : la **génération structurée native**. Si vous voulez que le LLM produise du JSON valide ou respecte un regex, SGLang le fait sans overhead significatif — le guided decoding est intégré au cœur du moteur.

```python
import sglang as sgl

@sgl.function
def multi_turn(s, question):
    s += sgl.user(question)
    s += sgl.assistant(sgl.gen("answer", max_tokens=256))
```

SGLang est le choix naturel si vous construisez des **agents multi-appels** avec des réponses structurées (JSON, formulaires, listes).

## Le tableau comparatif

| Aspect | vLLM | TGI | Triton | SGLang |
|---|---|---|---|---|
| Facilité d'install | pip install | docker run | Complexe | pip install |
| Performance brute | Très bien | Bien | Excellente | Très bien |
| Prefix caching | Bon | Partiel | Oui | Meilleur |
| Génération structurée | Oui | Oui | Non | Natif |
| Multi-modèles | Non | Non | Oui | Non |
| API OpenAI | Oui | Presque | Wrapper | Oui |

<Callout type="warning" ’>
Les benchmarks d'il y a 6 mois sont déjà obsolètes. Ces frameworks évoluent chaque mois avec des gains de performance significatifs. Testez toujours sur **vos données réelles** avec les versions les plus récentes avant de figer un choix.
</Callout>

**Conseil pratique** : commencez par TGI (une commande Docker = zéro setup) ou vLLM (`pip install` = 30 secondes). Ne changez que si vous identifiez un besoin spécifique que votre premier choix ne couvre pas. Le pire piège, c'est de passer des semaines à comparer des frameworks au lieu de déployer.
