## Le dilemme du chunking : trop petit ou trop gros ?

Quand tu construis un système RAG, tu dois découper tes documents en morceaux (chunks) pour les indexer et les retrouver. Ce choix de découpage est **la décision la plus impactante** de ton pipeline RAG, et paradoxalement la plus souvent bâclée.

**Chunks trop petits** (100 tokens) : chaque morceau manque de contexte. Une phrase isolée comme "Le résultat est de 42%" ne veut rien dire sans savoir de quel résultat on parle. Le retriever ramène des fragments incompréhensibles.

**Chunks trop gros** (2000 tokens) : chaque morceau contient tellement de sujets différents que l'embedding dilue l'information pertinente dans du bruit. Le retriever ramène des pavés dont seules 2 phrases sont utiles.

C'est le Goldilocks problem : il faut trouver la taille juste.

<Callout type="info" title="Le compromis incontournable">
Le chunk parfait contiendrait exactement une idée complète, sans bruit, avec assez de contexte pour être autonome. En réalité, c'est impossible à atteindre systématiquement. Des expérimentations empiriques convergent vers un sweet spot de 256-512 tokens pour environ 80% des cas d'usage.
</Callout>

## Stratégies de découpage

### 1. Taille fixe (naive)

Découper tous les N tokens avec un overlap (chevauchement) entre chunks consécutifs.

```python
from langchain.text_splitter import CharacterTextSplitter

splitter = CharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=100,
)
chunks = splitter.split_text(doc)
```

**Force** : simple, prévisible, rapide. Tu sais exactement combien de chunks tu auras.
**Faiblesse** : coupe arbitrairement au milieu des phrases et paragraphes. Un tableau peut se retrouver coupé en deux morceaux incompréhensibles.

### 2. Récursif (le défaut recommandé)

Essaie d'abord de couper sur les gros séparateurs (paragraphes `\n\n`), puis descend progressivement vers des séparateurs plus fins (sauts de ligne `\n`, phrases `.`, mots ` `). Le résultat : des chunks qui respectent la structure naturelle du texte autant que possible.

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=512,
    chunk_overlap=50,
)
```

**Force** : respecte la structure du texte, produit peu de "chunks orphelins" (fragments incomplets).
**Faiblesse** : taille variable selon la structure du document. Certains chunks seront proches de la limite, d'autres bien plus courts.

<Callout type="tip" title="Commence par RecursiveCharacterTextSplitter">
C'est le défaut de LangChain pour une bonne raison : il marche bien dans 90% des cas sans configuration. Si les résultats ne sont pas satisfaisants, affine ensuite avec une stratégie plus spécifique.
</Callout>

### 3. Sémantique

Calcule l'embedding de chaque phrase, puis détecte les ruptures thématiques en mesurant la similarité cosinus entre phrases consécutives. Quand la similarité chute brusquement, ça indique un changement de sujet : c'est la qu'on coupe.

**Force** : produit des chunks thématiquement cohérents. Chaque chunk traite d'un seul sujet.
**Faiblesse** : lent (il faut calculer l'embedding de chaque phrase individuellement), taille de chunks imprévisible, et la qualité dépend du modèle d'embeddings utilisé pour la détection.

Recommandé pour : documents scientifiques, rapports structurés. Pour du texte courant ou des FAQ, c'est généralement overkill.

### 4. Par structure du document

Exploiter la structure déjà présente dans le document : headers Markdown, sections HTML, chapitres PDF.

```python
# Exemple simplifié : couper sur les H2 Markdown
chunks = text.split("## ")
```

**Force** : respecte l'intention de l'auteur. Si un document a des sections, c'est que l'auteur a jugé que chaque section forme une unité cohérente.
**Faiblesse** : nécessite un parsing spécifique par format (Markdown, PDF, HTML, code source). Et tous les documents ne sont pas bien structurés.

Idéal pour : documentation technique, manuels, code source. Inadapté pour des textes sans structure (emails, transcriptions).

### 5. Parent-Child (avancé)

Stocker **deux niveaux** de chunks avec des rôles différents :

- **Enfants** (petits, ~200 tokens) : indexés dans la base vectorielle et utilisés pour le retrieval. Leur petite taille permet un matching précis.
- **Parents** (grands, ~1000 tokens) : retournés au LLM après que le retrieval ait trouvé un enfant pertinent. Leur grande taille donne au LLM assez de contexte pour répondre correctement.

C'est le meilleur des deux mondes : précision du retrieval (petits chunks) et richesse du contexte (grands chunks).

## Taille optimale par cas d'usage

| Cas d'usage | Taille | Overlap |
|---|---|---|
| QA factuelle | 256-512 | 50 |
| Résumé/analyse | 512-1024 | 100 |
| Code | Par fonction | 0 |
| Juridique | Par article | 50 |

<Callout type="paper" title="Que disent les benchmarks ?">
Les comparaisons empiriques montrent que 256-512 tokens offrent le meilleur compromis recall/précision pour le QA factuel. Au-delà de 1024 tokens, la qualité chute : les embeddings compressent trop d'informations hétérogènes en un seul vecteur, et les passages réellement pertinents se noient dans le bruit des passages voisins.
</Callout>


## Enrichissement des chunks

### Métadonnées

Chaque chunk devrait recevoir des métadonnées filtrables : source, page, date, auteur, section. Ça permet un filtrage **avant** la recherche vectorielle.

```json
{
  "text": "contenu du chunk...",
  "metadata": {
    "source": "rapport-2024.pdf",
    "page": 15,
    "section": "Résultats financiers",
    "date": "2024-03-15"
  }
}
```

Exemple : "retrouver dans les rapports 2024 seulement" devient un simple filtre sur la métadonnée `date`, suivi d'une recherche vectorielle sur le sous-ensemble filtré.

### Contextual Retrieval (Anthropic)

Ajouter un **résumé contextuel court** généré par LLM pour chaque chunk, expliquant sa position et son rôle dans le document d'origine :

> **Contexte** : Ce chunk provient du chapitre 3 du rapport annuel 2024, section résultats financiers du Q3.
>
> [Contenu original du chunk]

L'embedding capture alors non seulement le contenu du chunk mais aussi son contexte dans le document. Le retrieval s'améliore significativement car le modèle d'embeddings a plus d'information pour positionner le chunk dans l'espace vectoriel.

### Hypothetical Questions

Générer des questions auxquelles le chunk pourrait répondre, et les stocker avec le chunk. Lors du retrieval, au lieu de comparer la requête utilisateur au texte brut du chunk, on la compare aux questions hypothétiques. Ça rapproche l'espace de recherche de l'usage réel (l'utilisateur pose des questions, pas des affirmations).

## Pièges courants à éviter

<Callout type="warning" title="Erreurs de chunking qui détruisent la qualité">
1. **Ignorer la structure** : découper un tableau ou un bloc de code au milieu le rend incompréhensible
2. **Overlap nul** : les idées à cheval entre deux chunks sont tout simplement perdues
3. **Une seule taille pour tout** : appliquer le même chunking à du code, du texte narratif et des tableaux est une recette pour l'échec
4. **Pas de nettoyage** : chunks pollués par des headers répétés, des footers de page, de la pagination
5. **Chunks trop longs** : au-delà de 2000 tokens, le LLM tend à ignorer les parties non pertinentes du contexte fourni
</Callout>

## Comment évaluer ta stratégie

La meilleure méthode : créer un petit set d'évaluation (50-100 questions avec les passages attendus comme réponses). Faire varier la taille, l'overlap et la stratégie de chunking. **Mesurer le recall@5 et recall@10** sur ton dataset réel.

Le chunking optimal dépend de ton domaine, de tes types de requêtes et de ton modèle d'embeddings. Dix configurations testées empiriquement (ça prend quelques heures) valent mille débats théoriques. Les résultats te surprendront souvent : la configuration "intuitivement optimale" n'est pas toujours la meilleure.
