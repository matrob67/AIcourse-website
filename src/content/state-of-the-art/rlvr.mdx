## Le problème des reward models imparfaits

Le RLHF classique repose sur un **reward model** : un réseau de neurones entraîné sur les préférences humaines qui attribue un score de qualité à chaque réponse. En théorie, c'est élégant. En pratique, c'est une **approximation imparfaite**. Et les LLM, optimisés par RL pour maximiser ce score, finissent par exploiter les failles du reward model plutôt que par véritablement s'améliorer.

C'est le **reward hacking** : le modèle "triche" en trouvant des raccourcis qui plaisent au juge sans produire de meilleures réponses. Par exemple, un reward model qui préfère légèrement les réponses longues va pousser le LLM à produire des pavés verbeux.

La question naturelle : pour les tâches où on peut **vérifier objectivement** si la réponse est correcte, pourquoi s'embêter avec un reward model approximatif ?

<Callout type="info" title={"L'idée fondatrice du RLVR"}>
En maths, en code, en logique formelle : une réponse est **objectivement correcte ou fausse**. 2+2 fait 4, pas 5. Un programme passe ses tests, ou il ne les passe pas.

Au lieu d'un reward model qui approxime la qualité, le RLVR utilise une **vérification automatique exacte** : r=1 si correct, r=0 sinon. Pas d'approximation, pas de biais subjectif, pas de hacking possible.
</Callout>

## Comment ça marche concrètement

### Le pipeline RLVR

1. Le modèle reçoit un problème (mathématique, algorithmique, logique)
2. Il génère un raisonnement complet suivi d'une réponse finale
3. Un **vérificateur automatique** évalue la réponse :
   - **Maths** : compare numériquement avec la réponse correcte
   - **Code** : exécute la solution contre une suite de tests
   - **Logique formelle** : un proof checker vérifie la preuve
4. Récompense binaire : $r = 1$ (correct) ou $r = 0$ (incorrect)
5. L'algorithme de RL utilise cette récompense pour mettre à jour le modèle

### Les vérificateurs par domaine

| Domaine | Méthode de vérification | Fiabilité |
|---|---|---|
| Arithmétique simple | Comparaison numérique | 100% |
| Algèbre | Calcul symbolique (SymPy) | ~99% |
| Code | Exécution de tests | ~95% (dépend de la couverture des tests) |
| Preuves formelles | Lean/Isabelle | 100% |
| Logique SAT | SAT solver | 100% |
| QA factuelle | Matching de réponse | ~90% |

La fiabilité n'est pas toujours parfaite, mais elle est **largement supérieure** à celle d'un reward model entraîné sur des annotations humaines bruitées.

## RLVR en action : le cas DeepSeek-R1

DeepSeek-R1 est l'exemple le plus spectaculaire d'application du RLVR à grande échelle.

### GRPO avec récompenses vérifiables

Pour chaque problème d'entraînement :
1. Générer $G$ réponses différentes (typiquement 8 à 64)
2. Vérifier automatiquement chaque réponse : $r_i \in \{0, 1\}$
3. Calculer l'avantage relatif de chaque réponse : $A_i = \frac{r_i - \mu}{\sigma}$
4. Mettre à jour la policy : renforcer les réponses correctes, pénaliser les incorrectes

Les bonnes réponses sont amplifiées, les mauvaises atténuées. Simple, efficace, scalable.

### Le résultat fascinant : R1-Zéro

Voici le résultat le plus surprenant du RLVR. DeepSeek a entraîné un modèle (R1-Zéro) avec **uniquement** du RLVR, sans jamais lui montrer un seul exemple de chaîne de pensée. Pas de données de SFT avec des raisonnements étape par étape. Juste : "voici un problème, dis-moi la réponse, tu auras un point si c'est correct."

Et le modèle a **spontanément développé** le raisonnement structuré :
- Décomposition en étapes numérotées
- Vérification de ses propres calculs intermédiaires
- Retour en arrière quand une piste ne mène nulle part
- Exploration d'approches alternatives

<Callout type="paper" title={"Le raisonnement émerge naturellement de l'optimisation"}>
R1-Zéro apporte une preuve empirique fascinante : la chaîne de pensée structurée (Chain-of-Thought) est la **stratégie optimale** pour maximiser la probabilité de réponse correcte. Le modèle la découvre par lui-même grâce au RL, sans qu'on la lui enseigne. Le raisonnement pas à pas n'est pas juste un format pratique — c'est la solution émergente au problème d'optimisation.
</Callout>

## Les avantages du RLVR

### 1. Immunité au reward hacking

La récompense est **exacte et non manipulable**. Soit la réponse est 42 et le modèle dit 42 (correct), soit il dit autre chose (incorrect). Pas de zone grise, pas d'exploitation possible de biais subtils dans un reward model.

### 2. Scalabilité quasi-illimitée

Générer des problèmes avec réponse vérifiable est facile à automatiser :
- **Maths** : des générateurs algorithmiques créent des millions de problèmes avec solutions
- **Code** : les plateformes de compétition (Codeforces, LeetCode) fournissent des problèmes avec test suites
- **Preuves formelles** : les assistants de preuves vérifient automatiquement

Pas besoin d'annotateurs humains. Le pipeline de données est entièrement automatisé.

### 3. Signal sans ambiguïté

Une récompense binaire (correct/incorrect) donne un signal d'apprentissage limpide. Pas de subjectivité, pas de bruit inter-annotateurs, pas de discussions sur "est-ce que cette réponse vaut 7/10 ou 8/10 ?".

### 4. Coût réduit

En éliminant les annotateurs humains, le RLVR réduit considérablement le coût et le temps de l'entraînement par RL.

## Les limitations à connaître

### Applicable uniquement aux tâches vérifiables

C'est la restriction fondamentale : le RLVR ne fonctionne que si vous disposez d'un vérificateur automatique fiable. Pour les tâches créatives, conversationnelles, ou subjectives, un reward model appris reste nécessaire. On ne peut pas vérifier automatiquement si un poème est "beau" ou si une explication est "claire".

### Signal sparse (binaire)

La récompense binaire dit "tu as échoué" mais ne dit pas "pourquoi tu as échoué" ni "à quelle étape tu t'es trompé". Le modèle sait que sa réponse est fausse, mais pas ce qu'il aurait dû faire différemment.

C'est pourquoi les **PRM** (Process Reward Models, qui donnent un score à chaque étape intermédiaire du raisonnement) sont un complément naturel au RLVR.

<Callout type="warning" title="Attention à la suroptimisation du format">
Un piège subtil : le modèle peut apprendre à optimiser le **format** de sa réponse plutôt que son **raisonnement**. Si le vérificateur compare des chaînes de caractères, le modèle peut apprendre à produire le format exact attendu sans réellement raisonner. Des vérificateurs robustes (équivalence numérique, équivalence sémantique) atténuent ce problème.
</Callout>

## En pratique : combiner RLVR et RLHF

Les meilleurs systèmes actuels ne choisissent pas entre RLVR et RLHF — ils les combinent :

1. **Phase 1 — RLVR** : entraîner le raisonnement sur des tâches vérifiables (maths, code). Le modèle apprend à réfléchir correctement
2. **Phase 2 — RLHF/DPO** : aligner le style, la sécurité, et l'utilité sur des préférences humaines. Le modèle apprend à communiquer agréablement

DeepSeek-R1 illustre exactement cette approche : RLVR pur pour développer les capacités de raisonnement brut, puis SFT et rejection sampling pour polir la lisibilité et le style des réponses.

Deux phases, deux méthodes complémentaires, un résultat supérieur à chacune utilisée seule.

## Coin technique : générer les données d'entraînement

### Problèmes mathématiques synthétiques

```python
import random

def generate_algebra_problem():
    a = random.randint(1, 20)
    b = random.randint(1, 50)
    x = random.randint(1, 10)
    c = a * x + b
    problem = f"Si {a}x + {b} = {c}, que vaut x ?"
    answer = x
    return problem, answer
```

Ce type de générateur peut produire des millions de problèmes uniques avec leur solution exacte. Variez la difficulté en ajustant les bornes et la complexité des expressions.

### Problèmes de code

Utilisez les banques existantes de problèmes de programmation compétitive (Codeforces, LeetCode, Project Euler) avec leurs suites de tests comme vérificateurs. Chaque problème vient avec des inputs/outputs attendus qui servent directement de vérificateur.

## Le futur du RLVR

La trajectoire est claire : étendre le RLVR au-delà des maths et du code vers de nouveaux domaines :
- **Vérificateurs formels** pour la logique et les preuves dans plus de domaines
- **Exécution de code** comme vérificateur pour des tâches de plus en plus variées
- **Simulations** comme vérificateurs pour la planification et les sciences appliquées
- **Chaînes de vérification** pour des domaines complexes où aucun vérificateur unique ne suffit

<Callout type="tip" title={"L'avenir : investir dans les vérificateurs"}>
La tendance de fond est claire : **investir dans des vérificateurs de haute qualité** plutôt que dans des reward models. Un vérificateur parfait + RLVR produira toujours de meilleurs résultats qu'un reward model imparfait + RLHF. La frontière du raisonnement par IA sera définie par la qualité et la couverture des vérificateurs disponibles.
</Callout>

## Importance conceptuelle

Le RLVR représente un **changement de paradigme** dans l'entraînement des LLM :

- **Avant** : "Approximez la qualité des réponses avec un reward model appris."
- **Après** : "Quand la vérité objective est accessible, utilisez-la directement."

C'est un retour à un principe fondamental de l'IA : quand vous avez accès à une fonction d'évaluation parfaite, utilisez-la au lieu de l'approximer. Combiné avec les PRM pour évaluer chaque étape intermédiaire, le RLVR + PRM forme le système optimal actuel pour entraîner le raisonnement.
