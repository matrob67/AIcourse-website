## Au-delà du simple "appel-réponse"

Appeler un outil, obtenir un résultat, et c'est fini — c'est basique et ça couvre environ 5% des cas réels. Les agents en production doivent gérer la complexité : choisir le bon outil parmi 50 options, chaîner 5 outils pour une tâche complexe, gérer les erreurs avec élégance.

<Callout type="info" title="Les patterns sont les briques de base">
Les patterns de tool use sont les composants élémentaires des agents sophistiqués. Un agent puissant combine plusieurs patterns : du routing pour sélectionner le bon outil, du chaînage pour des workflows multi-étapes, du retry pour la robustesse, et de la validation pour la sécurité.
</Callout>

## Pattern 1 : Routing — choisir le bon outil

### Comment ça marche

Le LLM reçoit une question et doit décider quel outil utiliser parmi plusieurs options. C'est comme un standard téléphonique : l'appel arrive, et il faut l'aiguiller vers le bon département.

```
Question sur la météo → appeler weather_api
Question sur le code → appeler code_search
Question de calcul → appeler calculator
Question générale → appeler web_search
```

### Deux approches

**Implicite** : le LLM choisit via function calling (le modèle est libre de choisir). C'est simple mais moins fiable avec 20+ outils car l'attention du modèle se fragmente.

**Explicite** : un classifieur dédié route d'abord la requête vers la bonne catégorie, puis on sélectionne l'outil spécifique. C'est plus robuste.

<Callout type="tip" title="Routing hiérarchique pour les systèmes complexes">
Avec 20+ outils, organisez-les en hiérarchie. Niveau 1 : "est-ce une question de données, de communication, ou de calcul ?". Niveau 2 : "dans les données, c'est PostgreSQL ou MongoDB ?". Cela réduit l'incertitude à chaque étape.
</Callout>

## Pattern 2 : Chaînage séquentiel

### Une tâche complexe en étapes

Le résultat d'un outil alimente l'entrée du suivant, comme une chaîne de montage :

```
1. search_user("Marie Dupont") → {id: 456}
2. get_orders(user_id=456) → [{order_id: 789, ...}]
3. get_order_details(order_id=789) → {product: "Laptop", status: "livré"}
```

Le LLM orchestre naturellement ce chaînage via le pattern ReAct (Reasoning-Acting) : il raison, décide d'appeler un outil, observe le résultat, raison à nouveau, etc.

## Pattern 3 : Appels parallèles

### Économiser du temps

Quand les appels sont indépendants (ne se dépendent pas les uns des autres), les exécuter en parallèle au lieu de séquentiellement :

```
Utilisateur : "Compare la météo à Paris, Londres et Tokyo"
LLM → [weather("Paris"), weather("London"), weather("Tokyo")]  // tous en parallèle
```

Les modèles modernes supportent les **parallel tool calls**, ce qui réduit la latence. Au lieu de 3 secondes (3 appels séquentiels), vous obtenez 1 seconde.

## Pattern 4 : Outils composites

### Encapsuler la complexité

Combinez des outils simples en un outil plus haut-niveau :

```python
# Outil composite : chercher et résumer
def search_and_summarize(query: str) -> str:
    resultats = web_search(query)  # Outil 1 : recherche brute
    summary = llm.summarize(resultats)  # Outil 2 : résumer
    return summary
```

Du point de vue du LLM, c'est un seul outil. Mais en interne, c'est une pipeline. Avantages :

- **Simplicité** : l'interface qu'on donne au LLM est claire
- **Encapsulation** : la complexité est cachée
- **Efficacité** : le composite peut optimiser l'exécution

## Pattern 5 : Retry et fallback

### Quand ça échoue, réessayer intelligemment

#### Retry avec message d'erreur

```
LLM → weather_api("Paris, France") → Erreur: "Ville non trouvée"
Système → "L'outil a renvoyé une erreur. Essayez sans le pays."
LLM → weather_api("Paris") → Succès: {temp: 18, ...}
```

Le message d'erreur est feedback pour que le LLM corrige son appel. Pas de perte — on a appris.

#### Fallback vers un outil alternatif

```
LLM → internal_search(query) → "Aucun résultat trouvé"
Système → "Essayez une recherche web"
LLM → web_search(query) → [résultats pertinents]
```

C'est comme : "ce service n'a pas la réponse, demande à quelqu'un d'autre".

<Callout type="warning" title="Attention aux boucles infinies de retry">
Définissez un nombre maximal de retries (2-3). Si un outil échoue systématiquement 3 fois, le LLM doit accepter l'échec et proposer une alternative au lieu de réessayer indéfiniment. Sinon vous dépensez du temps/argent pour rien.
</Callout>

## Pattern 6 : Confirmation humaine

### Pour les actions qui comptent vraiment

Pour les actions à haut impact, demander confirmation :

```
LLM : "Je vais envoyer un email à 500 clients avec cette promotion. Confirmez-vous ?"
Utilisateur : "Oui"
LLM → send_email(recipients=500, template="promo_spring")
```

### Catégoriser par risque

| Niveau | Exemples | Confirmation |
|---|---|---|
| Lecture seule | Recherche, lecture de fichier | Jamais |
| Bas impact | Créer un brouillon, calculer quelque chose | Rarement |
| Impact moyen | Envoyer un message, modifier un document | Parfois |
| Haut impact | Supprimer, faire un paiement, email massif | Toujours |

## Pattern 7 : Validation des paramètres

### Vérifier avant d'exécuter

```python
def safe_sql_query(query: str) -> str:
    # Vérifier que ce n'est pas une requête destructive
    if any(keyword in query.upper() for keyword in ["DROP", "DELETE", "TRUNCATE"]):
        return "Erreur: les requêtes destructives sont interdites"
    # Vérifier la syntaxe
    if not is_valid_sql(query):
        return "Erreur: syntaxe SQL invalide"
    return execute_query(query)
```

C'est de la défense en profondeur : on ne fait pas confiance aveuglément au LLM.

## Pattern 8 : Outils avec état

### Certains outils se "souviennent"

Certains outils maintiennent un état entre les appels (comme une session ouverte) :

```python
class SpreadsheetTool:
    def __init__(self):
        self.workbook = None

    def open(self, path: str) -> str:
        self.workbook = load_workbook(path)
        return f"Ouvert: {path} ({len(self.workbook.sheets)} feuilles)"

    def read_cell(self, sheet: str, cell: str) -> str:
        return str(self.workbook[sheet][cell].value)

    def write_cell(self, sheet: str, cell: str, value: str) -> str:
        self.workbook[sheet][cell] = value
        return f"Écrit {value} dans {sheet}!{cell}"
```

Au lieu de charger le fichier à chaque appel, on le garde ouvert. C'est plus efficace.

## Bonnes pratiques de conception

### 1. Granularité — ni trop fin, ni trop gros

- **Trop fin** : le LLM doit orchestrer trop de petits appels (inefficace)
- **Trop gros** : les outils monolithiques manquent de flexibilité
- **Le bon niveau** : une action métier cohérente par outil

### 2. Descriptions comme contrat

Les descriptions des outils sont le "contrat" entre vous et le LLM :

```python
@tool("search_customers")
def search_customers(name: str, country: str = "FR") -> list[Customer]:
    """
    Cherche les clients par nom et pays.

    Exemples d'utilisation:
    - search_customers("Dupont") pour tous les Dupont en France
    - search_customers("Smith", "US") pour les Smith aux USA

    Retourne une liste de clients (max 100).
    """
    ...
```

Incluez des exemples et les cas limites.

### 3. Idempotence quand c'est possible

Un outil idempotent peut être appelé plusieurs fois sans nouvelle conséquence. Cela simplifie les retries.

```python
# Idempotent : pas de problème d'appeler deux fois
def set_user_preference(user_id, pref, value):
    db.set(f"user_{user_id}_{pref}", value)

# Pas idempotent : appeler deux fois charge deux fois
def add_credit(user_id, amount):
    db.increment(f"user_{user_id}_credit", amount)
```

### 4. Logging et observabilité

Loggez chaque appel d'outil : paramètres, durée, résultat. C'est essentiel pour le debugging et l'audit.

<Callout type="tip" title="Le pattern qui change tout">
Si vous ne retenez qu'un seul pattern, c'est **retry avec message d'erreur clair**. La plupart des échecs d'agents viennent d'erreurs d'outils mal gérées. Un message d'erreur descriptif permet souvent au LLM de corriger son appel automatiquement. "City not found" → LLM essaie sans le pays. C'est magique.
</Callout>
