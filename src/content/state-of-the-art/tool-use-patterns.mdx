## Au-dela du simple "appel-réponse"

Appeler un outil, recevoir un résultat, et passer a autre chose — c'est le cas de base, et ça couvre a peine 5% des situations réelles. En production, un agent doit gérer une complexité bien supérieure : choisir le bon outil parmi 50 options, enchaîner 5 appels pour accomplir une tâche, gérer les erreurs avec élégance, et protéger l'utilisateur des actions irréversibles.

Les patterns qu'on va voir sont les **briques de construction** de tout agent sérieux. Comprendre ces patterns, c'est comprendre comment construire des agents qui fonctionnent dans le monde réel, pas juste dans une démo.

<Callout type="info" title="Les patterns sont les fondations des agents">
Un agent sophistiqué n'est pas un seul algorithme brillant — c'est une combinaison intelligente de patterns élémentaires : du routing pour aiguiller vers le bon outil, du chaînage pour les workflows multi-étapes, du retry pour la robustesse, et de la validation pour la sécurité. Maîtrisez ces patterns et vous pouvez construire n'importe quel agent.
</Callout>

## Pattern 1 : Routing — aiguiller vers le bon outil

### Le problème

Le LLM reçoit une requête et doit décider quel outil utiliser parmi plusieurs options. C'est exactement comme un standard téléphonique : l'appel arrive, et il faut l'aiguiller vers le bon département sans se tromper.

| Question de l'utilisateur | Outil approprié |
|---|---|
| "Quel temps fait-il ?" | `weather_api` |
| "Trouve la fonction handleLogin" | `code_search` |
| "Calcule 15% de 340" | `calculator` |
| "Qui a inventé le téléphone ?" | `web_search` |

### Deux approches de routing

**Routing implicite** : le LLM choisit librement via le function calling natif. C'est simple a implémenter, mais la fiabilité se dégrade avec le nombre d'outils. Au-dela de 15-20 outils, l'attention du modèle se fragmente et il commence à choisir des outils inappropriés.

**Routing explicite** : un classifieur dédié (ou un prompt spécialisé) analyse d'abord la requête pour déterminer la catégorie, puis sélectionne l'outil spécifique dans cette catégorie. Plus robuste, mais plus complexe a maintenir.

<Callout type="tip" title="Routing hiérarchique pour les systèmes complexes">
Avec 20+ outils, organisez-les en hiérarchie a deux niveaux. Niveau 1 : "est-ce une question de données, de communication, ou de calcul ?". Niveau 2 : "dans les données, c'est PostgreSQL, MongoDB, ou Elasticsearch ?". Chaque niveau réduit l'espace de choix, rendant la décision plus fiable à chaque étape.
</Callout>

## Pattern 2 : Chaînage séquentiel

### Découper une tâche complexe en étapes

Le résultat d'un outil alimente l'entrée du suivant, comme une chaîne de montage industrielle. Chaque maillon produit ce dont le suivant a besoin :

1. `search_user("Marie Dupont")` &rarr; `{id: 456}`
2. `get_orders(user_id=456)` &rarr; `[{order_id: 789, ...}]`
3. `get_order_details(order_id=789)` &rarr; `{product: "Laptop", status: "livré"}`

Le LLM orchestre naturellement ce chaînage via le pattern **ReAct** (Reasoning-Acting) : il raisonne sur la situation, décide d'appeler un outil, observé le résultat, raisonne à nouveau, et ainsi de suite jusqu'à avoir toutes les informations nécessaires. C'est une boucle penser-agir-observer qui s'adapte dynamiquement.

## Pattern 3 : Appels parallèles

### Gagner du temps quand les appels sont indépendants

Quand plusieurs appels d'outils n'ont aucune dépendance entre eux (le résultat de l'un n'est pas nécessaire pour l'autre), les exécuter en parallèle divise la latence :

> **Utilisateur :** "Compare la météo a Paris, Londres et Tokyo"
> **LLM** &rarr; `[weather("Paris"), weather("London"), weather("Tokyo")]` — les trois en parallèle

Les modèles modernes supportent nativement les **parallel tool calls**. Au lieu de 3 secondes (3 appels séquentiels d'1 seconde chacun), vous obtenez la réponse en 1 seconde. Pour l'utilisateur, c'est transparent. Pour la latence, c'est transformateur.

## Pattern 4 : Outils composites

### Encapsuler la complexité derrière une interface simple

Combinez plusieurs outils simples en un outil de plus haut niveau :

```python
# Outil composite : chercher et résumer
def search_and_summarize(query: str) -> str:
    resultats = web_search(query)  # Outil 1 : recherche brute
    summary = llm.summarize(resultats)  # Outil 2 : résumer
    return summary
```

Du point de vue du LLM, c'est un seul outil avec une interface claire. Mais en interne, c'est un pipeline complet. Les avantages sont triples :

- **Simplicité** : le LLM voit un outil au lieu de deux, ce qui simplifie son choix
- **Encapsulation** : la logique de coordination est cachée dans l'outil, pas déléguée au LLM
- **Fiabilité** : le composite gère les détails (pagination, formatage) sans impliquer le LLM

## Pattern 5 : Retry et fallback

### Quand ça échoue, réessayer intelligemment

Les outils échouent. C'est normal. Ce qui compte, c'est comment l'agent réagit.

#### Retry avec feedback d'erreur

> **LLM** &rarr; `weather_api("Paris, France")` &rarr; Erreur : "Ville non trouvée"
> **Système** &rarr; "L'outil a renvoyé une erreur : ville non trouvée. Reformulez."
> **LLM** &rarr; `weather_api("Paris")` &rarr; Succès : `{temp: 18, ...}`

Le message d'erreur descriptif donne au LLM les informations nécessaires pour corriger son appel. C'est la clé : un bon message d'erreur transforme un échec en apprentissage en temps réel.

#### Fallback vers un outil alternatif

> **LLM** &rarr; `internal_search(query)` &rarr; "Aucun résultat trouvé"
> **Système** &rarr; "La base interne n'a rien trouvé. Essayez une source externe."
> **LLM** &rarr; `web_search(query)` &rarr; [résultats pertinents]

Quand un outil ne peut pas répondre, un autre prend le relais. C'est la robustesse par la redondance.

<Callout type="warning" title="Attention aux boucles infinies de retry">
Définissez toujours un nombre maximal de tentatives (2-3 max). Si un outil échoue systématiquement après 3 essais, le LLM doit accepter l'échec et proposer une alternative à l'utilisateur. Sinon, vous dépensez du temps et des tokens pour rien dans une boucle sans fin.
</Callout>

## Pattern 6 : Confirmation humaine

### Demander avant d'agir quand les enjeux sont élevés

Certaines actions sont irréversibles ou a fort impact. Pour celles-la, l'agent doit demander confirmation :

> **LLM :** "Je vais envoyer un email a 500 clients avec cette promotion. Confirmez-vous ?"
> **Utilisateur :** "Oui"
> **LLM** &rarr; `send_email(recipients=500, template="promo_spring")`

### Catégoriser les actions par niveau de risque

| Niveau | Exemples | Confirmation requise |
|---|---|---|
| Lecture seule | Recherche, consultation de fichier | Jamais |
| Bas impact | Créer un brouillon, faire un calcul | Rarement |
| Impact moyen | Envoyer un message, modifier un document | Parfois |
| Haut impact | Supprimer des données, paiement, email massif | Toujours |

Cette classification doit être définie à l'avance dans la configuration de l'agent, pas laissée au jugement du LLM.

## Pattern 7 : Validation des paramètres

### Vérifier avant d'exécuter

Ne faites jamais confiance aveuglément aux paramètres générés par le LLM. Validez-les avant l'exécution :

```python
def safe_sql_query(query: str) -> str:
    # Vérifier que ce n'est pas une requête destructive
    if any(keyword in query.upper() for keyword in ["DROP", "DELETE", "TRUNCATE"]):
        return "Erreur: les requêtes destructives sont interdites"
    # Vérifier la syntaxe
    if not is_valid_sql(query):
        return "Erreur: syntaxe SQL invalide"
    return execute_query(query)
```

C'est de la défense en profondeur : même si le LLM fait une erreur (ou est manipulé), la couche de validation bloque les actions dangereuses.

## Pattern 8 : Outils avec état

### Quand un outil se "souvient" entre les appels

Certains outils maintiennent un état persistant entre les appels successifs. C'est plus efficace que de tout recharger à chaque fois :

```python
class SpreadsheetTool:
    def __init__(self):
        self.workbook = None

    def open(self, path: str) -> str:
        self.workbook = load_workbook(path)
        return f"Ouvert: {path} ({len(self.workbook.sheets)} feuilles)"

    def read_cell(self, sheet: str, cell: str) -> str:
        return str(self.workbook[sheet][cell].value)

    def write_cell(self, sheet: str, cell: str, value: str) -> str:
        self.workbook[sheet][cell] = value
        return f"Écrit {value} dans {sheet}!{cell}"
```

L'outil ouvre le fichier une fois, puis toutes les opérations suivantes sont rapides car le fichier est déja en mémoire. C'est le même principe qu'une connexion à une base de données qu'on garde ouverte.

## Bonnes pratiques de conception

### 1. Granularité : trouver le juste milieu

- **Trop fin** : le LLM doit orchestrer des dizaines de micro-appels (lent, fragile)
- **Trop gros** : les outils monolithiques manquent de flexibilité
- **Le bon niveau** : une action métier cohérente par outil. "Chercher un client" est bon. "Faire un appel HTTP GET" est trop bas. "Gérer tout le CRM" est trop haut.

### 2. Les descriptions d'outils sont un contrat

Les descriptions sont le "mode d'emploi" que le LLM utilise pour décider quand et comment appeler un outil :

```python
@tool("search_customers")
def search_customers(name: str, country: str = "FR") -> list[Customer]:
    """
    Cherche les clients par nom et pays.

    Exemples d'utilisation:
    - search_customers("Dupont") pour tous les Dupont en France
    - search_customers("Smith", "US") pour les Smith aux USA

    Retourne une liste de clients (max 100).
    """
    ...
```

Des descriptions claires avec des exemples concrets améliorent drastiquement la fiabilité du tool use.

### 3. Idempotence quand c'est possible

Un outil idempotent produit le même résultat qu'on l'appelle une ou dix fois. Cela simplifie énormément les retries :

```python
# Idempotent : appeler deux fois ne pose aucun problème
def set_user_preference(user_id, pref, value):
    db.set(f"user_{user_id}_{pref}", value)

# Pas idempotent : appeler deux fois double l'effet
def add_credit(user_id, amount):
    db.increment(f"user_{user_id}_credit", amount)
```

### 4. Logging et observabilité

Loggez systématiquement chaque appel d'outil : paramètres, durée, résultat ou erreur. Sans logs, débugger un agent en production est un cauchemar. Avec des logs structurés, vous pouvez retracer exactement ce qui s'est passé et pourquoi.

<Callout type="tip" title="Le pattern qui change tout : retry avec message d'erreur clair">
Si vous ne retenez qu'un seul pattern de cette page, c'est celui-la. La majorité des échecs d'agents en production viennent d'erreurs d'outils mal gérées. Un message d'erreur descriptif permet au LLM de corriger son appel automatiquement dans la plupart des cas. "City not found" suffit au LLM pour essayer sans le pays. C'est simple, et ça transforme la fiabilité d'un agent.
</Callout>
