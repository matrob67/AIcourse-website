## Au-delà de l'arbre : fusionner et raffiner

Tree of Thoughts contraint le raisonnement en arbre : chaque thought a un parent. **Graph of Thoughts** lève cette restriction.

Un thought peut :
- **Combiner** plusieurs thoughts précédents (fusion)
- **Être amélioré** itérativement (raffinement)
- **Former des boucles** (revenir à une étape avec nouvelles infos)

<ArxivLink id="2308.09687" title="Graph of Thoughts: Solving Elaborate Problems with Large Language Models (Besta et al., 2023)" />

<Callout type="info" title="Arbre → Graphe">
Arbre : information coule parent → enfants. Graphe : on peut **fusionner** des branches, **raffiner** les idées, **boucler**. C'est plus riche, plus flexible.
</Callout>



## Les quatre opérations

**Generate (Branching)** : créer plusieurs thoughts candidats. Comme dans ToT.

**Aggregate (Merging)** : combiner deux thoughts en un seul.

```
Thought A: "Budget marketing = 50K€"
Thought B: "ROI attendu = 3.2×"
→ Agrégation: "Revenu attendu = 50K × 3.2 = 160K€"
```

**Refine (Improving)** : améliorer un thought en le critiquant et corrigeant.

```
v1: "Tri fusion = O(n²)"
→ Correction: "Tri fusion = O(n log n)"
```

**Score (Evaluation)** : évaluer la qualité. Comme ToT.

## Architecture GoT

```
[Problème]
    ├──→ [Thought 1a] → [2a] ──┐
    ├──→ [Thought 1b] → [2b] ──┤──→ [Agrégation] → [Raffinement] → [Solution]
    └──→ [Thought 1c] → [2c] ──┘
```

Contrairement à l'arbre, **les branches se recombinent**. C'est mimique du divide-and-conquer.

## Le Controller

Un **controller** orchestre les opérations :

1. Analyse l'état du graphe actuel
2. Décide quelle opération appliquer (generate, aggregate, refine, score)
3. Sélectionne les thoughts à opérer
4. Vérifie les conditions d'arrêt

<Callout type="paper" title="Résultats sur le tri">
Trier 128 nombres : CoT 12%, ToT 36%, GoT 62%. GoT excelle parce que le tri est naturellement décomposable (tri fusion). Les sous-listes sont triées indépendamment puis fusionnées.
</Callout>

## Cas d'usage

**Problèmes décomposables** : qui se divisent en sous-problèmes indépendants à recombiner.
- Tri et fusion
- Analyse multi-critères
- Planification

**Écriture collaborative** : chaque branche = paragraphe, agrégation = texte cohérent, raffinement = fluidité.

**Debugging** : hypothèses parallèles, agrégation des résultats tests, diagnostic raffiné.

## Comparaison des paradigmes

| Paradigme | Structure | Opérations | Coût relatif | Meilleur pour |
|---|---|---|---|---|
| CoT | Chaîne | Séquentiel | 1× | Simple, rapide |
| Self-Consistency | Chaînes multiples | Vote | K× | Robustesse |
| ToT | Arbre | Branch + Prune | 10-30× | Exploration |
| GoT | Graphe | Branch + Merge + Refine | 20-50× | Composites |

<Callout type="tip" title="La hiérarchie">
Commencez par CoT. Si inconsistent, ajoutez self-consistency. Si exploration nécessaire, passez à ToT. GoT = seulement pour problèmes vraiment décomposables.
</Callout>

## Limitations pratiques

**Orchestration complexe** : quand brancher, agréger, raffiner ? Si c'est le LLM qui décide, le surcoût en tokens est énorme.

**Coût prohibitif** : 5 branches × 3 niveaux × merges + refines = 50+ appels LLM pour un problème. Rarement justifiable pour tâches quotidiennes.

<Callout type="warning" title="Coût vs bénéfice">
GoT théoriquement élégant, mais rarement utilisé en production. Les modèles de raisonnement modernes (o1, DeepSeek-R1) internalisent l'exploration en un seul forward pass. Meilleur rapport coût/bénéfice pour la plupart.
</Callout>

## Perspective

GoT c'est l'aboutissement logique de CoT → ToT → GoT. Le raisonnement = problème de recherche dans un graphe. Même si GoT explicite est peu utilisé, ses idées (agrégation, raffinement) persistent dans les systèmes d'agents et de raisonnement modernes.

GoT montre ce qui est théoriquement possible. La pratique tend vers des approches plus légères qui atteignent les mêmes bénéfices.
