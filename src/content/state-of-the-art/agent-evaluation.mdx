## Le problème : comment tester un agent ?

Évaluer un modèle de langage est relativement simple : vous lui posez une question, il répond, c'est juste ou c'est faux. Mais un agent IA ? C'est beaucoup plus compliqué. Un agent prend une **série de décisions dans un environnement dynamique**. Il doit chercher des informations, décider quoi faire ensuite, modifier son approche si ça ne marche pas.

Évaluer ça, c'est comme évaluer un pilote de voiture : pas juste avec "pouvez-vous tourner le volant ?", mais "pouvez-vous vous garer dans un créneau difficile, gérer les imprévus, éviter les obstacles ?".

<Callout type="info" title="Ce qu'un bon benchmark d'agent mesure">
**Réussite** : la tâche est-elle complétée correctement ? **Efficacité** : combien d'étapes, combien ça coûte en tokens ? **Robustesse** : ça marche sur des variations de la même tâche ? **Sécurité** : l'agent a-t-il fait des actions dangereuses ou non-autorisées ?
</Callout>

## SWE-bench : tester les agents de code

### La mission

SWE-bench évalue les agents sur leur capacité à **résoudre des bugs réels dans de vrais projets**. Les tâches viennent de GitHub : Django, Flask, Scikit-learn, Matplotlib — des projets Python populaires avec des milliers d'utilisateurs.

Chaque tâche est une issue GitHub avec un commit de correction associé. C'est du test "à difficultés réelles".

### Comment ça fonctionne

1. Une issue est sélectionnée ("Le graphe de dépendances est cassé")
2. L'agent reçoit l'issue et accès au repository complet
3. Il doit produire un patch qui corrige le problème
4. Le patch est évalué avec les tests du projet

C'est objectif et reproductible.

### Trois versions

- **Full** (2294 tâches) : incluant les cas très difficiles, à peine solvables
- **Lite** (300 tâches) : sélection de tâches résolvables
- **Verified** : sous-ensemble vérifié manuellement

### Résultats actuels (2024-2025)

| Système | Taux de réussite (%) |
|---|---|
| Simple RAG | ~2 |
| Claude 3.5 Sonnet (avec agent) | ~49 |
| OpenAI o1 | ~69 |
| Claude Code | ~72 |
| Meilleur système | ~75 |

<Callout type="paper" title="Une progression spectaculaire">
En 12 mois, les scores sont passés de 3% à 75%. C'est la combinaison de modèles meilleurs + frameworks d'agents meilleurs. Mais l'écart se rétrécit : les tâches restantes sont les vraiment difficiles, et chaque point de gain demande plus d'effort.
</Callout>

## WebArena : agents web

### La mission

WebArena évalue les agents sur des tâches de **navigation web** dans des environnements réalistes. Pas juste "cliquez sur le bouton rouge", mais des tâches complexes qui nécessitent de naviguer, chercher des infos, remplir des formulaires.

Les environnements incluent :
- Un forum
- Un e-commerce
- Un CMS
- Un serveur GitLab

### Exemples de tâches

- "Trouve le produit le moins cher dans la catégorie électronique et ajoute-le au panier"
- "Crée un nouveau post dans le forum avec ce titre et ce contenu"
- "Résous l'issue #42 sur GitLab en soumettant une merge request"

### Métriques

- **Taux de réussite** : la tâche est-elle complétée correctement ?
- **Efficacité des étapes** : nombre d'actions nécessaires
- **Justesse fonctionnelle** : le résultat final fonctionne-t-il vraiment ?

### Résultats

Les meilleurs agents atteignent environ 30-40% de réussite. La navigation web autonome reste un défi majeur. Les erreurs typiques :

- Mauvaise interprétation de l'interface (cliquer sur le mauvais bouton)
- Perte de contexte spatial (oublier où on est)
- Incapacité à se rétablir après une erreur

## Autres benchmarks importants

### GAIA (Google AI Assist)

Tâches d'assistants IA généraux :
- 466 questions de difficulté variée
- Nécessite recherche web, calculs, raisonnement multi-étapes
- Les humains réussissent à 92%
- Les meilleurs agents à ~70%

### AgentBench

Évalue dans 8 environnements différents :
- Bash (ligne de commande)
- SQL (bases de données)
- Knowledge graphs
- Jeux texte et stratégie
- Navigation web

### ToolBench

Teste la capacité à utiliser des **vraies APIs du monde réel** :
- 16 000+ APIs différentes
- Tâches pratiques : "book a hotel under $100"
- Plus proche de la réalité que les benchmarks synthétiques

### OSWorld

Évalue dans des **environnements de bureau complets** (Ubuntu, Windows, macOS) :
- Éditer des documents
- Gérer des fichiers
- Utiliser des applications normales



## Concevoir votre propre benchmark

### Pour évaluer un agent sur votre cas d'usage

1. **Définir les tâches** : 50-200 tâches représentatives
2. **Critères de succès** : automatiques si possible (vérifiable sans humain)
3. **Varier la difficulté** : tâches faciles, moyennes, complexes
4. **Edge cases** : erreurs, données manquantes, ambiguïtés
5. **Mesurer le coût** : tokens et API calls par tâche

### Métriques à suivre

| Métrique | Explication | Importance |
|---|---|---|
| Taux de réussite | Tâche complétée correctement | Critique |
| Nombre d'étapes | Combien d'actions pour réussir | Coût/efficacité |
| Coût par tâche | Tokens x prix | Budget |
| Taux de récupération | Récupération après erreur | Robustesse |
| Violations de sécurité | Actions non-autorisées | Sécurité |
| Temps total | Durée end-to-end | UX |

<Callout type="tip" title="L'évaluation doit être automatisée">
Un benchmark qu'on ne peut pas lancer automatiquement ne sera jamais exécuté régulièrement. Investissez dans l'automatisation : vérifications automatiques des résultats, scripts de benchmark reproductibles, CI/CD intégrée. "Run make evaluate" doit être aussi simple que possible.
</Callout>

## Les limites des benchmarks actuels

### Contamination des données

Les benchmarks publics peuvent être "contaminés" : les modèles les plus récents ont peut-être vu les solutions de SWE-bench pendant l'entraînement, ce qui gonfle les scores de manière artificielle.

### Tâches plus simples que la réalité

Même les benchmarks les plus difficiles restent plus simples que les vraies tâches quotidiennes :
- Les tâches sont bien définies
- Les résultats sont objectifs
- L'environnement est contrôlé

La production implique du bruit, des imprévus, des cas non-anticipés.

<Callout type="warning" title="Benchmark ≠ Production">
Un excellent score sur SWE-bench ne garantit pas qu'un agent sera fiable chez vous. Les benchmarks testent des capacités isolées. La production est variée, imprévisible, et pleine de cas limites. L'évaluation continue en production (monitoring, feedback utilisateur) est indispensable.
</Callout>

## La future des évaluations

L'évaluation des agents est un domaine en construction. Le défi principal :

- **Benchmarks plus réalistes** : pas juste "résoudre une issue GitHub"
- **Plus diversifiés** : pas juste code ou web, mais tous les domaines
- **Résistants à la contamination** : éviter que les modèles les apprennent
- **Évaluation continue** : pas juste un snapshot, mais du monitoring en production

La combinaison de benchmarks statiques (SWE-bench) + évaluation continue (monitoring réel) reste la meilleure approche actuellement.
