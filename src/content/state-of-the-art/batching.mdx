## Le problème du batching classique

Imagine un restaurant où le serveur attend que les 8 clients à sa table aient tous fini de manger avant de débarrasser. Si un client finit son entrée en 5 minutes et un autre prend 45 minutes pour son plat, les 7 premiers restent plantés là à attendre. Le serveur est bloqué, les tables suivantes ne sont pas servies.

C'est exactement ce qui arrive avec le batching classique (statique) pour les LLM. Tu groupes 8 requêtes ensemble et les traites en parallèle. Mais si une requête génère 500 tokens et une autre seulement 50, le GPU calcule du vide pour les 7 requêtes terminées pendant que la dernière finit sa génération. C'est du gaspillage pur et simple.

<Callout type="info" ’>
Avec un batch statique de 8 requêtes de longueurs variées, le GPU passe jusqu'à **90% de son temps à calculer du padding** pour les requêtes déjà terminées. Le throughput réel est une fraction du potentiel théorique du matériel.
</Callout>

## Continuous Batching : la solution

### Le principe

Au lieu de traiter les requêtes du début à la fin en bloc, le continuous batching gère chaque **step de génération** individuellement :

1. À chaque step, générer **un token** pour chaque requête active dans le batch
2. Quand une requête finit (token EOS ou limite atteinte), la **retirer immédiatement** du batch
3. La remplacer instantanément par une **nouvelle requête** depuis la file d'attente
4. Le batch reste **constamment rempli** à pleine capacité

Le GPU ne tourne jamais à vide. Dès qu'une place se libère, une nouvelle requête prend le relais. C'est comme un restaurant où on débarrasse chaque assiette individuellement et où on sert le client suivant immédiatement.

### Prefill vs Decode

Les LLM ont deux phases de traitement très différentes :

- **Prefill** : traiter le prompt d'entrée en une seule passe. C'est une opération **parallélisable** qui exploite bien les GPU (toute la séquence d'entrée est traitée simultanément).
- **Decode** : générer la réponse token par token. C'est **séquentiel** par nature et beaucoup moins efficace en termes d'utilisation GPU.

Le continuous batching mélange les deux phases dans le même batch : certaines requêtes en sont à leur phase de prefill tandis que d'autres génèrent déjà des tokens. Ça maximise l'utilisation du GPU à tout instant.

<Callout type="tip" title="Gain de throughput">
Le continuous batching peut améliorer le throughput de **2 à 8x** par rapport au batching statique, selon la variabilité des longueurs de réponse. Plus les longueurs sont hétérogènes (ce qui est le cas en production), plus le gain est important.
</Callout>

## Dynamic Batching

Une variante complémentaire : adapter dynamiquement la composition du batch selon la charge :

- **File d'attente avec timeout** : regrouper les requêtes pendant 50ms avant de lancer le traitement, pour maximiser la taille du batch initial
- **Batch size adaptatif** : ajuster le nombre de requêtes simultanées selon la mémoire GPU disponible et les longueurs estimées des séquences

### Chunked Prefill

Une optimisation importante pour éviter les pics de latence : découper les longs prompts en petits chunks au lieu de les traiter d'un seul bloc.

Exemple : un prompt de 8000 tokens est découpé en chunks de 512 tokens. Chaque chunk est traité dans un batch mixte avec d'autres requêtes en cours de decode. Résultat : les longs prompts ne monopolisent plus le GPU pendant de longues périodes. Les requêtes de decode continuent de progresser normalement, ce qui maintient une latence stable pour tous.



## Gestion du KV Cache : le vrai défi

### Le problème mémoire

Le **KV cache** (Key-Value cache) stocke les clés et valeurs d'attention déjà calculées pour chaque requête. Sans lui, il faudrait recalculer toute l'attention à chaque nouveau token. Mais sa taille croît avec la longueur de la séquence :

Pour un modèle 7B (32 couches, 32 têtes, 128 dimensions par tête) :

| Granularité | Taille KV cache |
|---|---|
| Par token | 524 Ko |
| Pour 4096 tokens | 2 Go par requête |
| Pour 32 requêtes | 64 Go |

C'est **énorme**. Et le problème, c'est qu'on ne connaît pas la longueur finale de la réponse à l'avance. Combien réserver ?

### Stratégies d'allocation

Trois approches existent :

1. **Pré-allocation maximale** : réserver la longueur max pour chaque requête. Simple, mais gaspille 60-80% de la mémoire avec du vide.
2. **PagedAttention** : allocation par pages de taille fixe, à la demande. Réduit le gaspillage à < 4%. C'est le standard actuel.
3. **Radix Tree** : organiser les KV caches en arbre pour partager automatiquement les préfixes communs entre requêtes.

<Callout type="warning" title="Le vrai facteur limitant">
Sur les systèmes de serving modernes, le KV cache consomme **plus de mémoire que le modèle lui-même** pour de gros batch sizes. C'est le facteur limitant du nombre de requêtes simultanées. Toute l'ingénierie de serving de LLM tourne autour de la gestion efficace de cette mémoire.
</Callout>

## Scheduling avancé

### Priority-based scheduling

Toutes les requêtes ne se valent pas. Un chatbot interactif a besoin de réponses rapides, un traitement batch de documents peut attendre. Le scheduling par priorité permet de respecter les SLA (Service Level Agreements) de chaque type de requête.

### Preemption

Quand la mémoire GPU est saturée, il faut faire de la place :
- **Swap** : déplacer le KV cache d'une requête en RAM CPU (lent au retour, mais la requête n'est pas perdue)
- **Recompute** : supprimer le KV cache et le recalculer plus tard quand de la mémoire se libère

### Multi-step scheduling

Au lieu de prendre des décisions de scheduling à chaque token, planifier plusieurs steps de decode à l'avance. Ça réduit l'overhead de gestion et permet des optimisations de batching plus intelligentes.

## Métriques de performance

### Les trois mesures essentielles

- **TTFT** (Time To First Token) : le temps avant que le premier token de réponse apparaisse. Détermine la "réactivité" perçue par l'utilisateur.
- **TPOT** (Time Per Output Token) : le temps moyen entre chaque token généré. Détermine la "fluidité" de la génération.
- **Throughput** : le nombre total de tokens/seconde que le système produit, toutes requêtes confondues.

Le dilemme : augmenter le batch size augmente le throughput global, mais aussi le TPOT (chaque requête individuelle est légèrement ralentie car le GPU partage son temps).

<Callout type="info" title="Le compromis throughput-latence">
Pour un chatbot interactif, on vise un TPOT < 50ms pour une expérience fluide. Pour du traitement batch (résumer 1000 documents), on maximise le throughput total, la latence par requête important peu. Le scheduler doit jongler entre ces objectifs contradictoires.
</Callout>

## Impact réel

Le continuous batching est devenu **indispensable** en production. Sans lui, les coûts d'inférence seraient **5 à 10 fois plus élevés** pour le même niveau de service. C'est une optimisation système qui ne change rien au modèle lui-même mais transforme radicalement l'économie du déploiement.
